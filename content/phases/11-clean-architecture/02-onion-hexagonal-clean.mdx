## The Same Idea, Three Names

In interviews you will hear all three names. Understanding the relationship prevents confusion:

- **Hexagonal Architecture** (Alistair Cockburn, 2005) — "Ports and Adapters": the application sits in the center, external actors (DB, HTTP, UI, queues) connect via ports (interfaces) and adapters (implementations).
- **Onion Architecture** (Jeffrey Palermo, 2008) — visualizes the same idea as concentric rings (onion layers), names the rings slightly differently, and emphasizes the Domain Model at the center.
- **Clean Architecture** (Robert C. Martin, 2012) — synthesizes both, adds explicit names for 4 layers (Entities, Use Cases, Interface Adapters, Frameworks & Drivers), and popularized the term.

They all enforce one constraint: **inner layers cannot reference outer layers**.

## Visual Comparison

```
HEXAGONAL                    ONION                      CLEAN
─────────────────────        ─────────────────────      ─────────────────────
                             [ Infrastructure  ]        [ Frameworks/UI      ]
[ Adapter ] ──→ [ Port ]     [ Application Svc ]        [ Interface Adapters ]
                             [ Domain Services ]        [ Use Cases          ]
[ Application ]              [ Domain Model   ]         [ Entities           ]
                             (core)                     (core)
[ Adapter ] ──→ [ Port ]
```

In Hexagonal, the "application" is a single hexagon with ports on its edges. In Onion and Clean, the layering is explicit and named. The arrows always point inward.

## Hexagonal in C#: Ports and Adapters

Hexagonal thinking is particularly useful when modeling **driving adapters** (things that call your app: HTTP, gRPC, queues) vs. **driven adapters** (things your app calls: DB, email, external APIs):

```csharp
// PORT: defined by the Application, facing outward (driven)
// Lives in Application/Interfaces/
public interface INotificationPort
{
    Task SendOrderConfirmedAsync(OrderId orderId, string email, CancellationToken ct);
}

// ADAPTER: lives in Infrastructure, implements the port
public class SendGridNotificationAdapter : INotificationPort
{
    private readonly SendGridClient _client;

    public SendGridNotificationAdapter(SendGridClient client) => _client = client;

    public async Task SendOrderConfirmedAsync(OrderId orderId, string email, CancellationToken ct)
    {
        var msg = MailHelper.CreateSingleEmail(
            from: new EmailAddress("noreply@shop.com"),
            to: new EmailAddress(email),
            subject: "Order Confirmed",
            plainTextContent: $"Your order {orderId} is confirmed.",
            htmlContent: $"<p>Your order <b>{orderId}</b> is confirmed.</p>"
        );
        await _client.SendEmailAsync(msg, ct);
    }
}

// DRIVING ADAPTER: HTTP controller calls the application
[ApiController]
[Route("api/orders")]
public class OrdersController : ControllerBase  // This IS an adapter
{
    private readonly ISender _sender;
    public OrdersController(ISender sender) => _sender = sender;

    [HttpPost("{id:guid}/confirm")]
    public async Task<IActionResult> Confirm(Guid id, CancellationToken ct)
        => Ok(await _sender.Send(new ConfirmOrderCommand(id), ct));
}
```

The port/adapter naming makes it obvious: the HTTP controller is a "driving adapter" that drives the application; the database is a "driven adapter" that the application drives.

## Onion Architecture in C#: Named Rings

Onion architecture explicitly names the rings and often organizes projects by ring name:

```csharp
// Ring 1 (innermost): Domain Model
// Project: MyApp.Domain
public class Customer
{
    public CustomerId Id { get; private set; }
    public Email Email { get; private set; }
    // no EF, no frameworks
}

// Ring 2: Domain Services (still no framework dependencies)
// Project: MyApp.Domain
public interface ICustomerRepository { } // interface lives in Domain in Onion!
// Note: Clean Architecture puts interfaces in Application layer

// Ring 3: Application Services
// Project: MyApp.Application
public class CustomerService
{
    private readonly ICustomerRepository _repo; // uses Domain interface
    public CustomerService(ICustomerRepository repo) => _repo = repo;
}

// Ring 4 (outermost): Infrastructure + UI
// Project: MyApp.Infrastructure
public class EfCustomerRepository : ICustomerRepository { }
```

Notice: **Onion places repository interfaces in the Domain layer** (they're part of the domain contract), while **Clean Architecture places them in the Application layer** (they're application-level ports). This is the most common structural difference between the two.

<Callout type="senior">
**Why You Should Know All Three Names**

In interviews and architecture discussions, different teams use different vocabulary. A team that reads DDD literature might say "domain model" and "application service." A team that followed Clean Architecture templates says "entities" and "use cases." A team that read Cockburn says "ports and adapters."

They may all have the same codebase structure. The ability to recognize equivalences — and to explain "we're using Clean Architecture, which is essentially Onion with different naming" — signals architectural maturity.

More importantly: knowing Hexagonal helps you name the **direction** of adaptation. An HTTP controller is a driving adapter; a database is a driven adapter. This language helps in discussions about event-driven architecture: a message consumer (Kafka/RabbitMQ) is also a driving adapter. A message publisher is a driven adapter. The hexagon metaphor scales.
</Callout>
