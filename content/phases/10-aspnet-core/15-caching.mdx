## IMemoryCache: In-Process Caching

`IMemoryCache` stores data in the app's memory. It is fast (no serialization, no network) but does not survive restarts and is not shared across multiple instances.

```csharp
// Registration
builder.Services.AddMemoryCache(options =>
{
    options.SizeLimit = 1024;              // Max 1024 size units (you define what "1 unit" means)
    options.CompactionPercentage = 0.25;   // Remove 25% when limit is hit
    options.ExpirationScanFrequency = TimeSpan.FromMinutes(1);
});

// Usage
public class ProductService : IProductService
{
    private readonly IMemoryCache _cache;
    private readonly IProductRepository _repo;

    public ProductService(IMemoryCache cache, IProductRepository repo)
    {
        _cache = cache;
        _repo = repo;
    }

    public async Task<Product?> GetByIdAsync(int id)
    {
        var cacheKey = $"product:{id}";

        // GetOrCreate — atomic get-or-populate (not truly atomic under concurrency)
        return await _cache.GetOrCreateAsync(cacheKey, async entry =>
        {
            entry.AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(5);
            entry.SlidingExpiration = TimeSpan.FromMinutes(2);
            entry.Size = 1;             // Count toward SizeLimit
            entry.Priority = CacheItemPriority.Normal;

            // Register a callback when evicted
            entry.RegisterPostEvictionCallback((key, value, reason, state) =>
            {
                Console.WriteLine($"Cache evicted: {key}, reason: {reason}");
            });

            return await _repo.GetByIdAsync(id);
        });
    }

    public void Invalidate(int id)
    {
        _cache.Remove($"product:{id}");
    }
}
```

<Callout type="trap">
**Trap: GetOrCreateAsync Race Condition**

`GetOrCreateAsync` is NOT atomic. Under concurrent load, multiple threads can simultaneously miss the cache and all call the factory function — this is called **cache stampede** or **thundering herd**. For expensive operations:

```csharp
// Use SemaphoreSlim to serialize population of the same key
private readonly ConcurrentDictionary<string, SemaphoreSlim> _locks = new();

public async Task<Product?> GetByIdSafeAsync(int id)
{
    var key = $"product:{id}";

    if (_cache.TryGetValue(key, out Product? cached))
        return cached;

    var semaphore = _locks.GetOrAdd(key, _ => new SemaphoreSlim(1, 1));
    await semaphore.WaitAsync();
    try
    {
        // Double-check after acquiring lock
        if (_cache.TryGetValue(key, out cached))
            return cached;

        cached = await _repo.GetByIdAsync(id);
        _cache.Set(key, cached, TimeSpan.FromMinutes(5));
        return cached;
    }
    finally
    {
        semaphore.Release();
    }
}
```
</Callout>

## IDistributedCache: Multi-Server Caching

`IDistributedCache` abstracts Redis, SQL Server, and other distributed stores. All instances of your app share the same cache — essential for load-balanced deployments.

```csharp
// Registration — Redis (preferred)
builder.Services.AddStackExchangeRedisCache(options =>
{
    options.Configuration = builder.Configuration["Redis:ConnectionString"];
    options.InstanceName = "MyApp:"; // Key prefix to avoid collisions
});

// Registration — SQL Server (development/testing)
builder.Services.AddDistributedSqlServerCache(options =>
{
    options.ConnectionString = builder.Configuration["ConnectionStrings:Cache"];
    options.SchemaName = "dbo";
    options.TableName = "DistributedCache";
});

// Usage — requires manual serialization
public class SessionService : ISessionService
{
    private readonly IDistributedCache _cache;

    public SessionService(IDistributedCache cache) => _cache = cache;

    public async Task<UserSession?> GetSessionAsync(string sessionId, CancellationToken ct)
    {
        var bytes = await _cache.GetAsync($"session:{sessionId}", ct);
        if (bytes is null) return null;

        return JsonSerializer.Deserialize<UserSession>(bytes);
    }

    public async Task SetSessionAsync(string sessionId, UserSession session, CancellationToken ct)
    {
        var bytes = JsonSerializer.SerializeToUtf8Bytes(session);

        await _cache.SetAsync($"session:{sessionId}", bytes, new DistributedCacheEntryOptions
        {
            AbsoluteExpirationRelativeToNow = TimeSpan.FromHours(8),
            SlidingExpiration = TimeSpan.FromMinutes(30)
        }, ct);
    }

    public async Task InvalidateAsync(string sessionId, CancellationToken ct)
        => await _cache.RemoveAsync($"session:{sessionId}", ct);
}
```

## Output Caching (.NET 7+)

Output Caching caches entire HTTP responses at the middleware level — no changes to your action logic required:

```csharp
// Registration
builder.Services.AddOutputCache(options =>
{
    // Define reusable policies
    options.AddBasePolicy(policy => policy.Expire(TimeSpan.FromSeconds(10)));

    options.AddPolicy("Products", policy =>
        policy.Expire(TimeSpan.FromMinutes(5))
              .SetVaryByQuery("category", "page", "pageSize")
              .Tag("products")); // Tag for targeted invalidation

    options.AddPolicy("UserSpecific", policy =>
        policy.Expire(TimeSpan.FromMinutes(1))
              .SetVaryByHeader("Authorization") // Separate cache per user
              .SetVaryByRouteValue("userId"));
});

var app = builder.Build();
app.UseOutputCache(); // Before MapControllers

// Apply on Minimal API endpoints
app.MapGet("/products", async (IProductService svc) => await svc.GetAllAsync())
    .CacheOutput("Products");

app.MapGet("/products/{id:int}", async (int id, IProductService svc) =>
{
    var product = await svc.GetByIdAsync(id);
    return product is null ? Results.NotFound() : Results.Ok(product);
})
.CacheOutput(policy => policy.Expire(TimeSpan.FromMinutes(10)).Tag("products"));

// Apply on controller actions
[HttpGet]
[OutputCache(PolicyName = "Products")]
public async Task<IActionResult> GetAll() => Ok(await _service.GetAllAsync());
```

**Invalidation by tag (no more manual cache busting):**

```csharp
[HttpPost]
public async Task<IActionResult> Create(
    CreateProductRequest request,
    IOutputCacheStore cacheStore,
    CancellationToken ct)
{
    var product = await _service.CreateAsync(request);

    // Invalidate ALL cached responses tagged with "products"
    await cacheStore.EvictByTagAsync("products", ct);

    return CreatedAtAction(nameof(GetById), new { id = product.Id }, product);
}
```

<Callout type="senior">
**Senior Insight: Cache-Aside vs Output Caching — When to Use Each**

**Cache-aside** (IMemoryCache/IDistributedCache):
- You control what gets cached and when
- Cache complex domain objects, query results, computed aggregates
- You can invalidate specific keys on mutation
- Adds latency to first request (cache miss)

**Output caching** (Output Cache middleware):
- Framework caches entire HTTP responses for you
- Zero changes to action logic
- Vary by route, query, headers (including Authorization for user-specific caches)
- Tag-based invalidation in .NET 7+
- Cannot cache responses containing sensitive real-time data

The real-world pattern for high-traffic APIs:
1. Output cache for public list endpoints (products, categories)
2. Cache-aside for expensive computed results (recommendations, analytics)
3. Distributed cache for session and cross-service shared data

Never cache responses with `Authorization` headers visible in the cache key without also using `SetVaryByHeader("Authorization")` — otherwise User A sees User B's data.
</Callout>

<MythBuster
  myth="Adding caching to every endpoint is always a performance improvement"
  truth="Over-caching creates stale data bugs, hard-to-debug inconsistencies, and increased memory pressure. Cache only when: (1) the data is expensive to compute, (2) it changes infrequently, and (3) brief staleness is acceptable. Never cache user-specific data without per-user cache keys, and never cache responses containing sensitive real-time data like account balances."
/>
