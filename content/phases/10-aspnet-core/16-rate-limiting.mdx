## The Four Rate Limiting Algorithms

ASP.NET Core 7 ships `RateLimiterMiddleware` with four built-in algorithms. Each has different behavior under burst traffic:

<ComparisonTable
  headers={["Algorithm", "Burst Handling", "Smoothness", "Typical Use"]}
  rows={[
    ["Fixed Window", "Allows full burst at window start", "Uneven — resets sharply", "Simple per-user limits"],
    ["Sliding Window", "Smooths window boundary spikes", "Moderate", "More accurate request counting"],
    ["Token Bucket", "Allows bursts up to bucket size", "Good — refills gradually", "APIs with bursty-but-bounded clients"],
    ["Concurrency Limiter", "Limits simultaneous requests (not rate)", "N/A", "Protecting expensive resources"],
  ]}

## Fixed Window Limiter

Allows N requests in a fixed time window. Resets at the window boundary:

```csharp
builder.Services.AddRateLimiter(options =>
{
    options.AddFixedWindowLimiter("fixed", config =>
    {
        config.Window = TimeSpan.FromMinutes(1);
        config.PermitLimit = 100;         // 100 requests per minute
        config.QueueProcessingOrder = QueueProcessingOrder.OldestFirst;
        config.QueueLimit = 0;            // No queuing — reject immediately
    });

    // Response when rate limit is hit
    options.OnRejected = async (context, token) =>
    {
        context.HttpContext.Response.StatusCode = StatusCodes.Status429TooManyRequests;
        context.HttpContext.Response.Headers["Retry-After"] =
            context.Lease.TryGetMetadata(MetadataName.RetryAfter, out var retryAfter)
                ? ((int)retryAfter.TotalSeconds).ToString()
                : "60";

        await context.HttpContext.Response.WriteAsJsonAsync(new
        {
            error = "Too Many Requests",
            message = "Rate limit exceeded. Please slow down."
        }, token);
    };
});
```

**Problem with Fixed Window:** If your window resets at :00, a client can make 100 requests at :59 and 100 more at :01, effectively getting 200 requests in 2 seconds.

## Sliding Window Limiter

Tracks requests in segments of the window to smooth out boundary spikes:

```csharp
options.AddSlidingWindowLimiter("sliding", config =>
{
    config.Window = TimeSpan.FromMinutes(1);
    config.SegmentsPerWindow = 6;         // 6 segments = 10-second resolution
    config.PermitLimit = 100;             // 100 requests per minute total
    config.QueueLimit = 10;              // Queue up to 10 requests
    config.QueueProcessingOrder = QueueProcessingOrder.NewestFirst;
});
```

---

## Token Bucket Limiter

Tokens are added at a steady rate. Bursts are allowed up to the bucket size:

```csharp
options.AddTokenBucketLimiter("token-bucket", config =>
{
    config.TokenLimit = 50;              // Bucket holds max 50 tokens
    config.ReplenishmentPeriod = TimeSpan.FromSeconds(10);
    config.TokensPerPeriod = 10;         // Add 10 tokens every 10 seconds
    config.AutoReplenishment = true;     // Replenish automatically
    config.QueueLimit = 5;
    config.QueueProcessingOrder = QueueProcessingOrder.OldestFirst;
});
```

**Token Bucket is excellent for APIs** where legitimate clients occasionally burst (e.g., upload multiple files) but should not sustain high rates indefinitely.

---

## Concurrency Limiter

Limits simultaneous in-flight requests — not the rate. Useful for protecting expensive resources:

```csharp
options.AddConcurrencyLimiter("concurrency", config =>
{
    config.PermitLimit = 10;             // Max 10 simultaneous requests
    config.QueueLimit = 5;              // Queue 5 more — others get 429
    config.QueueProcessingOrder = QueueProcessingOrder.OldestFirst;
});
```

## Per-User Rate Limiting with Policy Partitioning

The real value of rate limiting is per-user or per-IP limits — not global limits:

```csharp
builder.Services.AddRateLimiter(options =>
{
    // Partition by authenticated user ID, fall back to IP for anonymous
    options.AddPolicy("PerUser", context =>
    {
        var userId = context.User.FindFirstValue(ClaimTypes.NameIdentifier);

        if (userId is not null)
        {
            // Authenticated users get higher limits
            return RateLimitPartition.GetTokenBucketLimiter(
                partitionKey: $"user:{userId}",
                factory: _ => new TokenBucketRateLimiterOptions
                {
                    TokenLimit = 200,
                    ReplenishmentPeriod = TimeSpan.FromMinutes(1),
                    TokensPerPeriod = 200,
                    AutoReplenishment = true
                });
        }

        // Anonymous users limited by IP
        var ip = context.Connection.RemoteIpAddress?.ToString() ?? "unknown";
        return RateLimitPartition.GetFixedWindowLimiter(
            partitionKey: $"ip:{ip}",
            factory: _ => new FixedWindowRateLimiterOptions
            {
                Window = TimeSpan.FromMinutes(1),
                PermitLimit = 20             // Anonymous: 20 req/min
            });
    });

    // Different limits for different tiers
    options.AddPolicy("PremiumUsers", context =>
    {
        var tier = context.User.FindFirstValue("subscription_tier");
        var userId = context.User.FindFirstValue(ClaimTypes.NameIdentifier) ?? "anon";

        return tier switch
        {
            "enterprise" => RateLimitPartition.GetNoLimiter("enterprise"),
            "premium" => RateLimitPartition.GetTokenBucketLimiter(
                $"premium:{userId}",
                _ => new TokenBucketRateLimiterOptions
                {
                    TokenLimit = 1000,
                    TokensPerPeriod = 1000,
                    ReplenishmentPeriod = TimeSpan.FromHours(1),
                    AutoReplenishment = true
                }),
            _ => RateLimitPartition.GetFixedWindowLimiter(
                $"free:{userId}",
                _ => new FixedWindowRateLimiterOptions
                {
                    PermitLimit = 100,
                    Window = TimeSpan.FromHours(1)
                })
        };
    });
});
```

**Apply to endpoints:**

```csharp
var app = builder.Build();

app.UseRateLimiter(); // Must be AFTER UseRouting, BEFORE MapControllers

// Apply policy to all endpoints
app.MapControllers().RequireRateLimiting("PerUser");

// Apply to specific Minimal API endpoints
app.MapGet("/expensive-report", async (IReportService svc) => await svc.GenerateAsync())
    .RequireRateLimiting("PremiumUsers");

// Disable for health checks
app.MapHealthChecks("/healthz").DisableRateLimiting();
```

**Apply in controllers:**

```csharp
[ApiController]
[EnableRateLimiting("PerUser")]        // Applies to all actions
public class OrdersController : ControllerBase
{
    [HttpGet]
    [EnableRateLimiting("PremiumUsers")] // Overrides controller-level for this action

    [HttpDelete("{id}")]
    [DisableRateLimiting]               // No rate limiting for this action
    public IActionResult Delete(int id) => Ok();
}
```

<Callout type="senior">
**Senior Insight: Rate Limiting Headers for Client Observability**

Clients need feedback to implement exponential backoff and user-facing messages. Add standard rate limit headers to every response:

```csharp
options.OnRejected = async (context, token) =>
{
    context.HttpContext.Response.StatusCode = StatusCodes.Status429TooManyRequests;

    if (context.Lease.TryGetMetadata(MetadataName.RetryAfter, out var retryAfter))
        context.HttpContext.Response.Headers["Retry-After"] =
            ((int)retryAfter.TotalSeconds).ToString();

    await context.HttpContext.Response.WriteAsJsonAsync(new ProblemDetails
    {
        Status = 429,
        Title = "Too Many Requests",
        Detail = "You have exceeded your request quota."
    }, token);
};

// Add informational headers on successful requests (custom middleware)
app.Use(async (context, next) =>
{
    await next(context);
    // Add headers after response — these are informational only
    // Real quota headers require access to the rate limiter lease metadata
    // Use a custom policy that exposes this via HttpContext.Items
});
```

Also note: in Kubernetes or behind a load balancer, rate limiting state must be in Redis — not in-memory per-instance. The built-in algorithms are in-memory only. For distributed rate limiting, use Redis with a StackExchange.Redis-backed custom `RateLimiter` or a library like `AspNetCoreRateLimit` with Redis backing.
</Callout>

  myth="Adding rate limiting middleware protects against all forms of API abuse"
  truth="Rate limiting is one layer of protection. It helps against accidental abuse and simple scripts, but determined attackers can distribute requests across IPs, use proxies, or target specific expensive operations below your limits.
