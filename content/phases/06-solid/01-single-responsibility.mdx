## What Is the Single Responsibility Principle?

Robert C. Martin's definition: **a class should have only one reason to change.**

The key word is *reason*. Not "one method," not "one field" — one *reason to change*, which maps to one *stakeholder* or *business concern* that can cause that class to be modified.

Think of it this way: if two different people in your organization could independently ask you to change the same class for unrelated reasons, that class has too many responsibilities.

---

## The Classic Violation

```csharp
// BAD: This class has at least THREE reasons to change:
// 1. Business logic changes (how invoices are calculated)
// 2. Data persistence changes (switching from SQL to a document DB)
// 3. Reporting/formatting changes (PDF instead of HTML)
public class Invoice
{
    public List<LineItem> LineItems { get; set; } = new();

    // Reason 1: Business logic
    public decimal CalculateTotal()
    {
        decimal subtotal = LineItems.Sum(li => li.Quantity * li.UnitPrice);
        decimal tax = subtotal * 0.2m;
        return subtotal + tax;
    }

    // Reason 2: Persistence
    public void SaveToDatabase(string connectionString)
    {
        using var conn = new SqlConnection(connectionString);
        conn.Open();
        // ... INSERT INTO Invoices ...
        foreach (var item in LineItems)
        {
            // ... INSERT INTO LineItems ...
        }
    }

    // Reason 3: Formatting / reporting
    public string GenerateHtmlReport()
    {
        var sb = new StringBuilder("<html><body><table>");
        foreach (var item in LineItems)
        {
            sb.Append($"<tr><td>{item.Name}</td><td>{item.UnitPrice:C}</td></tr>");
        }
        sb.Append($"<tr><td>Total</td><td>{CalculateTotal():C}</td></tr>");
        sb.Append("</table></body></html>");
        return sb.ToString();
    }
}
```

**The problem:** every time the tax calculation changes, the DBA schema changes, or the report format changes, you touch the same class — and risk breaking the other two concerns.

## The Refactored Version

```csharp
// GOOD: Three focused classes, each with one reason to change

// Responsibility 1: Business logic only
public class Invoice
{
    public int Id { get; set; }
    public List<LineItem> LineItems { get; set; } = new();

    public decimal CalculateSubtotal() =>
        LineItems.Sum(li => li.Quantity * li.UnitPrice);

    public decimal CalculateTax(decimal rate = 0.2m) =>
        CalculateSubtotal() * rate;

    public decimal CalculateTotal() =>
        CalculateSubtotal() + CalculateTax();
}

// Responsibility 2: Persistence
public class InvoiceRepository
{
    private readonly string _connectionString;

    public InvoiceRepository(string connectionString)
        => _connectionString = connectionString;

    public void Save(Invoice invoice)
    {
        using var conn = new SqlConnection(_connectionString);
        conn.Open();
        // INSERT logic here
    }

    public Invoice? GetById(int id)
    {
        // SELECT logic here
        return null;
    }
}

// Responsibility 3: Formatting / reporting
public class InvoiceHtmlFormatter
{
    public string Format(Invoice invoice)
    {
        var sb = new StringBuilder("<html><body><table>");
        foreach (var item in invoice.LineItems)
        {
            sb.Append($"<tr><td>{item.Name}</td><td>{item.UnitPrice:C}</td></tr>");
        }
        sb.Append($"<tr><td><strong>Total</strong></td><td>{invoice.CalculateTotal():C}</td></tr>");
        sb.Append("</table></body></html>");
        return sb.ToString();
    }
}
```

Now each class has **one reason to change**: a business rule change touches `Invoice`, a schema migration touches `InvoiceRepository`, and a redesign of the report touches `InvoiceHtmlFormatter`.

## Cohesion vs. Coupling

These two metrics are the practical tools for detecting SRP violations:

**Cohesion** — how closely related are the things inside a class? High cohesion = good. Low cohesion = SRP violation.

**Coupling** — how many other classes does this class depend on? Low coupling = good. High coupling = fragile design.

SRP directly improves cohesion. By splitting concerns into separate classes, each class becomes tightly focused and its internals all relate to the same concept.

```csharp
// LOW COHESION — unrelated methods crammed together
public class UserManager
{
    public User GetUser(int id) { /* ... */ return null!; }
    public void SendWelcomeEmail(User user) { /* SMTP logic */ }
    public string FormatUserForDisplay(User user) => $"{user.FirstName} {user.LastName}";
    public void WriteAuditLog(string message) { /* file I/O */ }
    public bool ValidatePassword(string password) { /* regex */ return true; }
}

// HIGH COHESION — each class is tightly focused
public class UserRepository   { /* only data access */         }
public class EmailService     { /* only email sending */       }
public class UserFormatter    { /* only display formatting */  }
public class AuditLogger      { /* only audit logging */       }
public class PasswordValidator{ /* only password rules */      }
```

  reality="SRP is about one *reason to change*, not one method. An Invoice class can have CalculateSubtotal(), CalculateTax(), and CalculateTotal() — all related to the same business concern. It is the mixing of unrelated concerns (calculation + persistence + formatting) that violates SRP."

## Real-World Smell: The God Class

The most common SRP violation in real codebases is the **God Class** — a class that "knows too much" and does too much. Spotting them:

- Class name contains words like `Manager`, `Handler`, `Helper`, `Utility`, `Service` that are so generic they describe nothing
- Class has 20+ methods spanning multiple domains
- Unit testing requires mocking 6+ dependencies
- Merge conflicts happen in this file multiple times a week

```csharp
// Smell: "OrderService" doing everything
public class OrderService
{
    // Order creation
    public Order CreateOrder(Cart cart) { /* ... */ return null!; }

    // Payment processing (should be PaymentService)
    public bool ChargeCard(Order order, CreditCard card) { /* Stripe API */ return true; }

    // Inventory (should be InventoryService)
    public void ReserveStock(Order order) { /* ... */ }

    // Email (should be NotificationService)
    public void SendConfirmationEmail(Order order) { /* SMTP */ }

    // Shipping (should be ShippingService)
    public string GenerateShippingLabel(Order order) { /* FedEx API */ return ""; }

    // Reporting (should be ReportingService)
    public decimal GetRevenueForPeriod(DateTime from, DateTime to) { /* SQL */ return 0m; }
}
```

<Callout type="senior">
**Senior Insight: SRP at Every Scale**

SRP applies at multiple levels of abstraction simultaneously:

- **Method level**: a method that both validates input and sends an email violates SRP
- **Class level**: the classic Invoice example above
- **Module/project level**: a single project that handles API routing, business logic, and database migration has too many reasons to change
- **Microservice level**: a "UserService" that handles authentication, profiles, billing, and notifications is a distributed monolith

In practice, the hardest SRP decisions are about *granularity*. Split too aggressively and you get an explosion of tiny, context-free classes that are hard to navigate. The heuristic senior engineers use: **split when you anticipate independent rates of change**. If the business logic and the persistence never change for different reasons, keeping them together might be pragmatic — until they do.

In .NET specifically, the SRP is what drives the layered architecture that ASP.NET Core applications use: Controllers (handle HTTP), Services (business logic), Repositories (data access), and Domain models (business rules) each have one primary reason to change.
</Callout>
