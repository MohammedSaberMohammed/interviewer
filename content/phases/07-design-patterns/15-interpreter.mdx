## The Problem: Evaluating Structured Language

You have a search query language: `"age > 18 AND city = 'London'"`. You need to parse it and evaluate it against objects. Hard-coding logic for every possible combination doesn't scale.

Interpreter represents the grammar as a class hierarchy where each class is a grammar rule, and `Interpret()` evaluates it.

---

## Classic Interpreter

```csharp
// Context — holds the variables being evaluated
public class RuleContext
{
    private readonly Dictionary<string, object> _variables = new();

    public void Set(string name, object value) => _variables[name] = value;
    public object Get(string name) => _variables[name];
}

// Abstract expression — all grammar rules implement this
public interface IExpression
{
    bool Interpret(RuleContext ctx);
}

// Terminal expressions (leaves)
public class GreaterThanExpression : IExpression
{
    private readonly string _field;
    private readonly int    _value;

    public GreaterThanExpression(string field, int value)
    {
        _field = field;
        _value = value;
    }

    public bool Interpret(RuleContext ctx)
        => Convert.ToInt32(ctx.Get(_field)) > _value;
}

public class EqualsExpression : IExpression
{
    private readonly string _field;
    private readonly string _value;

    public EqualsExpression(string field, string value)
    {
        _field = field;
        _value = value;
    }

    public bool Interpret(RuleContext ctx)
        => ctx.Get(_field).ToString() == _value;
}

// Non-terminal expressions (composites)
public class AndExpression : IExpression
{
    private readonly IExpression _left, _right;

    public AndExpression(IExpression left, IExpression right)
    {
        _left  = left;
        _right = right;
    }

    public bool Interpret(RuleContext ctx) =>
        _left.Interpret(ctx) && _right.Interpret(ctx);
}

public class OrExpression : IExpression
{
    private readonly IExpression _left, _right;
    public OrExpression(IExpression left, IExpression right)
    {
        _left = left; _right = right;
    }
    public bool Interpret(RuleContext ctx) =>
        _left.Interpret(ctx) || _right.Interpret(ctx);
}

// Build expression tree for: age > 18 AND city = 'London'
IExpression rule = new AndExpression(
    left:  new GreaterThanExpression("age", 18),
    right: new EqualsExpression("city", "London"));

// Evaluate against different contexts
var ctx1 = new RuleContext();
ctx1.Set("age", 25);
ctx1.Set("city", "London");
Console.WriteLine(rule.Interpret(ctx1)); // True

var ctx2 = new RuleContext();
ctx2.Set("age", 15);
ctx2.Set("city", "London");
Console.WriteLine(rule.Interpret(ctx2)); // False (age <= 18)
```

## Real .NET Example: LINQ Expression Trees

LINQ expression trees ARE the Interpreter pattern built into .NET. When you write a LINQ query against `IQueryable<T>`, C# builds an expression tree (not IL code):

```csharp
// This lambda is compiled to an Expression<Func<Order, bool>> — a tree, not IL
Expression<Func<Order, bool>> predicate = o => o.Total > 100 && o.Status == "Active";

// The expression tree is a data structure you can inspect
BinaryExpression andExpr   = (BinaryExpression)predicate.Body;
BinaryExpression leftExpr  = (BinaryExpression)andExpr.Left;   // o.Total > 100
BinaryExpression rightExpr = (BinaryExpression)andExpr.Right;  // o.Status == "Active"

Console.WriteLine(leftExpr.NodeType);  // GreaterThan
Console.WriteLine(rightExpr.NodeType); // Equal

// EF Core INTERPRETS the expression tree to generate SQL
var orders = await dbContext.Orders
    .Where(predicate) // EF reads the tree, produces: WHERE Total > 100 AND Status = 'Active'
    .ToListAsync();

// You can build expression trees dynamically at runtime
ParameterExpression param       = Expression.Parameter(typeof(Order), "o");
MemberExpression    totalProp   = Expression.Property(param, "Total");
ConstantExpression  threshold   = Expression.Constant(100m);
BinaryExpression    comparison  = Expression.GreaterThan(totalProp, threshold);

Expression<Func<Order, bool>> dynamic =
    Expression.Lambda<Func<Order, bool>>(comparison, param);

var bigOrders = await dbContext.Orders.Where(dynamic).ToListAsync();
```

## Mini Search DSL

```csharp
// A simple search query parser that builds an interpreter tree
// Input: "name:Alice age:>25 active:true"
public class SearchQueryParser
{
    public IExpression Parse(string query)
    {
        var parts = query.Split(' ', StringSplitOptions.RemoveEmptyEntries);
        var expressions = parts.Select(ParsePart).ToList();

        // Combine all parts with AND
        return expressions.Count == 1
            ? expressions[0]
            : expressions.Aggregate((acc, expr) => new AndExpression(acc, expr));
    }

    private IExpression ParsePart(string part)
    {
        var colonIdx = part.IndexOf(':');
        string field = part[..colonIdx];
        string value = part[(colonIdx + 1)..];

        if (value.StartsWith(">"))
            return new GreaterThanExpression(field, int.Parse(value[1..]));
        return new EqualsExpression(field, value);
    }
}

// Usage
var parser = new SearchQueryParser();
IExpression query = parser.Parse("city:London age:>18");

var ctx = new RuleContext();
ctx.Set("city", "London");
ctx.Set("age", 30);
Console.WriteLine(query.Interpret(ctx)); // True
```

  title="Expression Tree vs Compiled Lambda"
  description="What is the key difference between Expression<Func<T, bool>> and Func<T, bool> in LINQ?"
  code={`// A — Expression tree
Expression<Func\<Order, bool\>> expr = o => o.Total > 100;
var results1 = dbContext.Orders.Where(expr).ToList();

Func\<Order, bool\> func = o => o.Total > 100;
var results2 = dbContext.Orders.Where(func).ToList(); // AsQueryable falls back to IEnumerable`}

    "No difference — EF Core handles both the same way",
    "expr is a data structure that EF Core translates to SQL; func is compiled IL that runs in-memory on ALL loaded records",

    "Both translate to SQL — the difference is only syntax"

<Callout type="senior">
**Building Dynamic Query Composers with Expression Trees**

A powerful real-world use: building `WHERE` clauses dynamically from filter DTOs, avoiding SQL injection and maintaining type safety:

```csharp
public static class QueryableExtensions
{
    // Build a predicate from a filter DTO using expression tree composition
    public static IQueryable<Order> ApplyFilter(
        this IQueryable<Order> query, OrderFilterDto filter)
    {
        if (filter.MinTotal.HasValue)
        {
            decimal min = filter.MinTotal.Value;
            query = query.Where(o => o.Total >= min);
        }

        if (!string.IsNullOrEmpty(filter.Status))
        {
            string status = filter.Status;
            query = query.Where(o => o.Status == status);
        }

        if (filter.CustomerId.HasValue)
        {
            int cid = filter.CustomerId.Value;
            query = query.Where(o => o.CustomerId == cid);
        }

        return query; // EF combines all expressions into one SQL WHERE clause
    }
}

// Usage
var orders = await dbContext.Orders
    .ApplyFilter(new OrderFilterDto { MinTotal = 100, Status = "Active" })
    .OrderBy(o => o.CreatedAt)
    .Take(50)
    .ToListAsync();
// SQL: SELECT TOP 50 ... WHERE Total >= 100 AND Status = 'Active' ORDER BY CreatedAt
```

This pattern — composing `IQueryable<T>` with expression trees — is more readable and maintainable than string-based SQL builders, and is fully type-safe.
</Callout>
