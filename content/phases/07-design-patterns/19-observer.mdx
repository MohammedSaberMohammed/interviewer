## The Problem: Keeping Dependents in Sync

When an order is placed, multiple parts of your system need to react: send a confirmation email, update inventory, notify the shipping system, record an analytics event. Hard-coding calls to all of them from the order service creates tight coupling — adding a new reactor requires modifying the publisher.

Observer decouples publishers from subscribers: publishers emit events, subscribers register and handle them independently.

## C# Events — Built-in Observer

```csharp
// Subject (Observable) — raises events
public class OrderService
{
    // Event declaration — the subscription mechanism
    public event EventHandler<OrderPlacedEventArgs>? OrderPlaced;

    public async Task PlaceOrderAsync(PlaceOrderDto dto)
    {
        // ... create the order ...
        var order = new Order { Id = 42, CustomerId = dto.CustomerId };

        // Notify all observers
        OnOrderPlaced(new OrderPlacedEventArgs(order));
    }

    protected virtual void OnOrderPlaced(OrderPlacedEventArgs e)
        => OrderPlaced?.Invoke(this, e);
}

public class OrderPlacedEventArgs : EventArgs
{
    public Order Order { get; }
    public OrderPlacedEventArgs(Order order) => Order = order;
}

// Observers — subscribe to the event
var orderService = new OrderService();

orderService.OrderPlaced += async (sender, e) =>
    await emailService.SendConfirmationAsync(e.Order.Id);

orderService.OrderPlaced += async (sender, e) =>
    await inventoryService.ReserveAsync(e.Order.Items);

orderService.OrderPlaced += (sender, e) =>
    analytics.Track("order_placed", new { e.Order.Id, e.Order.Total });

// When PlaceOrderAsync runs, ALL subscribers are notified
```

## `INotifyPropertyChanged` — Observer in WPF/MAUI/Blazor

The MVVM pattern uses Observer for UI binding:

```csharp
// ViewModel implements INotifyPropertyChanged
public class OrderViewModel : INotifyPropertyChanged
{
    public event PropertyChangedEventHandler? PropertyChanged;

    private string _status = "Pending";
    public string Status
    {
        get => _status;
        set
        {
            if (_status == value) return;
            _status = value;
            PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(nameof(Status)));
            // UI controls observing this property automatically re-render
        }
    }
}

// Binding in MAUI/Blazor/WPF — UI is the subscriber
// <Label Text="{Binding Status}" />
// When Status changes, the label automatically updates — Observer pattern
```

## `IObservable<T>` — Push-Based Reactive Streams

`IObservable<T>` is the formal Observer interface in the BCL. Reactive Extensions (Rx.NET) builds on it:

```csharp
// IObservable<T> = subject that pushes data (producer)
// IObserver<T>   = subscriber that receives data (consumer)

public interface IObservable<out T>
{
    IDisposable Subscribe(IObserver<T> observer);
}

public interface IObserver<in T>
{
    void OnNext(T value);      // new item
    void OnError(Exception ex); // error occurred
    void OnCompleted();         // stream ended
}

// Using System.Reactive (Rx.NET)
// Real-time stock price stream
IObservable<decimal> priceStream = Observable.Interval(TimeSpan.FromSeconds(1))
    .Select(_ => GetCurrentPrice("MSFT"));

// Subscriber — process only significant changes
IDisposable subscription = priceStream
    .DistinctUntilChanged()              // skip unchanged prices
    .Where(price => price > 400m)        // only prices above $400
    .Buffer(TimeSpan.FromMinutes(1))     // batch into 1-minute windows
    .Subscribe(
        prices => Console.WriteLine($"High price window: {prices.Max():C}"),
        error  => Console.WriteLine($"Stream error: {error.Message}"),
        ()     => Console.WriteLine("Stream completed"));

// Later — unsubscribe
subscription.Dispose();
```

## Domain Events — Observer for DDD

In Domain-Driven Design, domain events use Observer to decouple domain logic from side effects:

```csharp
// Domain event
public record OrderPlacedDomainEvent(Order Order) : INotification;

// Domain entity raises events
public class Order
{
    private readonly List<INotification> _domainEvents = new();
    public IReadOnlyList<INotification> DomainEvents => _domainEvents;

    public void Place()
    {
        Status = "Placed";
        _domainEvents.Add(new OrderPlacedDomainEvent(this));
    }

    public void ClearDomainEvents() => _domainEvents.Clear();
}

// Infrastructure dispatches events after SaveChanges
public class AppDbContext : DbContext
{
    private readonly IMediator _mediator;

    public override async Task<int> SaveChangesAsync(CancellationToken ct = default)
    {
        int result = await base.SaveChangesAsync(ct);

        // Dispatch domain events after successful persist
        var events = ChangeTracker.Entries<Order>()
            .SelectMany(e => e.Entity.DomainEvents)
            .ToList();

        foreach (var evt in events)
            await _mediator.Publish(evt, ct);

        return result;
    }
}
```

<Callout type="senior">
**Weak Events — Preventing Leaks Without Manual Unsubscription**

WPF and MAUI provide WeakEventManager to break the strong reference chain:

```csharp
// WeakEventManager holds weak references to subscribers
// The subscriber can be GC'd even while subscribed
WeakEventManager<EventAggregator, DataChangedArgs>.AddHandler(
    aggregator,
    nameof(EventAggregator.DataChanged),
    OnDataChanged);

// Rx.NET approach — composition makes unsubscription automatic
// Using .TakeUntil() to automatically unsubscribe when a signal fires
var viewDestroyed = new Subject<Unit>();

Observable.FromEventPattern<DataChangedArgs>(
    h => aggregator.DataChanged += h,
    h => aggregator.DataChanged -= h)
    .TakeUntil(viewDestroyed) // auto-unsubscribe when view is destroyed
    .Subscribe(e => Console.WriteLine("Data changed"));

// When view is destroyed:
viewDestroyed.OnNext(Unit.Default); // all subscriptions auto-cleaned up

// The IDisposable pattern is the safest:
public class DataView : IDisposable
{
    private readonly EventAggregator _agg;

    public DataView(EventAggregator agg)
    {
        _agg = agg;
        _agg.DataChanged += OnDataChanged;
    }

    private void OnDataChanged(object? s, DataChangedArgs e) { /* ... */ }

    public void Dispose() => _agg.DataChanged -= OnDataChanged;
}
```

**Interview talking point**: Event memory leaks are one of the most common C# bugs in long-running applications. Always unsubscribe in Dispose() or use weak event patterns.
</Callout>

  options={[
