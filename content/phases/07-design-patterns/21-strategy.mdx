## The Problem: Hard-Coded Algorithms

When you embed a sorting or calculation algorithm directly in a class, switching to a different algorithm means modifying the class — violating OCP. Strategy extracts the algorithm behind an interface so it can be swapped without touching the context.

```csharp
// Without Strategy — hard-coded sorting, can't change without modifying class
public class ReportGenerator
{
    public List<Order> GenerateSortedReport(List<Order> orders)
    {
        // Hard-coded: always sort by date
        return orders.OrderBy(o => o.Date).ToList();
    }
}
```

## Strategy Pattern Structure

```csharp
// Strategy interface
public interface ISortStrategy<T>
{
    IEnumerable<T> Sort(IEnumerable<T> items);
}

// Concrete strategies
public class SortByDateAscending : ISortStrategy<Order>
{
    public IEnumerable<Order> Sort(IEnumerable<Order> orders)
        => orders.OrderBy(o => o.Date);
}

public class SortByTotalDescending : ISortStrategy<Order>
{
    public IEnumerable<Order> Sort(IEnumerable<Order> orders)
        => orders.OrderByDescending(o => o.Total);
}

public class SortByCustomerName : ISortStrategy<Order>
{
    public IEnumerable<Order> Sort(IEnumerable<Order> orders)
        => orders.OrderBy(o => o.CustomerName, StringComparer.OrdinalIgnoreCase);
}

// Context — uses whichever strategy is injected
public class ReportGenerator
{
    private ISortStrategy<Order> _sortStrategy;

    public ReportGenerator(ISortStrategy<Order> sortStrategy)
        => _sortStrategy = sortStrategy;

    // Strategy can be changed at runtime
    public void SetSortStrategy(ISortStrategy<Order> strategy)
        => _sortStrategy = strategy;

    public List<Order> GenerateSortedReport(List<Order> orders)
        => _sortStrategy.Sort(orders).ToList();
}

// Usage — inject strategy
var generator = new ReportGenerator(new SortByTotalDescending());
var report = generator.GenerateSortedReport(orders);

// Swap strategy at runtime
generator.SetSortStrategy(new SortByDateAscending());
```

## Modern C# Alternative: `Func<T, TKey>` Delegates

For simple strategies, delegates eliminate the need for strategy classes entirely:

```csharp
public class ReportGenerator
{
    public List<Order> GenerateSortedReport(
        IEnumerable<Order> orders,
        Func<IEnumerable<Order>, IEnumerable<Order>> sortStrategy)
            => sortStrategy(orders).ToList();
}

// Usage — lambdas as strategies (no classes needed)
var generator = new ReportGenerator();
var byTotal  = generator.GenerateSortedReport(orders, o => o.OrderByDescending(x => x.Total));
var byDate   = generator.GenerateSortedReport(orders, o => o.OrderBy(x => x.Date));
```

## Real .NET BCL Example: `IComparer<T>`

`IComparer<T>` is the classic Strategy interface in the BCL:

```csharp
// Custom comparison strategies
public class OrderByTotalDescending : IComparer<Order>
{
    public int Compare(Order? x, Order? y) =>
        Comparer<decimal>.Default.Compare(y?.Total ?? 0, x?.Total ?? 0);
}

public class OrderByCustomer : IComparer<Order>
{
    public int Compare(Order? x, Order? y) =>
        string.Compare(x?.CustomerName, y?.CustomerName, StringComparison.OrdinalIgnoreCase);
}

// The sort algorithm (Array.Sort, List.Sort) is the Context
// The IComparer<T> implementation is the Strategy
var orders = new List<Order> { /* ... */ };

orders.Sort(new OrderByTotalDescending()); // inject strategy

// Or use Comparer<T>.Create() for inline strategies (C# 7+)
orders.Sort(Comparer<Order>.Create((x, y) =>
    string.Compare(x.CustomerName, y.CustomerName, StringComparison.Ordinal)));

// LINQ's OrderBy also accepts IComparer<T>
var sorted = orders.OrderBy(o => o.CustomerName, StringComparer.OrdinalIgnoreCase);
```

`IEqualityComparer<T>` is the same pattern for equality checking — used by `Dictionary<TKey, TValue>`, `HashSet<T>`, LINQ's `GroupBy`/`Distinct`:

```csharp
// Custom equality strategy for case-insensitive email matching
public class EmailEqualityComparer : IEqualityComparer<User>
{
    public bool Equals(User? x, User? y) =>
        string.Equals(x?.Email, y?.Email, StringComparison.OrdinalIgnoreCase);

    public int GetHashCode(User obj) =>
        obj.Email.ToLowerInvariant().GetHashCode();
}

var uniqueUsers = users.Distinct(new EmailEqualityComparer()).ToList();
var groupedByEmail = users.GroupBy(u => u.Email, StringComparer.OrdinalIgnoreCase);
```

## Pricing Rules — Real-World Strategy

```csharp
public interface IDiscountStrategy
{
    decimal CalculateDiscount(Order order);
}

public class NoDiscount : IDiscountStrategy
{
    public decimal CalculateDiscount(Order order) => 0;
}

public class PercentageDiscount : IDiscountStrategy
{
    private readonly decimal _percent;
    public PercentageDiscount(decimal percent) => _percent = percent;
    public decimal CalculateDiscount(Order order) => order.Total * (_percent / 100);
}

public class VolumeDiscount : IDiscountStrategy
{
    public decimal CalculateDiscount(Order order) =>
        order.ItemCount > 10 ? order.Total * 0.15m : 0;
}

public class LoyaltyDiscount : IDiscountStrategy
{
    private readonly int _loyaltyYears;
    public LoyaltyDiscount(int years) => _loyaltyYears = years;
    public decimal CalculateDiscount(Order order) =>
        _loyaltyYears > 3 ? order.Total * 0.20m : order.Total * 0.05m;
}

// Register strategies in DI by key (.NET 8+)
builder.Services.AddKeyedTransient<IDiscountStrategy, NoDiscount>("none");
builder.Services.AddKeyedTransient<IDiscountStrategy, VolumeDiscount>("volume");
builder.Services.AddKeyedSingleton<IDiscountStrategy>("percent10", (_, _) => new PercentageDiscount(10));
```

<Callout type="senior">
**Strategy + DI = Extensible Business Rules**

Combining Strategy with DI makes business rules plug-in-able without modifying the core system:

```csharp
// Multiple validators as strategies
public interface IOrderValidator
{
    ValidationResult Validate(Order order);
}

// Register all validators — they'll ALL be injected as IEnumerable<IOrderValidator>
builder.Services.AddTransient<IOrderValidator, StockAvailabilityValidator>();
builder.Services.AddTransient<IOrderValidator, CustomerCreditValidator>();
builder.Services.AddTransient<IOrderValidator, FraudDetectionValidator>();
builder.Services.AddTransient<IOrderValidator, GeographicRestrictionValidator>();

// Composite strategy — runs all registered validators
public class OrderValidationService
{
    private readonly IEnumerable<IOrderValidator> _validators;

    public OrderValidationService(IEnumerable<IOrderValidator> validators)
        => _validators = validators;

    public IEnumerable<ValidationError> Validate(Order order) =>
        _validators
            .Select(v => v.Validate(order))
            .Where(r => !r.IsValid)
            .SelectMany(r => r.Errors);
}
```

Adding a new validation rule is a single line in DI registration — no modification to OrderValidationService. This is Strategy + Open/Closed Principle at its most practical.
</Callout>
