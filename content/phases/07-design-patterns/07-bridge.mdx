## The Problem: Class Explosion

Imagine a `Shape` with two dimensions of variation: shape type (Circle, Square) and rendering platform (Vector, Raster). Solving this with inheritance produces a combinatorial explosion:

```
CircleVectorRenderer
CircleRasterRenderer
SquareVectorRenderer
SquareRasterRenderer
TriangleVectorRenderer  ← adding Triangle needs 2 more classes
TriangleRasterRenderer
```

Adding one new renderer requires adding N new classes (one per shape). Adding one new shape requires adding M new classes (one per renderer). Bridge eliminates this explosion.

## Structure

```csharp
// Implementation interface — the "implementation" side of the bridge
public interface IRenderer
{
    void RenderCircle(double radius);
    void RenderSquare(double side);
}

// Concrete implementations
public class VectorRenderer : IRenderer
{
    public void RenderCircle(double r)  => Console.WriteLine($"Drawing circle (r={r}) in vector");
    public void RenderSquare(double s)  => Console.WriteLine($"Drawing square (s={s}) in vector");
}

public class RasterRenderer : IRenderer
{
    public void RenderCircle(double r)  => Console.WriteLine($"Drawing circle (r={r}) as pixels");
    public void RenderSquare(double s)  => Console.WriteLine($"Drawing square (s={s}) as pixels");
}

// Abstraction — holds a reference to the implementation (the bridge)
public abstract class Shape
{
    protected readonly IRenderer _renderer; // THE BRIDGE

    protected Shape(IRenderer renderer) => _renderer = renderer;

    public abstract void Draw();
    public abstract void Resize(double factor);
}

// Refined abstractions — one class per shape type only
public class Circle : Shape
{
    private double _radius;

    public Circle(IRenderer renderer, double radius) : base(renderer)
        => _radius = radius;

    public override void Draw()          => _renderer.RenderCircle(_radius);
    public override void Resize(double f) => _radius *= f;
}

public class Square : Shape
{
    private double _side;

    public Square(IRenderer renderer, double side) : base(renderer)
        => _side = side;

    public override void Draw()          => _renderer.RenderSquare(_side);
    public override void Resize(double f) => _side *= f;
}

// Usage — vary shape and renderer independently
IRenderer vector = new VectorRenderer();
IRenderer raster = new RasterRenderer();

Shape c1 = new Circle(vector, 5.0);
Shape c2 = new Circle(raster, 5.0);
Shape s1 = new Square(vector, 10.0);

c1.Draw(); // "Drawing circle (r=5) in vector"
c2.Draw(); // "Drawing circle (r=5) as pixels"

// Adding a new renderer? Just implement IRenderer — zero changes to shapes.
// Adding a new shape? Just extend Shape — zero changes to renderers.
```

## Real .NET Example: `DbConnection` and ADO.NET

ADO.NET's design is Bridge-like:

```csharp
// Abstraction: DbConnection (abstract class)
//   ↓ bridge
// Implementation: provider-specific drivers

DbConnection connection = new SqlConnection(connectionString);  // SQL Server
DbConnection pgConn     = new NpgsqlConnection(pgConnStr);      // PostgreSQL
DbConnection sqLite     = new SqliteConnection("Data Source=:memory:"); // SQLite

// DbCommand is the abstraction; SqlCommand/NpgsqlCommand are implementations
DbCommand cmd = connection.CreateCommand();
cmd.CommandText = "SELECT * FROM Orders";

// The abstraction (DbCommand) works the same regardless of the concrete connection
await using DbDataReader reader = await cmd.ExecuteReaderAsync();
while (await reader.ReadAsync())
{
    Console.WriteLine(reader.GetString(0));
}
```

`DbConnection` is the abstraction; the provider-specific implementations (`SqlConnection`, `NpgsqlConnection`) are the implementation side. Your code works against `DbConnection` and can switch providers.

## Logging Bridge in .NET

`Microsoft.Extensions.Logging` also uses Bridge-like decomposition:

```csharp
// ILogger is the abstraction
// ILoggerProvider implementations are the "implementation" side

// Configure multiple implementations behind a single ILogger interface
builder.Logging.AddConsole();    // ConsoleLoggerProvider
builder.Logging.AddApplicationInsights(); // AppInsightsLoggerProvider

// In your class — only knows about ILogger<T> (the abstraction)
public class OrderService
{
    private readonly ILogger<OrderService> _logger;

    public OrderService(ILogger<OrderService> logger) => _logger = logger;

    public void PlaceOrder(Order order)
    {
        // The bridge dispatches to all registered providers
        _logger.LogInformation("Order {Id} placed", order.Id);
    }
}
```

<Callout type="senior">
**Bridge as a Design Decision — Prefer Composition**

The Bridge pattern formalizes a key object-oriented design principle: **favor composition over inheritance**. Whenever you find yourself creating class hierarchies with N×M combinations, ask: "Are there two independent dimensions of variation?" If yes, Bridge is likely the right tool.

In modern C# with DI:

```csharp
// Instead of NxM inheritance hierarchy, use composition via DI
public abstract class NotificationService
{
    private readonly IMessageChannel _channel; // implementation bridge

    protected NotificationService(IMessageChannel channel) => _channel = channel;

    protected abstract string FormatMessage(string content);

    public async Task SendAsync(string content)
    {
        string formatted = FormatMessage(content);
        await _channel.SendAsync(formatted);
    }
}

// Channels (implementation dimension)
public interface IMessageChannel { Task SendAsync(string message); }
public class EmailChannel   : IMessageChannel { /* ... */ }
public class SmsChannel     : IMessageChannel { /* ... */ }
public class PushChannel    : IMessageChannel { /* ... */ }

// Notification types (abstraction dimension)
public class AlertNotification  : NotificationService { /* format as ALERT: ... */ }
public class SummaryNotification: NotificationService { /* format as daily summary */ }

// Register with DI — wiring up bridges declaratively
builder.Services.AddKeyedSingleton<IMessageChannel, EmailChannel>("email");
builder.Services.AddKeyedSingleton<IMessageChannel, SmsChannel>("sms");
```

In practice, teams achieve Bridge-like decoupling through DI without necessarily calling it "Bridge." The pattern is the principle, not the label.
</Callout>
