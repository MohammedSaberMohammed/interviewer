## The Problem: Hard-Coded `new`

When you write `new SqlConnection()` inside a method, you've coupled your code to a concrete type. Switching databases means touching every call site. The Factory Method pattern moves that `new` into an overridable method, letting subclasses swap the concrete type.

```csharp
// Tightly coupled — changing the parser means changing this class
public class ReportGenerator
{
    public Report Generate(string data)
    {
        var parser = new CsvParser(); // hard-coded — no way to swap
        return parser.Parse(data);
    }
}
```

---

## Structure of Factory Method

```csharp
// Product interface
public interface IParser
{
    Report Parse(string data);
}

// Concrete products
public class CsvParser : IParser
{
    public Report Parse(string data) =>
        new Report($"{data.Split('\n').Length} rows parsed from CSV");
}

public class JsonParser : IParser
{
    public Report Parse(string data) =>
        new Report("JSON document parsed");
}

// Creator — declares the factory method as abstract
public abstract class ReportGenerator
{
    // THE factory method — subclasses override this
    protected abstract IParser CreateParser();

    // Template algorithm uses the factory method
    public Report Generate(string data)
    {
        IParser parser = CreateParser(); // polymorphic call
        Report report = parser.Parse(data);
        // shared post-processing here...
        return report;
    }
}

// Concrete creators — each overrides the factory method
public class CsvReportGenerator : ReportGenerator
{
    protected override IParser CreateParser() => new CsvParser();
}

public class JsonReportGenerator : ReportGenerator
{
    protected override IParser CreateParser() => new JsonParser();
}

// Usage — adding XML requires zero changes to existing code
ReportGenerator generator = new CsvReportGenerator();
Report report = generator.Generate("name,age\nAlice,30");
```

Adding XML support means adding `XmlParser` and `XmlReportGenerator` — zero changes to existing classes (Open/Closed Principle).

## Simplified: Factory Method as a Delegate

Modern C# often replaces the abstract class with a `Func<T>` delegate — lighter weight and more composable:

```csharp
public class ReportGenerator
{
    private readonly Func<IParser> _parserFactory;

    public ReportGenerator(Func<IParser> parserFactory)
    {
        _parserFactory = parserFactory;
    }

    public Report Generate(string data)
    {
        IParser parser = _parserFactory(); // factory method via delegate
        return parser.Parse(data);
    }
}

// Inject the factory — easy to mock in tests
var csvGenerator  = new ReportGenerator(() => new CsvParser());
var jsonGenerator = new ReportGenerator(() => new JsonParser());

// Or register in DI
builder.Services.AddTransient<ReportGenerator>(sp =>
    new ReportGenerator(() => new CsvParser()));
```

This trades inheritance for composition — often cleaner in modern C#.

## Real .NET BCL Example: `DbProviderFactory`

ADO.NET's `DbProviderFactory` is a textbook Factory Method — it abstracts over SQL Server, PostgreSQL, SQLite, and others:

```csharp
// Get a provider-specific factory
DbProviderFactory factory = DbProviderFactories.GetFactory("Microsoft.Data.SqlClient");

// Factory methods return provider-specific objects — without knowing the concrete type
DbConnection  connection = factory.CreateConnection()!;   // returns SqlConnection
DbCommand     command    = factory.CreateCommand()!;      // returns SqlCommand
DbDataAdapter adapter    = factory.CreateDataAdapter()!;  // returns SqlDataAdapter

connection.ConnectionString = "Server=.;Database=MyDb;Trusted_Connection=true;";
await connection.OpenAsync();

// Switch to PostgreSQL — only the factory changes, nothing else
DbProviderFactory pgFactory    = DbProviderFactories.GetFactory("Npgsql");
DbConnection      pgConnection = pgFactory.CreateConnection()!; // NpgsqlConnection
```

<Callout type="senior">
**Factory Method in ASP.NET Core DI — Keyed Services**

ASP.NET Core's DI container uses factory methods when you need per-resolution control. In .NET 8+, keyed services make named variants of an abstraction first-class:

```csharp
// Register named variants (.NET 8+)
builder.Services.AddKeyedTransient<IParser, CsvParser>("csv");
builder.Services.AddKeyedTransient<IParser, JsonParser>("json");
builder.Services.AddKeyedTransient<IParser, XmlParser>("xml");

// Resolve by key — the container acts as a factory
public class ParserConsumer
{
    public ParserConsumer([FromKeyedServices("csv")] IParser parser)
    {
        // Gets CsvParser
    }
}

// Or resolve dynamically via IKeyedServiceProvider
public class DynamicParserFactory
{
    private readonly IKeyedServiceProvider _sp;
    public DynamicParserFactory(IKeyedServiceProvider sp) => _sp = sp;

    public IParser GetParser(string format) =>
        _sp.GetRequiredKeyedService<IParser>(format);
}
```

**Rule of thumb**: If your factory just news up objects, use DI. If your factory has logic (caching, pooling, configuration-driven decisions), a dedicated factory class adds value.
</Callout>
