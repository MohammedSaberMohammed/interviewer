## The Problem: Cross-Cutting Concerns Without Subclassing

You have an `IOrderRepository` and you want to add logging and caching. Subclassing works once — but you can't subclass twice cleanly. And if you add logging to the class itself, it now does two things (SRP violation).

Decorator wraps an object with another object that implements the same interface — adding behavior before or after the inner call, without modifying the original.

---

## The Pattern

```csharp
// The interface
public interface IOrderRepository
{
    Task<Order?> GetByIdAsync(int id);
    Task SaveAsync(Order order);
}

// The real implementation
public class SqlOrderRepository : IOrderRepository
{
    private readonly AppDbContext _db;
    public SqlOrderRepository(AppDbContext db) => _db = db;

    public Task<Order?> GetByIdAsync(int id) => _db.Orders.FindAsync(id).AsTask();
    public Task SaveAsync(Order order)
    {
        _db.Orders.Update(order);
        return _db.SaveChangesAsync();
    }
}

// Decorator 1: Logging
public class LoggingOrderRepository : IOrderRepository
{
    private readonly IOrderRepository _inner; // wraps the real repo
    private readonly ILogger<LoggingOrderRepository> _logger;

    public LoggingOrderRepository(IOrderRepository inner, ILogger<LoggingOrderRepository> logger)
    {
        _inner  = inner;
        _logger = logger;
    }

    public async Task<Order?> GetByIdAsync(int id)
    {
        _logger.LogInformation("Getting order {Id}", id);
        var result = await _inner.GetByIdAsync(id); // delegate to inner
        _logger.LogInformation("Got order {Id}: {Found}", id, result != null);
        return result;
    }

    public async Task SaveAsync(Order order)
    {
        _logger.LogInformation("Saving order {Id}", order.Id);
        await _inner.SaveAsync(order);
        _logger.LogInformation("Saved order {Id}", order.Id);
    }
}

// Decorator 2: Caching
public class CachingOrderRepository : IOrderRepository
{
    private readonly IOrderRepository _inner;
    private readonly IMemoryCache     _cache;

    public CachingOrderRepository(IOrderRepository inner, IMemoryCache cache)
    {
        _inner = inner;
        _cache = cache;
    }

    public async Task<Order?> GetByIdAsync(int id)
    {
        if (_cache.TryGetValue($"order:{id}", out Order? cached))
            return cached;

        var order = await _inner.GetByIdAsync(id);
        if (order != null)
            _cache.Set($"order:{id}", order, TimeSpan.FromMinutes(5));
        return order;
    }

    public Task SaveAsync(Order order)
    {
        _cache.Remove($"order:{id}"); // invalidate on write
        return _inner.SaveAsync(order);
    }
}
```

---

## Composing Decorators

```csharp
// Stack decorators: CachingOrderRepository → LoggingOrderRepository → SqlOrderRepository
IOrderRepository repo = new CachingOrderRepository(
    inner: new LoggingOrderRepository(
        inner:  new SqlOrderRepository(dbContext),
        logger: logger),
    cache: memoryCache);

// Or via DI — Scrutor library makes this easy
builder.Services.AddScoped<SqlOrderRepository>();
builder.Services.Decorate<IOrderRepository, LoggingOrderRepository>();
builder.Services.Decorate<IOrderRepository, CachingOrderRepository>();
// Resulting chain: CachingOrderRepository → LoggingOrderRepository → SqlOrderRepository
```

## Real .NET Example: `DelegatingHandler` (HttpClient Pipeline)

`HttpClient`'s message handler pipeline is the Decorator pattern in the BCL:

```csharp
// DelegatingHandler is the abstract Decorator base for HttpClient
public class RetryHandler : DelegatingHandler
{
    private readonly int _maxRetries;

    public RetryHandler(int maxRetries = 3) => _maxRetries = maxRetries;

    protected override async Task<HttpResponseMessage> SendAsync(
        HttpRequestMessage request, CancellationToken ct)
    {
        for (int attempt = 0; attempt < _maxRetries; attempt++)
        {
            try
            {
                // Delegate to inner handler (the next in the chain)
                HttpResponseMessage response = await base.SendAsync(request, ct);
                if (response.IsSuccessStatusCode) return response;
            }
            catch (HttpRequestException) when (attempt < _maxRetries - 1)
            {
                await Task.Delay(TimeSpan.FromSeconds(Math.Pow(2, attempt)), ct);
            }
        }
        return await base.SendAsync(request, ct); // last attempt, let it throw
    }
}

public class LoggingHandler : DelegatingHandler
{
    private readonly ILogger _logger;
    public LoggingHandler(ILogger<LoggingHandler> logger) => _logger = logger;

    protected override async Task<HttpResponseMessage> SendAsync(
        HttpRequestMessage request, CancellationToken ct)
    {
        _logger.LogInformation("HTTP {Method} {Url}", request.Method, request.RequestUri);
        var response = await base.SendAsync(request, ct);
        _logger.LogInformation("HTTP {StatusCode}", response.StatusCode);
        return response;
    }
}

// Register handler chain
builder.Services.AddTransient<RetryHandler>();
builder.Services.AddTransient<LoggingHandler>();

builder.Services.AddHttpClient("api", c => c.BaseAddress = new Uri("https://api.example.com"))
    .AddHttpMessageHandler<LoggingHandler>()  // outer
    .AddHttpMessageHandler<RetryHandler>();   // inner
// Chain: LoggingHandler → RetryHandler → HttpClientHandler (actual HTTP)
```

  explanation="The outermost decorator (CachingRepo) executes first. It checks the cache. If there's a hit, it returns immediately — LoggingRepo and SqlRepo never run. If there's a cache miss, CachingRepo calls its inner (LoggingRepo), which logs and then calls SqlRepo. The call flows from outer to inner. Decorator order matters: put caching outside logging so cache hits don't produce log noise."

<Callout type="senior">
**Scrutor — Decorator Registration in .NET DI**

.NET's built-in DI container doesn't support decorators directly (you'd have to resolve the inner manually). The **Scrutor** NuGet package adds `Decorate<T>()`:

```csharp
// Without Scrutor — manual decorator wiring (fragile)
builder.Services.AddScoped<IOrderRepository>(sp =>
    new CachingOrderRepository(
        inner: new LoggingOrderRepository(
            inner:  sp.GetRequiredService<SqlOrderRepository>(),
            logger: sp.GetRequiredService<ILogger<LoggingOrderRepository>>()),
        cache: sp.GetRequiredService<IMemoryCache>()));

// With Scrutor — clean declarative decoration
builder.Services.AddScoped<IOrderRepository, SqlOrderRepository>();
builder.Services.Decorate<IOrderRepository, LoggingOrderRepository>();
builder.Services.Decorate<IOrderRepository, CachingOrderRepository>();
```

**Alternative: Source-generated AOP** — libraries like **Metalama** and **PostSharp** use code generation to weave decorator-like behavior via attributes, avoiding wrapper classes entirely:

```csharp
[Log, Cache] // aspects applied via source generation
public async Task<Order?> GetByIdAsync(int id)
    => await _db.Orders.FindAsync(id);
```

For interview questions: understand the manual Decorator pattern first, then mention Scrutor as the practical DI solution.
</Callout>
