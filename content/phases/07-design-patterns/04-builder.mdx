## The Problem: Constructor Overload Hell

When an object requires many optional parameters, constructors become unwieldy. Telescoping constructors (a chain of overloads adding one parameter each) are hard to read and error-prone — swapping two `string` arguments compiles silently but produces wrong behavior.

```csharp
// Telescoping constructor anti-pattern
public HttpRequest(string url) : this(url, "GET") { }
public HttpRequest(string url, string method) : this(url, method, null) { }
public HttpRequest(string url, string method, string? body) : this(url, method, body, 30) { }
public HttpRequest(string url, string method, string? body, int timeoutSeconds) { ... }

// Which argument is which? Easy to mix up
var req = new HttpRequest("https://api.example.com", "POST", null, 60);
```

## The Builder Pattern

```csharp
// The product — immutable once built
public sealed class HttpRequest
{
    public string  Url            { get; }
    public string  Method         { get; }
    public string? Body           { get; }
    public int     TimeoutSeconds { get; }
    public Dictionary<string, string> Headers { get; }

    // Private constructor — only the builder can create it
    private HttpRequest(Builder b)
    {
        Url            = b.Url;
        Method         = b.Method;
        Body           = b.Body;
        TimeoutSeconds = b.TimeoutSeconds;
        Headers        = new Dictionary<string, string>(b.Headers);
    }

    // Nested builder class
    public class Builder
    {
        internal string  Url            { get; private set; }
        internal string  Method         { get; private set; } = "GET";
        internal string? Body           { get; private set; }
        internal int     TimeoutSeconds { get; private set; } = 30;
        internal Dictionary<string, string> Headers { get; } = new();

        public Builder(string url) => Url = url;

        // Fluent methods — each returns 'this' for chaining
        public Builder WithMethod(string method)  { Method = method; return this; }
        public Builder WithBody(string body)      { Body   = body;   return this; }
        public Builder WithTimeout(int seconds)   { TimeoutSeconds = seconds; return this; }
        public Builder WithHeader(string key, string value)
        {
            Headers[key] = value;
            return this;
        }

        // Terminal method — validates and constructs the product
        public HttpRequest Build()
        {
            if (string.IsNullOrWhiteSpace(Url))
                throw new InvalidOperationException("URL is required.");
            return new HttpRequest(this);
        }
    }
}

// Usage — readable, self-documenting
HttpRequest request = new HttpRequest.Builder("https://api.example.com/orders")
    .WithMethod("POST")
    .WithBody(@"{""item"": ""book""}")
    .WithHeader("Authorization", "Bearer token123")
    .WithHeader("Content-Type", "application/json")
    .WithTimeout(60)
    .Build();
```

## Modern C# Alternative: `init` Properties + Object Initializer

For simpler cases, `init`-only properties with C# 9 object initializers can replace a full Builder:

```csharp
public sealed class HttpRequestOptions
{
    public required string Url            { get; init; }
    public string          Method         { get; init; } = "GET";
    public string?         Body           { get; init; }
    public int             TimeoutSeconds { get; init; } = 30;

    // Validate after construction
    public HttpRequestOptions Validate()
    {
        if (!Uri.TryCreate(Url, UriKind.Absolute, out _))
            throw new InvalidOperationException("Invalid URL");
        return this;
    }
}

// Clean initialization syntax
var options = new HttpRequestOptions
{
    Url    = "https://api.example.com",
    Method = "POST",
    Body   = "{}",
}.Validate();
```

Use a full Builder when: you need validation during construction, construction is multi-step, or you need to build multiple variations from the same builder state.

## Real .NET BCL Example: `StringBuilder`

`StringBuilder` is the canonical .NET Builder — it accumulates string fragments and produces a `string` at the end:

```csharp
var sb = new StringBuilder();

sb.Append("SELECT ")
  .Append("Id, Name, Email ")
  .Append("FROM Users ")
  .AppendLine("WHERE IsActive = 1")
  .Append("ORDER BY Name");

string sql = sb.ToString(); // build the final product

// More complex: build HTML
var html = new StringBuilder()
    .AppendLine("<ul>")
    .AppendJoin('\n', items.Select(i => $"  <li>{i}</li>"))
    .AppendLine()
    .AppendLine("</ul>")
    .ToString();
```

## Real ASP.NET Core Example: `WebApplicationBuilder`

```csharp
// WebApplicationBuilder is a Builder — configure services and middleware step by step
WebApplicationBuilder builder = WebApplication.CreateBuilder(args);

// Add services (configure the dependency injection container)
builder.Services.AddControllers();
builder.Services.AddDbContext<AppDbContext>(o =>
    o.UseSqlServer(builder.Configuration.GetConnectionString("Default")));
builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer();

// Build the application — terminal step produces WebApplication
WebApplication app = builder.Build();

// Configure middleware pipeline
app.UseHttpsRedirection();
app.UseAuthentication();
app.UseAuthorization();
app.MapControllers();

app.Run();
```

`WebApplicationBuilder` separates service registration (construction) from the running application (product). You can't use the app until `Build()` is called.

<Callout type="senior">
**Step Builder Pattern — Compile-Time Construction Order**

For objects where steps must happen in a specific order, the Step Builder pattern uses separate interfaces per step:

```csharp
public interface IFromStep    { IWhereStep From(string table); }
public interface IWhereStep   { IBuildStep Where(string condition); }
public interface IBuildStep   { string Build(); }

public class QueryBuilder : IFromStep, IWhereStep, IBuildStep
{
    private string _table = "";
    private string _where = "";

    private QueryBuilder() { }

    // Entry point — returns the first step interface only
    public static IFromStep Create() => new QueryBuilder();

    IWhereStep IFromStep.From(string table) { _table = table; return this; }
    IBuildStep IWhereStep.Where(string c)   { _where = c;     return this; }
    string     IBuildStep.Build()           => $"SELECT * FROM {_table} WHERE {_where}";
}

// Usage — only valid methods are available at each step
string sql = QueryBuilder.Create()
    .From("Users")       // returns IWhereStep — .Build() not available yet
    .Where("IsActive=1") // returns IBuildStep
    .Build();

// This does NOT compile — Build() isn't on IFromStep
// QueryBuilder.Create().Build();
```

This encodes the correct construction sequence in the type system — invalid usage is a compile error, not a runtime exception.
</Callout>
