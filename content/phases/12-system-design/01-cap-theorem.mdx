## What Is CAP Theorem?

CAP Theorem, proven by Eric Brewer in 2000, states that a distributed data store can provide **at most two of three guarantees** simultaneously:

- **C — Consistency**: Every read receives the most recent write or an error. All nodes see the same data at the same time.
- **A — Availability**: Every request receives a (non-error) response — but the response may not contain the most recent write.
- **P — Partition Tolerance**: The system continues operating even when network partitions cause message loss between nodes.

The critical insight: **in a real distributed system, network partitions are not optional**. They will happen. This means the real choice is always between **C and A** when a partition occurs.

```
Normal operation:          During a partition:
┌──────────┐               ┌──────────┐   ~~~~~   ┌──────────┐
│  Node A  │◄──────────────│  Node B  │           │  Node C  │
│  (write) │               │          │           │(isolated)│
└──────────┘               └──────────┘           └──────────┘

You must choose:
  CP → refuse reads on Node C (return error) — prioritize correctness
  AP → serve stale data on Node C — prioritize uptime
```

## The Three System Categories

### CP Systems — Consistency + Partition Tolerance

When a partition occurs, a CP system **refuses to serve potentially stale data**. Nodes that cannot confirm they have the latest state return an error or become unavailable.

**Examples**: HBase, Zookeeper, etcd, MongoDB (with majority write concern), SQL Server (synchronous replication)

**Use when**: Financial transactions, inventory counts, anything where showing stale data causes business harm.

### AP Systems — Availability + Partition Tolerance

When a partition occurs, an AP system **continues serving requests** even if some nodes may have stale data. Consistency is achieved eventually once the partition heals.

**Examples**: DynamoDB (by default), Cassandra, CouchDB, DNS, shopping cart systems

**Use when**: High read throughput, user-facing features that tolerate slight staleness (social media feeds, product recommendations).

### CA Systems — Consistency + Availability (No Partition Tolerance)

This category only exists in **single-node or LAN-connected systems** where you assume partitions never happen. Real distributed systems cannot choose this.

**Examples**: Single-node PostgreSQL, single-node MySQL (not distributed)

## CAP Is a Spectrum, Not a Binary Choice

Modern distributed systems do not sit squarely in one camp. **PACELC** (proposed by Daniel Abadi) is a more nuanced model:

```
PACELC:
  If Partition → choose between Availability and Consistency (like CAP)
  Else (no partition) → choose between Latency and Consistency
```

Even without partitions, replication lag means you trade **latency for consistency**. For example:
- Azure Cosmos DB exposes 5 explicit consistency levels (strong → eventual)
- DynamoDB allows per-request consistency choice
- PostgreSQL allows `SET SESSION CHARACTERISTICS AS TRANSACTION READ ONLY` for read replicas

```csharp
// Cosmos DB — choose consistency per operation
var options = new ItemRequestOptions
{
    ConsistencyLevel = ConsistencyLevel.Session // or Strong, BoundedStaleness, Eventual
};

var item = await container.ReadItemAsync<Order>(
    id: "order-123",
    partitionKey: new PartitionKey("customer-456"),
    requestOptions: options
);
```

## Interview War Story: The Shopping Cart Problem

Amazon famously chose AP for shopping carts. Here's the reasoning:

**CP choice**: If you can't confirm cart state across nodes, return an error. Customer sees "service unavailable" during checkout.

**AP choice**: Serve potentially stale cart. In the worst case, the customer sees an item that's no longer in their cart — then you reconcile. The business prefers a slightly wrong cart over no cart at all.

This is why Dynamo (the precursor to DynamoDB) was born — Amazon needed AP semantics with eventual consistency and application-level conflict resolution.

<Callout type="senior">
**Senior Insight: CAP Applies Per Operation, Not Per System**

You do not pick one CAP profile for your entire application. Sophisticated architectures apply different trade-offs to different data:

- **User profile reads**: AP — serve stale, it's fine if the name is 1 second old
- **Account balance reads**: CP — never serve stale, risk of overdraft
- **Order placement writes**: CP — require quorum before confirming
- **Analytics dashboards**: AP — eventual consistency is acceptable

In code, this means using **different data stores or consistency configurations** for different bounded contexts. A single microservice might use strong consistency for writes and read replicas for reads, accepting the replication lag. That's a deliberate CAP choice at the operation level.

Always ask in interviews: "What's the acceptable staleness window for this data?" That answer drives your CAP positioning.
</Callout>

## Applying CAP in a System Design Interview

When asked to design a system, address CAP explicitly with this framework:

1. **Identify consistency requirements**: Which operations absolutely cannot tolerate stale data?
2. **Identify availability requirements**: What is the acceptable downtime? SLA?
3. **Assume partitions will happen**: Never choose CA for distributed systems.
4. **State your choice and justify**: "I'm choosing AP for the feed service because a 5-second stale feed is acceptable. For payments, I'm choosing CP because showing an incorrect balance has direct financial consequences."

```csharp
// Example: Dual-write pattern with explicit consistency choice
public class OrderService
{
    private readonly IOrderRepository _orders;       // CP: SQL with strong consistency
    private readonly IFeedRepository _feed;          // AP: Cassandra with eventual consistency

    public async Task PlaceOrderAsync(PlaceOrderCommand cmd)
    {
        // CP: This must succeed with strong consistency — financial record
        var order = await _orders.SaveAsync(cmd); // throws on partition failure

        // AP: Best-effort — feed update can lag or fail without blocking order
        await _feed.PublishActivityAsync(order).ConfigureAwait(false);
        // If feed fails, we accept staleness — order is what matters
    }
}
```
