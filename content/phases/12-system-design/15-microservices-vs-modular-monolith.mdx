## The Spectrum of Modularity

There is not a binary choice between monolith and microservices. Architecture exists on a spectrum:

```
Tightly Coupled        Modular           Deployed          Fully
Monolith           ──► Monolith      ──► Separately    ──► Independent
(Big Ball of Mud)     (Clean DDD)       (2–5 services)    Microservices
                                                          (20+ services)
```

The goal at every level is the same: **high cohesion, low coupling**. The question is whether you achieve this within a single process or across network boundaries.

## What Is a Modular Monolith?

A modular monolith is a **single deployable unit** structured into highly cohesive, loosely coupled modules. Modules enforce hard boundaries — no circular dependencies, no direct database cross-references.

```
MyApp/
  ├── Orders/
  │     ├── Application/  (commands, queries, handlers)
  │     ├── Domain/       (Order, OrderItem, domain events)
  │     ├── Infrastructure/  (OrderRepository, SQL migrations)
  │     └── OrderModule.cs   (DI registration, exposes IOrderContract)
  ├── Inventory/
  │     ├── Application/
  │     ├── Domain/
  │     ├── Infrastructure/
  │     └── InventoryModule.cs
  └── Program.cs (wires modules together)
```

**Module contract**: Each module exposes an interface (`IOrderContract`) that other modules may call. Modules may NOT directly access each other's repositories or database tables.

```csharp
// Module boundary enforcement
// Orders module's public contract
public interface IOrderContract
{
    Task<OrderSummary?> GetOrderAsync(Guid orderId);
    Task<Guid> PlaceOrderAsync(PlaceOrderRequest request);
}

// Inventory module calls Orders via contract — never directly accesses Orders DB
public class InventoryService
{
    private readonly IOrderContract _orders; // injected, not directly coupled

    public async Task ProcessShipmentAsync(Guid orderId)
    {
        var order = await _orders.GetOrderAsync(orderId);
        // ... process based on order summary
    }
}

// Module registration
public static class InventoryModule
{
    public static IServiceCollection AddInventoryModule(this IServiceCollection services)
    {
        services.AddScoped<IInventoryService, InventoryService>();
        services.AddScoped<IInventoryContract, InventoryService>();
        // Inventory's own DbContext — separate schema, never shared
        services.AddDbContext<InventoryDbContext>(o =>
            o.UseSqlServer(conn, b => b.MigrationsHistoryTable("__EFMigrationsHistory", "inventory")));
        return services;
    }
}
```

## What Microservices Actually Require

Microservices are **independently deployable** services, each with its own process, database, and deployment pipeline. The key word is "independently deployable" — not just "small."

Prerequisites for microservices to work:
1. **Independent deployability**: Service A can be deployed without redeploying Service B
2. **Team autonomy**: Each service owned by a team that can make decisions independently
3. **Operational maturity**: Container orchestration (Kubernetes/ACA), distributed tracing, health checks, circuit breakers, API gateway
4. **Service contracts**: API versioning, backward compatibility, contract testing (Pact)
5. **Distributed data management**: Database per service, saga pattern, eventual consistency

```csharp
// Microservice communication patterns
// Synchronous: REST or gRPC
// Asynchronous: Message broker (preferred for loose coupling)

// In a microservice — you cannot call another service's repository
// You must go over the network
public class OrderService
{
    private readonly HttpClient _inventoryClient; // HTTP to InventoryService
    private readonly IPublishEndpoint _bus;       // Async to message broker

    public async Task PlaceOrderAsync(CreateOrderCommand cmd)
    {
        // Synchronous check (acceptable for critical path)
        var available = await _inventoryClient
            .GetFromJsonAsync<bool>(\$"/api/inventory/check?items={string.Join(",", cmd.ItemIds)}");

        if (!available) throw new InsufficientStockException();

        var order = new Order(cmd);
        await _orderRepo.SaveAsync(order);

        // Async event — decoupled, resilient
        await _bus.Publish(new OrderPlacedEvent(order.Id, cmd.Items));
    }
}
```

## Migrating: Monolith to Microservices (Strangler Fig)

When a modular monolith outgrows itself, migrate incrementally using the **Strangler Fig pattern** — extract one module at a time.

```
Before:                        After step 1:              After step 2:
┌──────────────────┐            ┌──────────────────┐       ┌──────┐  ┌──────┐
│ Modular Monolith │  ──────►   │ Monolith          │  ──►  │Order │  │Notif.│
│ Orders          │            │ Orders    (intact) │       │Svc   │  │Svc   │
│ Inventory       │            │ Inventory (intact) │       └──────┘  └──────┘
│ Notifications   │            └─────────────────────┘      ┌──────────────┐
└──────────────────┘            ┌──────────────────┐         │ Monolith     │
                                │ Notification Svc │         │ Inventory    │
                                │  (extracted)     │         └──────────────┘
                                └──────────────────┘
```

The API Gateway sits in front and routes traffic — old routes hit the monolith, new routes hit extracted services.

```csharp
// YARP reverse proxy as strangler fig router
// Routes new Notification paths to the extracted service
// Routes everything else to the legacy monolith

// appsettings.json (YARP config)
{
  "ReverseProxy": {
    "Routes": {
      "notifications": {
        "ClusterId": "notifications-cluster",
        "Match": { "Path": "/api/notifications/{**catch-all}" }
      },
      "monolith": {
        "ClusterId": "monolith-cluster",
        "Match": { "Path": "/{**catch-all}" }
      }
    },
    "Clusters": {
      "notifications-cluster": {
        "Destinations": { "primary": { "Address": "http://notification-svc" } }
      },
      "monolith-cluster": {
        "Destinations": { "primary": { "Address": "http://legacy-monolith" } }
      }
    }
  }
}
```

<Callout type="senior">
**Senior Insight: Most Teams Should Start with a Modular Monolith**

The industry is slowly course-correcting after a decade of microservices hype. Shopify, Stack Overflow, Basecamp, and many high-traffic systems run well-structured monoliths at enormous scale.

The argument for starting modular monolith:
1. **Module boundaries are hard to get right** — getting them wrong in a monolith means a refactor. Getting them wrong in microservices means a distributed systems nightmare.
2. **Operational overhead is real** — each microservice needs its own CI/CD pipeline, health checks, deployment config, logging pipeline, and on-call runbooks. With 20 services and 3 engineers, this dominates engineering time.
3. **Network is not free** — a method call in a modular monolith is nanoseconds. An HTTP call is milliseconds with retry/timeout complexity.
4. **Extracting modules is possible later** — a well-factored module with a clean interface can be extracted to a microservice when there's genuine need (independent scaling, independent deployment cadence, team growth).

The right answer in a system design interview: "I'd start with a modular monolith, extract services when specific scaling or team autonomy needs justify the operational cost, and use the Strangler Fig pattern to migrate incrementally."
</Callout>

<MythBuster
  myth="Microservices are inherently more scalable than monoliths."
  reality="Scalability comes from stateless services, horizontal scaling, and database design — not from splitting into multiple processes. A well-designed monolith behind a load balancer scales horizontally just fine. Microservices enable DIFFERENT scaling rates for different components (scale the checkout service more than the admin panel), which is valuable only when different components truly have different load profiles. Running 20 microservices on Kubernetes when a single monolith on 3 VMs would serve the load is paying a high operational tax for no benefit."
