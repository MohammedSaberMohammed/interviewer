## Resilience Is Not Optional

In a distributed system, partial failures are the norm, not the exception. Services go down for deployments, networks have transient errors, databases have momentary spikes. Without resilience patterns:

- A transient error causes a 500 instead of a retry
- A slow downstream causes unbounded thread pool exhaustion
- A failing service pulls down all services that depend on it (cascading failure)

Polly (now deeply integrated into .NET 8 via `Microsoft.Extensions.Resilience`) provides composable policies to handle these scenarios.

## Polly v8 Architecture: ResiliencePipeline

Polly v8 (shipped with .NET 8) moved from the old `Policy` API to `ResiliencePipeline`. Strategies are composed in order — each strategy wraps the one inside it.

```csharp
// Install: Install-Package Microsoft.Extensions.Http.Resilience

// The pipeline executes strategies from OUTSIDE IN:
// Timeout → CircuitBreaker → Retry → actual call
var pipeline = new ResiliencePipelineBuilder()
    .AddTimeout(TimeSpan.FromSeconds(10))           // outer: overall timeout
    .AddCircuitBreaker(new CircuitBreakerStrategyOptions
    {
        FailureRatio = 0.5,
        MinimumThroughput = 10,
        SamplingDuration = TimeSpan.FromSeconds(30),
        BreakDuration = TimeSpan.FromSeconds(15)
    })
    .AddRetry(new RetryStrategyOptions
    {
        MaxRetryAttempts = 3,
        Delay = TimeSpan.FromMilliseconds(200),
        BackoffType = DelayBackoffType.Exponential,
        UseJitter = true
    })
    .Build();

var result = await pipeline.ExecuteAsync(async ct =>
    await _httpClient.GetFromJsonAsync<Product>("/api/products/123", ct)
);
```

## Strategy 1: Retry

Retry handles **transient failures** — errors that are expected to resolve themselves quickly.

**Key decisions**:
- **What to retry**: Only transient errors (5xx, network timeout, `HttpRequestException`). Never retry 400 Bad Request or 401 Unauthorized.
- **How many times**: Usually 3–5. More retries increase latency and can amplify load.
- **Backoff strategy**: Exponential with jitter to avoid thundering herd.
- **Jitter**: Randomizes retry delays so all clients do not retry at the same moment.

```csharp
// HttpClient with retry via Microsoft.Extensions.Http.Resilience
builder.Services.AddHttpClient<IProductClient, ProductClient>()
    .AddStandardResilienceHandler(options =>
    {
        // Retry: 3 attempts, exponential backoff, jitter, on 5xx and network errors
        options.Retry.MaxRetryAttempts = 3;
        options.Retry.Delay = TimeSpan.FromMilliseconds(300);
        options.Retry.BackoffType = DelayBackoffType.Exponential;
        options.Retry.UseJitter = true;

        // Only retry on transient HTTP errors
        options.Retry.ShouldHandle = args => args.Outcome switch
        {
            { Exception: HttpRequestException } => PredicateResult.True(),
            { Result.StatusCode: HttpStatusCode.ServiceUnavailable } => PredicateResult.True(),
            { Result.StatusCode: HttpStatusCode.TooManyRequests } => PredicateResult.True(),
            _ => PredicateResult.False()
        };

        // Respect Retry-After header (from 429 responses)
        options.Retry.OnRetry = args =>
        {
            _logger.LogWarning("Retry attempt {Attempt} after {Delay}ms",
                args.AttemptNumber, args.RetryDelay.TotalMilliseconds);
            return default;
        };
    });
```

## Strategy 2: Circuit Breaker

The circuit breaker prevents cascading failures. When a downstream service is consistently failing, it is worse to keep retrying — you exhaust threads, accumulate latency, and amplify load on the already-struggling service.

**Three states**:
- **Closed** (normal): Requests flow through, failures are counted
- **Open** (broken): Requests fail immediately without hitting downstream — "fast fail"
- **Half-Open** (probe): One request is allowed through to test if recovery occurred

```
Normal:   ──requests──►[closed]──►  downstream
Failures exceed threshold:
Broken:   ──requests──►[open]──► immediate exception (no downstream call)
After break duration:
Probe:    ──1 request──►[half-open]──► downstream
Success → closed again
Failure → open again
```

```csharp
// Manual circuit breaker configuration
builder.Services.AddHttpClient<IInventoryClient, InventoryClient>()
    .AddResilienceHandler("inventory-pipeline", builder =>
    {
        builder.AddCircuitBreaker(new HttpCircuitBreakerStrategyOptions
        {
            // Open circuit when 50% of requests in 30-second window fail (min 10 requests)
            FailureRatio = 0.5,
            MinimumThroughput = 10,
            SamplingDuration = TimeSpan.FromSeconds(30),
            // Keep circuit open for 15 seconds before probing
            BreakDuration = TimeSpan.FromSeconds(15),
            OnOpened = args =>
            {
                _logger.LogError("Circuit OPENED for InventoryService. Break for {Duration}s",
                    args.BreakDuration.TotalSeconds);
                return default;
            },
            OnClosed = args =>
            {
                _logger.LogInformation("Circuit CLOSED — InventoryService recovered");
                return default;
            }
        });

        builder.AddRetry(new HttpRetryStrategyOptions
        {
            MaxRetryAttempts = 2  // retry before circuit counts failures
        });
    });
```

## Strategy 3: Timeout

Timeouts prevent slow downstream services from consuming all threads indefinitely.

**Two types**:
- **Per-attempt timeout**: Each individual attempt (including retries) has a max duration
- **Overall timeout**: The entire operation including all retries has a max duration

```csharp
builder.Services.AddHttpClient<IPaymentClient, PaymentClient>()
    .AddResilienceHandler("payment-pipeline", builder =>
    {
        // Overall timeout: abort the entire request chain after 10 seconds
        builder.AddTimeout(TimeSpan.FromSeconds(10));

        builder.AddRetry(new HttpRetryStrategyOptions { MaxRetryAttempts = 3 });

        // Per-attempt timeout: each individual attempt times out in 2 seconds
        builder.AddTimeout(new HttpTimeoutStrategyOptions
        {
            Timeout = TimeSpan.FromSeconds(2)
        });
    });
```

<Callout type="trap">
**Trap: HttpClient.Timeout vs Polly Timeout**

`HttpClient.Timeout` throws `TaskCanceledException` with a confusing message. Polly's timeout throws `TimeoutRejectedException` and is composable with retry and circuit breaker. Always use Polly's timeout inside a pipeline rather than relying solely on `HttpClient.Timeout`.
</Callout>

## Strategy 4: Bulkhead (Concurrency Limiter)

A bulkhead isolates resources so that a surge in one operation does not starve others. Named after ship compartments — a leak in one compartment does not flood the whole ship.

```csharp
// Limit concurrent calls to the PaymentService to 10
// Additional requests queue (up to 5) or are rejected
var bulkheadPipeline = new ResiliencePipelineBuilder()
    .AddConcurrencyLimiter(new ConcurrencyLimiterStrategyOptions
    {
        MaxConcurrentExecutions = 10,
        QueuedTasksLimit = 5
    })
    .Build();
```

**Use case**: You have 3 downstream dependencies. Without bulkheads, if Dependency C is slow, it fills all threads and Dependency A and B become unreachable too. With bulkheads, each dependency gets its own thread budget.

<Callout type="senior">
**Senior Insight: Pipeline Order Is Critical**

The order you add strategies to a `ResiliencePipelineBuilder` matters — strategies are composed as nested wrappers:

```
outer ──► [Timeout] ──► [CircuitBreaker] ──► [Retry] ──► [actual call]
```

**Correct order** (outer to inner):
1. **Timeout** (outermost): Enforces an absolute deadline on the entire operation
2. **Circuit Breaker**: Cuts off call before retry attempts if circuit is open
3. **Retry** (innermost): Retries the actual call

Wrong order — Retry outside CircuitBreaker: The circuit breaker would count one failure and the retry would try again, potentially reopening the circuit before it gets a chance to break properly.

Also: **never retry non-idempotent operations** (POST, PATCH that creates data). Only retry reads and idempotent writes. `AddStandardResilienceHandler` does this correctly by default — it only retries on `GET` and `HEAD` requests. Override `ShouldHandle` for custom idempotency.
</Callout>

<MythBuster
  myth="Adding retry to all HTTP calls makes your service more resilient."
  reality="Retry without a circuit breaker can cause cascading failure. If a downstream service is overwhelmed, retrying every failed request 3 times triples the load on the already-struggling service. A circuit breaker must short-circuit retries when the failure rate exceeds a threshold — failing fast allows the downstream service to recover without being drowned in retry traffic."
/>
