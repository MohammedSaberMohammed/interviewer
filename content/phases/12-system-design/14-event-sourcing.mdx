## What Is Event Sourcing?

Traditional applications store **current state**: an Order row has `Status = "Shipped"`. To understand what happened, you need audit logs bolted on afterward.

Event sourcing flips this: **state changes are the primary data**. Every mutation is recorded as an immutable event. Current state is derived by replaying events.

## Core Concepts

### Event Stream

All events for one aggregate (e.g., one Order) form an **event stream** identified by the aggregate ID. Events within a stream are strictly ordered by sequence number.

### Append-Only

Events are **never updated or deleted**. An "undo" is another event (`OrderCancelled`). This immutability is what gives event sourcing its audit trail guarantee.

### Aggregate Reconstitution

To load an aggregate's current state, replay all its events in order.

```csharp
// Domain event types
public abstract record DomainEvent(Guid AggregateId, DateTimeOffset OccurredAt, int Version);

public record OrderCreated(Guid AggregateId, DateTimeOffset OccurredAt, int Version,
    Guid CustomerId, List<OrderItem> Items) : DomainEvent(AggregateId, OccurredAt, Version);

public record OrderItemAdded(Guid AggregateId, DateTimeOffset OccurredAt, int Version,
    OrderItem Item) : DomainEvent(AggregateId, OccurredAt, Version);

public record OrderPaid(Guid AggregateId, DateTimeOffset OccurredAt, int Version,
    decimal Amount, string PaymentRef) : DomainEvent(AggregateId, OccurredAt, Version);

public record OrderShipped(Guid AggregateId, DateTimeOffset OccurredAt, int Version,
    string TrackingNumber) : DomainEvent(AggregateId, OccurredAt, Version);

// Aggregate — applies events to build current state
public class Order
{
    public Guid Id { get; private set; }
    public Guid CustomerId { get; private set; }
    public OrderStatus Status { get; private set; }
    public List<OrderItem> Items { get; private set; } = new();
    public decimal TotalPaid { get; private set; }
    public int Version { get; private set; }

    // Reconstitute from event stream
    public static Order Reconstitute(IEnumerable<DomainEvent> events)
    {
        var order = new Order();
        foreach (var evt in events)
        {
            order.Apply(evt);
        }
        return order;
    }

    private void Apply(DomainEvent evt)
    {
        Version = evt.Version;
        switch (evt)
        {
            case OrderCreated e:
                Id = e.AggregateId;
                CustomerId = e.CustomerId;
                Items = e.Items.ToList();
                Status = OrderStatus.Pending;
                break;

            case OrderItemAdded e:
                Items.Add(e.Item);
                break;

            case OrderPaid e:
                TotalPaid = e.Amount;
                Status = OrderStatus.Paid;
                break;

            case OrderShipped e:
                Status = OrderStatus.Shipped;
                break;
        }
    }

    // Command handlers — validate business rules, emit events
    public IReadOnlyList<DomainEvent> Ship(string trackingNumber)
    {
        if (Status != OrderStatus.Paid)
            throw new InvalidOperationException("Cannot ship unpaid order");

        return new[] { new OrderShipped(Id, DateTimeOffset.UtcNow, Version + 1, trackingNumber) };
    }
}
```

## The Event Store

An event store is a database optimized for append-only, stream-based access. You can implement one using a standard relational database.

```csharp
// Simple event store using SQL Server
public class SqlEventStore
{
    private readonly string _connectionString;

    public async Task AppendAsync(Guid streamId, IEnumerable<DomainEvent> events, int expectedVersion)
    {
        await using var conn = new SqlConnection(_connectionString);
        await conn.OpenAsync();

        // Optimistic concurrency check — prevent lost updates
        var currentVersion = await conn.QuerySingleOrDefaultAsync<int?>(
            "SELECT MAX(Version) FROM EventStore WHERE StreamId = @StreamId",
            new { StreamId = streamId }
        ) ?? 0;

        if (currentVersion != expectedVersion)
            throw new OptimisticConcurrencyException(
                \$"Expected version {expectedVersion}, but stream is at version {currentVersion}");

        foreach (var evt in events)
        {
            await conn.ExecuteAsync(@"
                INSERT INTO EventStore (StreamId, EventType, Payload, Version, OccurredAt)
                VALUES (@StreamId, @EventType, @Payload, @Version, @OccurredAt)",
                new
                {
                    StreamId = streamId,
                    EventType = evt.GetType().Name,
                    Payload = JsonSerializer.Serialize(evt, evt.GetType()),
                    evt.Version,
                    evt.OccurredAt
                });
        }
    }

    public async Task<IReadOnlyList<DomainEvent>> LoadStreamAsync(Guid streamId)
    {
        await using var conn = new SqlConnection(_connectionString);
        var rows = await conn.QueryAsync(
            "SELECT EventType, Payload FROM EventStore WHERE StreamId = @StreamId ORDER BY Version",
            new { StreamId = streamId }
        );

        return rows.Select(row => DeserializeEvent(row.EventType, row.Payload)).ToList();
    }

    private static DomainEvent DeserializeEvent(string typeName, string payload)
    {
        var type = Type.GetType(\$"MyApp.Domain.Events.{typeName}")
            ?? throw new InvalidOperationException(\$"Unknown event type: {typeName}");
        return (DomainEvent)JsonSerializer.Deserialize(payload, type)!;
    }
}
```

## Projections: Building Read Models from Events

Events are raw facts. **Projections** transform the event stream into queryable read models.

```csharp
// Projection handler — builds an OrderSummary read model
public class OrderSummaryProjection :
    IProjection<OrderCreated>,
    IProjection<OrderPaid>,
    IProjection<OrderShipped>
{
    private readonly ReadDbContext _db;

    public async Task ProjectAsync(OrderCreated evt)
    {
        _db.OrderSummaries.Add(new OrderSummary
        {
            Id = evt.AggregateId,
            CustomerId = evt.CustomerId,
            Status = "Pending",
            TotalAmount = evt.Items.Sum(i => i.Price * i.Quantity),
            CreatedAt = evt.OccurredAt
        });
        await _db.SaveChangesAsync();
    }

    public async Task ProjectAsync(OrderPaid evt)
    {
        var summary = await _db.OrderSummaries.FindAsync(evt.AggregateId);
        if (summary is null) return;
        summary.Status = "Paid";
        summary.PaidAt = evt.OccurredAt;
        await _db.SaveChangesAsync();
    }

    public async Task ProjectAsync(OrderShipped evt)
    {
        var summary = await _db.OrderSummaries.FindAsync(evt.AggregateId);
        if (summary is null) return;
        summary.Status = "Shipped";
        summary.ShippedAt = evt.OccurredAt;
        await _db.SaveChangesAsync();
    }
}
```

**Key benefit — projection rebuild**: If you add a new field or fix a bug in a projection, you can **replay the entire event history** to rebuild the projection from scratch. This is impossible with traditional state stores.

## Snapshots: Handling Long Event Streams

Reconstituting an aggregate with thousands of events is slow. **Snapshots** periodically capture the current state so you only replay events after the snapshot.

```csharp
public async Task<Order> LoadOrderAsync(Guid orderId)
{
    // Try to load snapshot first
    var snapshot = await _snapshotStore.GetLatestAsync(orderId);

    IEnumerable<DomainEvent> events;
    if (snapshot is not null)
    {
        // Only replay events AFTER the snapshot version
        events = await _eventStore.LoadStreamFromVersionAsync(orderId, snapshot.Version + 1);
        var order = (Order)snapshot.State;
        foreach (var evt in events) order.Apply(evt);
        return order;
    }

    // Full replay — no snapshot
    events = await _eventStore.LoadStreamAsync(orderId);
    return Order.Reconstitute(events);
}
```

<Callout type="senior">
**Senior Insight: Event Sourcing Is Overkill for Most Applications**

Event sourcing genuinely shines for:
- **Financial ledgers**: Every transaction is an event; balance = sum of debits/credits
- **Audit-required domains**: Healthcare, compliance, supply chain
- **Collaborative editing**: Operational Transforms use event sourcing internally
- **Complex DDD aggregates**: Where history drives business logic

It is overkill for:
- CRUD applications with no historical query needs
- Simple microservices with no temporal requirements
- Systems where the team is small and operational complexity matters

The interview trap: candidates hear "event sourcing" and treat it as a silver bullet. The correct answer is: "I'd use event sourcing in bounded contexts where audit history or temporal queries are a hard requirement — typically the payment and inventory services. Other bounded contexts like user profiles would use traditional state storage."

Event sourcing also pairs naturally with CQRS — the event stream is the write model, projections are the read models. But CQRS does not require event sourcing. They are independent patterns.
</Callout>

<MythBuster
  myth="Event sourcing gives you a built-in audit log for free."
  reality="Event sourcing stores domain events — business facts. An audit log captures who did what, when, from which IP, with what context. These overlap but are not the same. Domain events may omit audit metadata (user ID, request ID, IP address). You need to enrich events with correlation metadata at the command handling layer for them to serve as proper audit records. Also, events are technical data structures — compliance audits often require human-readable records, which still need a projection or formatting layer."
