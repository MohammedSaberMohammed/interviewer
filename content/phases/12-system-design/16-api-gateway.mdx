## What Does an API Gateway Do?

An API gateway is a reverse proxy that sits between clients and your backend services. It centralizes cross-cutting concerns that would otherwise be duplicated in every service:

```
Clients:               Gateway:                     Services:
┌──────────┐           ┌─────────────────────┐      ┌─────────────┐
│ Web App  │──HTTPS──► │ • Authentication    │ ──►  │ OrderService│
│ Mobile   │           │ • Rate limiting      │ ──►  │ UserService │
│ Partners │           │ • Request routing   │ ──►  │ PaymentSvc  │
└──────────┘           │ • TLS termination   │      └─────────────┘
                       │ • Logging & tracing │
                       │ • Response caching  │
                       └─────────────────────┘
```

**Core functions**:
- **Routing**: Map `/api/orders/**` to OrderService, `/api/users/**` to UserService
- **Authentication**: Validate JWTs, pass claims to downstream services
- **Rate limiting**: Throttle at the edge, protecting all services at once
- **TLS termination**: Handle HTTPS; internal services can use HTTP
- **Aggregation**: Fan-out multiple service calls and merge responses

## Option 1: YARP (Yet Another Reverse Proxy)

YARP is Microsoft's open-source, .NET-native reverse proxy. It runs as a standard ASP.NET Core app, giving you full access to the middleware pipeline.

```csharp
// Install: Install-Package Yarp.ReverseProxy

// Program.cs
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddReverseProxy()
    .LoadFromConfig(builder.Configuration.GetSection("ReverseProxy"));

// Add cross-cutting middleware
builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer();
builder.Services.AddRateLimiter(options =>
    options.AddFixedWindowLimiter("global", o =>
    {
        o.Window = TimeSpan.FromMinutes(1);
        o.PermitLimit = 1000;
    }));

var app = builder.Build();

app.UseAuthentication();
app.UseAuthorization();
app.UseRateLimiter();

// Custom middleware: inject correlation ID before proxying
app.Use(async (ctx, next) =>
{
    var correlationId = ctx.Request.Headers["X-Correlation-Id"].FirstOrDefault()
        ?? Guid.NewGuid().ToString();
    ctx.Request.Headers["X-Correlation-Id"] = correlationId;
    ctx.Response.Headers["X-Correlation-Id"] = correlationId;
    await next();
});

app.MapReverseProxy(proxyPipeline =>
{
    // Transform requests — strip auth header before forwarding
    proxyPipeline.UseSessionAffinity();
    proxyPipeline.UseLoadBalancing();
});

app.Run();
```

```json

}
```

**YARP strengths**:
- Native .NET — full access to ASP.NET Core middleware
- Highly configurable — dynamic config reload without restart
- Integrates with existing .NET auth, rate limiting, logging
- Great for adding custom logic (JWT forwarding, tenant routing, feature flags)

## Option 2: Ocelot

Ocelot is an older .NET API gateway with a JSON-first configuration approach. Simpler for basic routing, but less flexible than YARP for custom middleware.

```csharp
// Program.cs
builder.Configuration.AddJsonFile("ocelot.json");
builder.Services.AddOcelot();

var app = builder.Build();
await app.UseOcelot();
```

```json
// ocelot.json
{
  "Routes": [
    {
      "UpstreamPathTemplate": "/api/orders/{everything}",
      "UpstreamHttpMethod": ["Get", "Post", "Put"],
      "DownstreamPathTemplate": "/orders/{everything}",
      "DownstreamScheme": "http",
      "DownstreamHostAndPorts": [
        { "Host": "order-service", "Port": 8080 }
      ],
      "AuthenticationOptions": {
        "AuthenticationProviderKey": "Bearer"
      },
      "RateLimitOptions": {
        "EnableRateLimiting": true,
        "Period": "1m",
        "Limit": 100
      }
    }
  ]
}
```

**Ocelot vs YARP**: YARP is now Microsoft's preferred approach. Ocelot has not seen major updates recently. New projects should default to YARP.

## Option 3: Azure API Management (APIM)

Azure APIM is a fully managed gateway with a rich policy engine, developer portal, and enterprise features.

```xml
<!-- APIM policy — rate limit + JWT validation + response caching -->
<policies>
  <inbound>
    <rate-limit calls="100" renewal-period="60" />
    <validate-jwt header-name="Authorization" require-expiration-time="true">
      <openid-config url="https://login.microsoftonline.com/{tenant}/.well-known/openid-configuration" />
      <required-claims>
        <claim name="aud"><value>api://my-app</value></claim>
      </required-claims>
    </validate-jwt>
    <cache-lookup vary-by-developer="false" vary-by-developer-groups="false" />
    <set-header name="X-Forwarded-For" exists-action="override">
      <value>@(context.Request.IpAddress)</value>
    </set-header>
  </inbound>
  <outbound>
    <cache-store duration="300" />
  </outbound>
</policies>
```

**APIM strengths**: Developer portal, subscription management, API versioning, analytics, mock responses, managed identity integration.

## The Backends for Frontends (BFF) Pattern

A single generic API gateway forces all clients (web, mobile, IoT) to work with the same API shape. The **BFF pattern** creates a dedicated gateway per client type.

```
Web App  ──► Web BFF       ──►  ┌──────────────────┐
Mobile   ──► Mobile BFF   ──►  │  Microservices   │
Partners ──► Partner BFF  ──►  │  (upstream)      │
                                └──────────────────┘
```

Each BFF:
- Aggregates multiple service calls into one optimized response for its client
- Shapes the API to exactly what the client needs (no over-fetching or under-fetching)
- Is owned by the team responsible for that client

```csharp
// Mobile BFF — aggregate multiple service calls for the dashboard endpoint
[ApiController, Route("mobile/api")]
public class MobileDashboardController : ControllerBase
{
    private readonly IOrderClient _orders;
    private readonly IUserClient _users;
    private readonly INotificationClient _notifications;

    [HttpGet("dashboard")]
    public async Task<IActionResult> GetDashboard()
    {
        var userId = User.FindFirst("sub")!.Value;

        // Fan out — parallel calls to three services
        var (orders, user, notifications) = await (
            _orders.GetRecentOrdersAsync(userId, limit: 5),
            _users.GetProfileAsync(userId),
            _notifications.GetUnreadCountAsync(userId)
        ).WhenAll();

        // Return exactly what mobile needs — no extra fields
        return Ok(new MobileDashboard(
            UserName: user.FirstName,
            RecentOrders: orders.Select(o => new OrderSummary(o.Id, o.Status, o.Total)),
            UnreadNotifications: notifications.Count
        ));
    }
}

// Extension for tuple Task.WhenAll
private static async Task<(T1, T2, T3)> WhenAll<T1, T2, T3>(
    this (Task<T1> t1, Task<T2> t2, Task<T3> t3) tasks)
{
    await Task.WhenAll(tasks.t1, tasks.t2, tasks.t3);
    return (tasks.t1.Result, tasks.t2.Result, tasks.t3.Result);
}
```

<Callout type="senior">
**Senior Insight: Don't Put Business Logic in the Gateway**

API gateways handle cross-cutting infrastructure concerns — routing, auth, rate limiting, TLS, logging. A common mistake is putting business logic in the gateway: conditional routing based on user role, data transformation, validation.

When business logic leaks into the gateway:
- The gateway becomes a bottleneck — every change requires a gateway deployment
- Testing becomes harder — the gateway isn't a standard service with unit tests
- Multiple teams need to coordinate on the same component (breaks Conway's Law)

The rule: **if it's about the network or security, it belongs in the gateway. If it's about the business, it belongs in a service.** JWT validation in the gateway is correct. "Route gold-tier users to the premium inventory service" is business logic — move it to the service or a separate routing microservice.

One nuance: BFF controllers that aggregate calls are legitimate business composition — they live at the edge but are owned by a team and are testable like any service.
</Callout>
