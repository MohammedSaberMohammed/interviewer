## L4 vs L7 Load Balancing

**Layer 4 (Transport)**: Routes based on IP address and port only. Does not inspect HTTP headers or content. Fast and low-latency. Cannot route based on URL path, cookies, or request body.

**Layer 7 (Application)**: Routes based on HTTP headers, URL path, cookies, hostname. Can make intelligent routing decisions (route `/api/admin` to admin servers, route based on `Cookie: region=eu`). Slightly higher latency due to request parsing.

```
L4 Load Balancer:              L7 Load Balancer:
Client ──TCP──► LB ──TCP──►    Client ──HTTP──► LB ──HTTP──►
  knows: src IP, dst port        knows: URL, headers, cookies, body
  cannot: inspect content        can: route by path, hostname, cookie
  example: Azure Load Balancer   example: Azure App Gateway, NGINX, YARP
```

## Algorithm 1: Round-Robin

Distribute requests evenly across all healthy backends in rotation. Simple and fair when all requests have similar processing cost.

```
Request 1 → Server A
Request 2 → Server B
Request 3 → Server C
Request 4 → Server A  (cycles)
```

**Problem**: Does not account for server capacity or current load. A slow request on Server A still receives the next request in rotation, even if Server A is backlogged.

**Weighted Round-Robin**: Assign weights based on server capacity. A server with 4 CPUs gets 2× the traffic of a 2-CPU server.

```csharp
// YARP — round-robin load balancing
{
  "ReverseProxy": {
    "Clusters": {
      "order-cluster": {
        "LoadBalancingPolicy": "RoundRobin",
        "Destinations": {
          "server-a": { "Address": "http://order-svc-a:8080" },
          "server-b": { "Address": "http://order-svc-b:8080" },
          "server-c": { "Address": "http://order-svc-c:8080" }
        }
      }
    }
  }
}
```

## Algorithm 2: Least Connections

Route each request to the backend with the fewest active connections. Better than round-robin when request processing times vary.

```
Active connections: A=10, B=2, C=7
New request → Server B (fewest active)
```

**Weighted Least Connections**: Normalize by server capacity. A server with 8 CPUs and 10 connections is less loaded than one with 2 CPUs and 10 connections.

```csharp
// YARP supports least requests (similar to least connections)
{
  "ReverseProxy": {
    "Clusters": {
      "order-cluster": {
        "LoadBalancingPolicy": "LeastRequests",
        "Destinations": {
          "server-a": { "Address": "http://order-svc-a:8080" },
          "server-b": { "Address": "http://order-svc-b:8080" }
        }
      }
    }
  }
}
```

## Algorithm 3: Sticky Sessions (Session Affinity)

Route a client's requests to the **same backend** for the duration of a session. Typically implemented using a cookie containing the backend identifier.

```
Client X → (first request) → Server B
LB sets cookie: X-Backend=server-b

Client X → (subsequent requests) → Server B (sticky via cookie)
Client Y → Server A (different backend)
```

**Why it exists**: Stateful applications that store session data in memory require the same server to handle all requests from a user. Shopping cart data, wizard state, WebSocket connections.

```csharp
// YARP — session affinity (sticky sessions)
{
  "ReverseProxy": {
    "Routes": {
      "webapp-route": {
        "ClusterId": "webapp-cluster",
        "Match": { "Path": "/{**catch-all}" },
        "SessionAffinity": {
          "Enabled": true,
          "Policy": "Cookie",        // or "CustomHeader"
          "AffinityKeyName": ".X-Backend-Affinity"
        }
      }
    },
    "Clusters": {
      "webapp-cluster": {
        "LoadBalancingPolicy": "RoundRobin",
        "Destinations": {
          "server-a": { "Address": "http://webapp-a:8080" },
          "server-b": { "Address": "http://webapp-b:8080" }
        }
      }
    }
  }
}
```

## Algorithm 4: IP Hash

The client's IP address is hashed and the result determines which backend handles the request. Provides session affinity without cookies.

```
hash("192.168.1.1") % 3 = 1 → Server B (always)
hash("192.168.1.2") % 3 = 0 → Server A (always)
```

**Problem**: Clients behind NAT (e.g., corporate proxies) all appear as the same IP — all their traffic goes to one server.

---

## Algorithm 5: Random with Two Choices (Power of Two)

Pick two backends at random and choose the one with fewer active connections. Surprisingly effective — produces near-optimal load distribution with O(1) overhead.

```
Randomly selected: Server A (15 conns), Server C (8 conns)
Route to: Server C (fewer connections)
```

This algorithm is used by Nginx, HAProxy, and Envoy due to its simplicity and effectiveness.

## Health Checks: The Foundation of Any Load Balancer

A load balancer is only useful if it routes to **healthy** backends. Health checks continuously probe backends and remove unhealthy ones from the rotation.

```csharp
// ASP.NET Core health check endpoint
builder.Services.AddHealthChecks()
    .AddSqlServer(connectionString, name: "database", tags: new[] { "ready" })
    .AddRedis(redisConnectionString, name: "redis", tags: new[] { "ready" })
    .AddCheck<CustomBusinessCheck>("business-invariants", tags: new[] { "ready" });

app.MapHealthChecks("/health/live", new HealthCheckOptions
{
    // Liveness: is the process running?
    Predicate = check => check.Tags.Contains("live") || !check.Tags.Any()
});

app.MapHealthChecks("/health/ready", new HealthCheckOptions
{
    // Readiness: is the service ready to accept traffic?
    Predicate = check => check.Tags.Contains("ready"),
    ResponseWriter = UIResponseWriter.WriteHealthCheckUIResponse
});
```

```yaml
# Kubernetes — readiness probe removes pod from Service endpoints if failing
readinessProbe:
  httpGet:
    path: /health/ready
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 10
  failureThreshold: 3

# Liveness probe restarts pod if it stops responding
livenessProbe:
  httpGet:
    path: /health/live
    port: 8080
  initialDelaySeconds: 30
  periodSeconds: 15
```

<Callout type="senior">
**Senior Insight: Sticky Sessions Are an Anti-Pattern for Horizontal Scaling**

Sticky sessions solve a stateful session problem by forcing statefulness into the load balancer. This creates several issues:

1. **Uneven load**: One server can receive disproportionate traffic if some users make more requests than others
2. **Failover breaks sessions**: If the sticky server dies, the user's session is lost
3. **Prevents true horizontal scaling**: You cannot freely add/remove servers without disrupting active sessions
4. **Deployment complexity**: Rolling deployments must drain stickied connections before restarting a server

The better solution: **externalize session state** (Redis, SQL) so any server can serve any request. Then use stateless round-robin or least-connections.

```csharp
// Replace in-memory session with distributed Redis session
builder.Services.AddStackExchangeRedisCache(options =>
{
    options.Configuration = builder.Configuration["Redis:ConnectionString"];
});

builder.Services.AddSession(options =>
{
    options.Cookie.HttpOnly = true;
    options.Cookie.IsEssential = true;
    options.IdleTimeout = TimeSpan.FromMinutes(30);
});
// Sessions now stored in Redis — any server can handle any request
```

WebSockets are the legitimate use case for sticky sessions — a WebSocket connection is stateful by nature and must stay on one server. Use sticky sessions specifically for WebSocket endpoints and stateless load balancing everywhere else.
</Callout>

<MythBuster
  myth="More load balancer instances always means better performance."
  reality="Load balancers themselves can become bottlenecks. An L7 load balancer that inspects and terminates TLS has CPU overhead — if you have 100,000 requests per second, a single NGINX instance on a 4-core VM will saturate. The fix is either horizontal scaling of the load balancer tier (typically done via L4 ECMP routing in front of L7 balancers) or using Azure Front Door / Azure Application Gateway which scale automatically. Additionally, adding load balancer tiers increases latency — each tier adds a network hop."
/>
