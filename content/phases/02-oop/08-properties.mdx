## Properties Are Methods in Disguise

At the IL level, a property is compiled into one or two methods: `get_PropertyName()` and `set_PropertyName()`. This matters because:
- Properties can be `virtual` and `override`
- Properties can be part of interfaces
- Properties can be abstract
- Properties cannot be passed by `ref` (unlike fields)
- Data-binding, serialization, and reflection frameworks all work with properties

```csharp
public class Circle
{
    private double _radius;

    // Full property — explicit backing field, validation in setter
    public double Radius
    {
        get => _radius;
        set
        {
            if (value < 0) throw new ArgumentOutOfRangeException(nameof(value), "Radius cannot be negative");
            _radius = value;
        }
    }

    // Computed property — derived from other state, no backing field
    public double Area => Math.PI * _radius * _radius;
    public double Circumference => 2 * Math.PI * _radius;

    // Expression-body property (same as above, just syntax sugar)
    public double Diameter => _radius * 2;
}
```

## Auto-Properties

Auto-properties let the compiler generate the backing field, eliminating boilerplate:

```csharp
public class Person
{
    // Auto-property: compiler creates a private backing field
    public string FirstName { get; set; } = "";
    public string LastName  { get; set; } = "";

    // Read-only auto-property: can only be set in constructor or field initializer
    public DateTime CreatedAt { get; } = DateTime.UtcNow;

    // Private setter: publicly readable, only writable from within the class
    public int LoginCount { get; private set; }

    // Protected setter: readable everywhere, settable by derived classes
    public string Status { get; protected set; } = "Active";

    public void RecordLogin() => LoginCount++; // uses private setter
}
```

<Callout type="info">
**Auto-property default values** (the `= value` initializers) run as part of field initialization, before the constructor body. They're shorthand for initializing the backing field. For reference types, always initialize auto-properties to avoid null reference warnings in .NET 6+ nullable-enabled projects.
</Callout>

## Init-Only Setters (C# 9)

The `init` accessor lets a property be set during object initialization but not afterward. This enables immutability without requiring constructor parameters for every field:

```csharp
public class OrderDto
{
    public int Id       { get; init; }
    public string Item  { get; init; } = "";
    public decimal Amount { get; init; }
    public DateTime PlacedAt { get; init; } = DateTime.UtcNow;
}

// Set during object initializer
var order = new OrderDto
{
    Id = 42,
    Item = "Widget",
    Amount = 9.99m
};

// Try to set after initialization — COMPILE ERROR
// order.Id = 100; // ERROR: Init-only property 'Id' can only be assigned in an object initializer

// 'with' expression creates a new object with some values changed
var modified = order with { Amount = 12.99m };
Console.WriteLine(order.Amount);   // 9.99 — original unchanged
Console.WriteLine(modified.Amount); // 12.99

// Records use init automatically — but you can use init on plain classes too:
public class ImmutablePoint
{
    public double X { get; init; }
    public double Y { get; init; }
    public double Z { get; init; }
}
```

<Callout type="tip">
**`init` vs `readonly` field:** A `readonly` field can only be set in the constructor. An `init` property can be set in the constructor OR in an object initializer. This makes `init` much more flexible for builder-style APIs and JSON deserialization (which `System.Text.Json` supports fully — it can set `init` properties during deserialization).
</Callout>

## Required Members (C# 11)

`required` forces consumers to provide a value during object initialization. Without it, forgetting to set a property goes unnoticed until runtime:

```csharp
public class DatabaseOptions
{
    // required: object initializer MUST provide these — compile error if missing
    public required string ConnectionString { get; init; }
    public required string DatabaseName     { get; init; }

    // Optional with default
    public int CommandTimeoutSeconds { get; init; } = 30;
    public bool EnableRetry          { get; init; } = true;
}

// COMPILE ERROR if you miss required members:
// var opts = new DatabaseOptions(); // ERROR: Required member 'ConnectionString' not set
// var opts = new DatabaseOptions { ConnectionString = "..." }; // ERROR: DatabaseName missing

// CORRECT:
var opts = new DatabaseOptions
{
    ConnectionString = "Server=localhost;Database=MyDb",
    DatabaseName     = "MyDb"
    // CommandTimeoutSeconds not provided — uses default of 30
};

// To suppress required for a specific constructor (e.g., for deserialization):
public class DatabaseOptions
{
    public required string ConnectionString { get; init; }
    public required string DatabaseName     { get; init; }

    [SetsRequiredMembers] // Tells compiler this constructor sets all required members
    public DatabaseOptions(string connectionString, string databaseName)
    {
        ConnectionString = connectionString;
        DatabaseName = databaseName;
    }

    public DatabaseOptions() { } // Parameterless — for deserializers
}
```

## Asymmetric Visibility

Properties can have different access levels for getter and setter:

```csharp
public class Account
{
    // Publicly readable, only settable within the class
    public decimal Balance { get; private set; }

    // Publicly readable, settable by derived classes
    public string Status { get; protected set; } = "Active";

    // Publicly readable, settable within the assembly
    public int TransactionCount { get; internal set; }

    // Publicly readable, init-only (set only via object initializer or constructor)
    public string AccountNumber { get; init; } = "";

    public void Deposit(decimal amount)
    {
        if (amount <= 0) throw new ArgumentOutOfRangeException(nameof(amount));
        Balance += amount;         // private set — only accessible here
        TransactionCount++;        // internal set — accessible within assembly
    }
}
```

<Callout type="senior">
**Senior Insight: Property Changed Notification and Lazy Initialization**

Two patterns worth knowing for production code:

```csharp
// Pattern 1: INotifyPropertyChanged for MVVM/data binding
public class ViewModel : INotifyPropertyChanged
{
    private string _name = "";

    public string Name
    {
        get => _name;
        set
        {
            if (_name == value) return; // Avoid unnecessary notifications
            _name = value;
            OnPropertyChanged();        // Notify binding subscribers
        }
    }

    public event PropertyChangedEventHandler? PropertyChanged;

    protected virtual void OnPropertyChanged([CallerMemberName] string? propertyName = null)
        => PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propertyName));
}

// Pattern 2: Lazy initialization in a property
public class ExpensiveService
{
    // Lazy<T> is thread-safe by default (LazyThreadSafetyMode.ExecutionAndPublication)
    private readonly Lazy<DatabaseConnection> _connection
        = new(() => new DatabaseConnection(/* expensive init */));

    public DatabaseConnection Connection => _connection.Value;

    // Without Lazy<T> — thread-unsafe double-checked locking (get it wrong and it's a bug):
    private DatabaseConnection? _conn;
    private readonly object _lock = new();

    public DatabaseConnection ConnectionManual
    {
        get
        {
            if (_conn is null)
                lock (_lock)
                    _conn ??= new DatabaseConnection();
            return _conn;
        }
    }
}
```

`Lazy<T>` with no arguments uses `LazyThreadSafetyMode.ExecutionAndPublication` — only one thread executes the factory, others wait. Pass `isThreadSafe: false` if you know the property will only be accessed from one thread (saves a lock acquisition on every read).
</Callout>
