## Why This Principle Exists

Inheritance creates one of the tightest forms of coupling in OOP. A derived class is coupled to its base class's:
- **Implementation details** (not just the public contract)
- **Protected members** (the "fragile base class" problem)
- **Constructor signatures** (every derived class must chain to a base constructor)
- **Behavioral assumptions** (every override must honor Liskov Substitution)

When you inherit for code reuse rather than for a genuine is-a relationship, you pay all these coupling costs for no semantic benefit. Composition lets you reuse behavior without the coupling.

## The Classic Anti-Pattern: Inheriting for Reuse

```csharp
// WRONG: inheriting from Stack to get "List with undo" behavior
// Stack<T> is a LIFO structure — this has nothing to do with undo history
public class UndoableList<T> : Stack<T>
{
    // We wanted the push/pop for undo, but now UndoableList IS a Stack
    // Callers can call Push, Pop, Peek on our "list" type — which is wrong!

    private readonly List<T> _items = new();

    public void AddItem(T item)
    {
        _items.Add(item);
        Push(item); // push to the inherited Stack for undo tracking
    }

    public void UndoLast()
    {
        if (Count > 0)
        {
            Pop();
            _items.RemoveAt(_items.Count - 1);
        }
    }
}

// Problem: callers can bypass our API and use the inherited Stack methods directly
var list = new UndoableList<int>();
list.AddItem(1);
list.Push(99);       // Bypasses our list — Stack state is now out of sync!
list.UndoLast();     // Now what? The Stack and List are inconsistent
```

---

## The Composition Alternative

Instead of inheriting, compose the type you want to reuse:

```csharp
// CORRECT: compose the behaviors you need
public class UndoableList<T>
{
    private readonly List<T> _items = new();
    private readonly Stack<T> _undoStack = new(); // private implementation detail

    // Expose ONLY the API that makes sense for an undoable list
    public int Count => _items.Count;
    public IReadOnlyList<T> Items => _items.AsReadOnly();

    public void Add(T item)
    {
        _items.Add(item);
        _undoStack.Push(item);
    }

    public bool UndoLast()
    {
        if (_undoStack.Count == 0) return false;
        _undoStack.Pop();
        _items.RemoveAt(_items.Count - 1);
        return true;
    }
}

// Now callers CANNOT call Push/Pop/Peek on our type — those are implementation details
var list = new UndoableList<int>();
list.Add(1);
list.Add(2);
list.UndoLast();
Console.WriteLine(list.Count); // 1 — clean, no way to corrupt state
```

## Real-World Pattern: The Decorator

The Decorator pattern is composition in its purest form — wrapping an object to add behavior without modifying the original class:

```csharp
public interface IOrderService
{
    Task<Order?> GetOrderAsync(int id);
    Task<int> CreateOrderAsync(Order order);
}

// Core implementation
public class OrderService : IOrderService
{
    private readonly IOrderRepository _repo;
    public OrderService(IOrderRepository repo) => _repo = repo;

    public Task<Order?> GetOrderAsync(int id) => _repo.GetByIdAsync(id);
    public Task<int> CreateOrderAsync(Order order) => _repo.InsertAsync(order);
}

// Decorator 1: adds logging — no changes to OrderService
public class LoggingOrderService : IOrderService
{
    private readonly IOrderService _inner;
    private readonly ILogger<LoggingOrderService> _logger;

    public LoggingOrderService(IOrderService inner, ILogger<LoggingOrderService> logger)
    {
        _inner  = inner;
        _logger = logger;
    }

    public async Task<Order?> GetOrderAsync(int id)
    {
        _logger.LogInformation("Getting order {Id}", id);
        var order = await _inner.GetOrderAsync(id);
        _logger.LogInformation("Got order {Id}: {Found}", id, order is not null);
        return order;
    }

    public async Task<int> CreateOrderAsync(Order order)
    {
        _logger.LogInformation("Creating order for {Item}", order.ItemName);
        var id = await _inner.CreateOrderAsync(order);
        _logger.LogInformation("Created order {Id}", id);
        return id;
    }
}

// Decorator 2: adds caching — no changes to OrderService or LoggingOrderService
public class CachingOrderService : IOrderService
{
    private readonly IOrderService _inner;
    private readonly IMemoryCache _cache;

    public CachingOrderService(IOrderService inner, IMemoryCache cache)
    {
        _inner = inner;
        _cache = cache;
    }

    public async Task<Order?> GetOrderAsync(int id)
    {
        var cacheKey = $"order:{id}";
        if (_cache.TryGetValue<Order>(cacheKey, out var cached))
            return cached;

        var order = await _inner.GetOrderAsync(id);
        if (order is not null)
            _cache.Set(cacheKey, order, TimeSpan.FromMinutes(5));
        return order;
    }

    public Task<int> CreateOrderAsync(Order order) => _inner.CreateOrderAsync(order);
}

// In DI setup — layers compose cleanly
// services.AddScoped<IOrderService>(sp =>
//     new CachingOrderService(
//         new LoggingOrderService(
//             new OrderService(sp.GetRequiredService<IOrderRepository>()),
//             sp.GetRequiredService<ILogger<LoggingOrderService>>()),
//         sp.GetRequiredService<IMemoryCache>()));
```

## Real-World Pattern: The Strategy

Strategy replaces inheritance-based behavior variation with injected behavior:

```csharp
// WRONG: inheriting to vary one behavior
public abstract class ReportExporter
{
    public abstract void ExportTo(Stream output, ReportData data); // varies
    public void Log(string msg) => Console.WriteLine(msg);        // shared
    // But now: PdfExporter, CsvExporter, ExcelExporter all inherit Report
    // They're coupled to the base class, can't be composed, can't be tested in isolation
}

// CORRECT: inject the varying behavior as a strategy
public interface IExportStrategy
{
    void Export(Stream output, ReportData data);
}

public class PdfExportStrategy : IExportStrategy
{
    public void Export(Stream output, ReportData data) { /* PDF logic */ }
}

public class CsvExportStrategy : IExportStrategy
{
    public void Export(Stream output, ReportData data)
    {
        using var writer = new StreamWriter(output, leaveOpen: true);
        writer.WriteLine(string.Join(",", data.Headers));
        foreach (var row in data.Rows)
            writer.WriteLine(string.Join(",", row));
    }
}

// The exporter composes the strategy — can change at runtime
public class ReportExporter
{
    private readonly IExportStrategy _strategy;
    private readonly ILogger<ReportExporter> _logger;

    public ReportExporter(IExportStrategy strategy, ILogger<ReportExporter> logger)
    {
        _strategy = strategy;
        _logger = logger;
    }

    public void Export(Stream output, ReportData data)
    {
        _logger.LogInformation("Exporting report using {Strategy}", _strategy.GetType().Name);
        _strategy.Export(output, data);
        _logger.LogInformation("Export complete");
    }
}
```

<Callout type="senior">
**Senior Insight: Inheritance Is Not Bad — Misused Inheritance Is Bad**

The principle is "prefer composition" — not "never use inheritance." There are cases where inheritance is the right tool:

1. **True specialization**: `ArgumentNullException` IS an `ArgumentException` IS an `Exception`. The hierarchy communicates semantics, not just code reuse.

2. **Template Method pattern**: Base class defines the algorithm skeleton with abstract steps — derived classes fill in the steps. This is intentional inheritance design, not accidental.

3. **Framework integration**: `ControllerBase`, `BackgroundService`, `DbContext` — framework types designed explicitly for inheritance, with documented extension points and contracts.

The signal that inheritance is being misused:
- "I need the Save() method from UserRepository, so OrderRepository will extend UserRepository"
- "We share 80% of logic, let's create a base class" (without asking if the types truly share identity)
- Protected methods that are implementation details, not documented extension points
- Constructor parameters exploding as the hierarchy grows (passing 12 args to base())

When you see these patterns, composition is almost always the cleaner solution. Extract the shared behavior into a service or strategy, inject it, and keep the types independent.

```csharp
// Before: inheritance for code reuse (bad)
public class UserRepository : BaseRepository<User>
{
    // Inherits 20 methods from BaseRepository, uses 3 of them
}
public class OrderRepository : BaseRepository<Order>
{
    // Also inherits those 20 methods, uses 5 different ones
}

// After: composition with shared utilities (good)
public class UserRepository
{
    private readonly IDbQueryExecutor _executor; // injected — contains shared query logic
    private readonly IDbTransactionManager _tx;  // injected — shared transaction logic

    // Uses exactly the methods it needs, no more
    public Task<User?> GetByIdAsync(int id) => _executor.QueryFirstOrDefaultAsync<User>(...);
}
```
</Callout>

<MythBuster
  myth="Composition over inheritance means you should never use inheritance in C#."
  reality="The principle means: when choosing between inheritance and composition for CODE REUSE, prefer composition. Inheritance for subtype polymorphism (Exception hierarchies, abstract base classes designed for extension, framework integration) is perfectly valid and often correct. The GOF authors who coined the principle still used inheritance extensively — they just didn't inherit from concrete classes for the sole purpose of getting free methods."
