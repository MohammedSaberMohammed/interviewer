## What Partial Classes Do

The `partial` keyword lets a single class, struct, or interface definition span multiple source files. At compile time, all `partial` parts are merged into one class definition. From the CLR's perspective, there is no such thing as a partial class — it's just a class.

```csharp
// File: Customer.cs
public partial class Customer
{
    public int Id { get; set; }
    public string Name { get; set; } = "";
    public string Email { get; set; } = "";
}

// File: Customer.Orders.cs  (same namespace, same assembly)
public partial class Customer
{
    private readonly List<Order> _orders = new();
    public IReadOnlyList<Order> Orders => _orders.AsReadOnly();

    public void AddOrder(Order order)
    {
        if (order is null) throw new ArgumentNullException(nameof(order));
        _orders.Add(order);
    }
}

// File: Customer.Validation.cs
public partial class Customer
{
    public bool IsValid()
        => !string.IsNullOrWhiteSpace(Name) &&
           !string.IsNullOrWhiteSpace(Email) &&
           Email.Contains('@');
}

// Compiled result: one class Customer with all members from all three files
var c = new Customer { Id = 1, Name = "Alice", Email = "alice@example.com" };
c.AddOrder(new Order());
Console.WriteLine(c.IsValid()); // true — method from Customer.Validation.cs
```

## The Primary Use Case: Generated + Hand-Written Code

Partial classes exist primarily to let **code generators** (or designers) write one half of a class while **developers** write the other half. The two halves stay in separate files — the generated file can be regenerated without overwriting hand-written code.

```csharp
// Generated file: Customer.g.cs (DO NOT EDIT — generated by EF Core)
// [This simulates what EF Core's scaffolding might produce]
public partial class Customer
{
    public int CustomerId { get; set; }
    public string? CompanyName { get; set; }
    public string? ContactName { get; set; }

    // Navigation properties
    public virtual ICollection<Order> Orders { get; set; } = new List<Order>();
}

// Hand-written file: Customer.cs (developer writes this)
public partial class Customer
{
    // Business logic that survives database schema regeneration
    public string DisplayName
        => ContactName ?? CompanyName ?? $"Customer #{CustomerId}";

    public bool HasOrders => Orders.Any();

    public decimal TotalOrderValue
        => Orders.SelectMany(o => o.OrderDetails)
                 .Sum(d => d.UnitPrice * d.Quantity);
}
```

**Real-world scenarios where partial classes appear:**
- **WinForms / WPF designers**: `Form1.Designer.cs` (generated) + `Form1.cs` (your code)
- **Entity Framework scaffolding**: generated entity classes + your business logic
- **ASP.NET Razor Pages**: `Page.cshtml.cs` (hand-written) — though Blazor uses this differently
- **Source generators**: tools that add members to your classes at compile time

## Partial Methods

A partial method is a method declared in one partial part and optionally implemented in another:

```csharp
// Part 1: declaration (in generated code)
public partial class DataModel
{
    // Declaration: no implementation
    partial void OnPropertyChanged(string propertyName);

    public string Name
    {
        get => _name;
        set
        {
            _name = value;
            OnPropertyChanged(nameof(Name)); // Call may be eliminated if not implemented
        }
    }
    private string _name = "";
}

// Part 2: implementation (in hand-written code — optional)
public partial class DataModel
{
    // Provide the implementation
    partial void OnPropertyChanged(string propertyName)
    {
        Console.WriteLine($"Property changed: {propertyName}");
        // Could also raise INotifyPropertyChanged events, log, validate, etc.
    }
}
```

<Callout type="info">
**If a partial method has no implementation, the compiler removes all calls to it entirely.** There is zero runtime overhead. This makes partial methods a clean extension point in generated code: "call this hook if the developer provides one, otherwise do nothing." No reflection, no delegates, no overhead.
</Callout>

## Partial Method Rules (Classic vs Extended)

C# 9 extended partial methods. There are now two kinds:

<ComparisonTable
  headers={["Feature", "Classic partial methods (pre-C# 9)", "Extended partial methods (C# 9+)"]}
  rows={[
    ["Return type", "void only", "Any return type"],
    ["Access modifier", "Implicitly private", "Any access modifier (including public)"],
    ["out parameters", "Not allowed", "Allowed"],
    ["Implementation required", "No — eliminated if missing", "Yes — must be implemented"],
    ["Primary use", "Optional hooks in generated code", "Source generator API contracts"],
  ]}
/>

```csharp
// Extended partial methods (C# 9+): can have return types and access modifiers
// But MUST be implemented — they're not optional
public partial class UserService
{
    // Source generator declares this — MUST be implemented
    public partial User CreateUser(string email, string name);

    // Generator can also declare with implementation as the extension point
    partial void ValidateUserData(string email, string name);
}

public partial class UserService
{
    public partial User CreateUser(string email, string name)
    {
        ValidateUserData(email, name);
        return new User { Email = email, Name = name, CreatedAt = DateTime.UtcNow };
    }
}
```

<Callout type="senior">
**Senior Insight: Partial Classes Power Source Generators**

Source generators (Roslyn SDK, .NET 5+) use partial classes as their core mechanism. A source generator inspects your code at compile time and generates additional partial class files that are injected into the compilation. This is how many modern .NET frameworks work:

```csharp
// You write this in your code:
[GenerateLogger]
public partial class OrderService
{
    public void ProcessOrder(int orderId) { /* your logic */ }
}

// The source generator sees your class and generates (in memory, added to compilation):
// OrderService.g.cs
public partial class OrderService
{
    private readonly ILogger<OrderService> _logger;

    // Generated: strongly-typed, zero-allocation logging method
    public void LogOrderProcessed(int orderId)
        => _logger.Log(LogLevel.Information, "Processing order {OrderId}", orderId);
}
```

Real examples in .NET ecosystem:
- `System.Text.Json` source gen: `[JsonSerializable]` generates serializer code (no reflection, AOT-safe)
- `LoggerMessage.Define`: Microsoft.Extensions.Logging generates strongly-typed log methods
- `INotifyPropertyChanged` generators (CommunityToolkit.Mvvm): generates boilerplate for MVVM
- Regex source gen: `[GeneratedRegex("pattern")]` generates compiled regex at build time

When you see `partial` on a class you didn't put it there, look for a source generator attribute — that's where the other half lives. Understanding this is increasingly important as the ecosystem moves toward AOT compilation and trimming, both of which require source generators over reflection.
</Callout>

<MythBuster
  myth="Partial classes are mainly useful for organizing large classes into smaller files."
  reality="Using partial classes purely to split a large class into multiple files is an anti-pattern — if a class is so big you need multiple files, it's violating the Single Responsibility Principle and needs to be refactored. The legitimate use cases are: (1) separating generated code from hand-written code, (2) source generator integration, (3) designer support (WinForms/WPF). If you're splitting a hand-written class across files just for size, that's a design smell."
/>
