## What Is the Task-based Async Pattern?

The **Task-based Async Pattern (TAP)** is the modern approach to asynchronous programming in .NET, introduced in .NET 4.5 alongside the `async`/`await` keywords. It is built on two core types:

- `Task` — represents an asynchronous operation that produces no value (like `void`)
- `Task<T>` — represents an asynchronous operation that produces a value of type `T`

Before TAP, .NET had two older patterns:
- **APM** (Asynchronous Programming Model): `BeginXxx`/`EndXxx` pairs with `IAsyncResult`
- **EAP** (Event-based Asynchronous Pattern): `OperationAsync()` + `OperationCompleted` event

TAP replaced both. If you encounter APM APIs today, `Task.Factory.FromAsync` wraps them.

## Task States

A `Task` is a promise — it starts in one state and transitions through a lifecycle:

```csharp
// A task in each of the terminal states
Task completed = Task.CompletedTask;             // Status: RanToCompletion
Task<int> completedWithValue = Task.FromResult(42); // Status: RanToCompletion
Task cancelled = Task.FromCanceled(new CancellationToken(true)); // Status: Canceled
Task faulted = Task.FromException(new IOException("disk error")); // Status: Faulted

Console.WriteLine(completed.Status);      // RanToCompletion
Console.WriteLine(completed.IsCompleted); // True
Console.WriteLine(faulted.IsFaulted);     // True
Console.WriteLine(faulted.Exception);     // AggregateException wrapping IOException
```

The states are: **Created → WaitingForActivation → WaitingToRun → Running → [RanToCompletion | Canceled | Faulted]**

Most tasks you create with `async`/`await` skip directly to `WaitingForActivation` — they are "hot" (already started) by default.

## The async/await Keywords

`async` marks a method as asynchronous. `await` suspends the current method until the awaited task completes, **without blocking the calling thread**.

```csharp
// WITHOUT async/await — blocking (bad for scalability)
public string DownloadPageBlocking(string url)
{
    using var client = new HttpClient();
    return client.GetStringAsync(url).Result; // BLOCKS the thread — danger!
}

// WITH async/await — non-blocking (correct)
public async Task<string> DownloadPageAsync(string url)
{
    using var client = new HttpClient();
    return await client.GetStringAsync(url); // Releases thread while waiting
}
```

When you `await` a task:
1. If the task is already complete, execution continues synchronously (no overhead)
2. If not complete, the current method is **suspended** — the thread returns to the caller
3. When the task completes, execution **resumes** after the `await` (on the captured context)

## Async Method Return Types

```csharp
// Task — for async operations with no return value
public async Task SendEmailAsync(string to, string body)
{
    await smtpClient.SendMailAsync(new MailMessage("from@ex.com", to, "Subject", body));
}

// Task<T> — for async operations returning a value
public async Task<User> GetUserAsync(int id)
{
    var json = await httpClient.GetStringAsync($"/users/{id}");
    return JsonSerializer.Deserialize<User>(json)!;
}

// ValueTask<T> — for hot paths where the result is often synchronously available
public async ValueTask<int> GetCachedCountAsync()
{
    if (_cache.TryGetValue("count", out int cached))
        return cached; // Synchronous path — no heap allocation!

    int count = await _db.QueryScalarAsync<int>("SELECT COUNT(*) FROM items");
    _cache["count"] = count;
    return count;
}

// IAsyncEnumerable<T> — for async streams (covered in lesson 08)
public async IAsyncEnumerable<Product> StreamProductsAsync()
{
    await foreach (var row in _db.QueryAsync("SELECT * FROM products"))
        yield return MapProduct(row);
}
```

## Common TAP Combinators

```csharp
// Run two things in parallel and wait for both
var (user, orders) = await (
    GetUserAsync(userId),
    GetOrdersAsync(userId)
).WhenAll(); // Extension method, or:

Task<User> userTask = GetUserAsync(userId);
Task<Order[]> ordersTask = GetOrdersAsync(userId);
await Task.WhenAll(userTask, ordersTask);
User user = userTask.Result;     // .Result is safe AFTER WhenAll completes
Order[] orders = ordersTask.Result;

// Take the first result from multiple concurrent operations
string fastest = await Task.WhenAny(
    FetchFromPrimaryAsync(),
    FetchFromReplicaAsync()
).Unwrap(); // WhenAny returns Task<Task<string>>, Unwrap gets the inner result

// Run on the thread pool (CPU-bound work)
int result = await Task.Run(() => ExpensiveCpuCalculation(data));

// Delay without blocking
await Task.Delay(TimeSpan.FromSeconds(1));
```

<Callout type="warning">
**WhenAll and exceptions**: If multiple tasks throw, `WhenAll` collects all exceptions into an `AggregateException`. When you `await` it, only the first exception is rethrown. To see all exceptions, inspect `task.Exception.InnerExceptions` or use `Task.WhenAll(tasks).ContinueWith(...)`.
</Callout>

## Async All the Way Down

One of the most critical TAP rules: **async propagates upward**. Once you go async, every caller should also be async.

```csharp
// WRONG: mixing sync and async — causes deadlocks in some contexts
public string GetDataWrong()
{
    return GetDataAsync().Result; // Potential deadlock in ASP.NET Framework / WinForms!
}

// RIGHT: async all the way up
public async Task<string> GetDataRight()
{
    return await GetDataAsync();
}

// The entry point adapts — console apps can use .GetAwaiter().GetResult() at the top level ONLY
static async Task Main(string[] args) // C# 7.1+ supports async Main
{
    var result = await GetDataRight();
    Console.WriteLine(result);
}
```

<MythBuster
  myth="async makes your code run faster"
  reality="async/await does not add parallelism or speed up individual operations. It frees the calling thread while waiting for I/O — making your app more scalable (more concurrent requests with the same thread count), not faster per request. A single HTTP request takes the same time whether async or sync. The benefit is at scale."

<Callout type="senior">
**Senior Insight: ValueTask vs Task allocation cost**

`Task<T>` always allocates a heap object. `ValueTask<T>` is a struct — when the result is synchronously available, it completes with zero heap allocation. This is why all high-frequency BCL methods (like `Stream.ReadAsync`) return `ValueTask<int>` in .NET Core.

**Rules for choosing:**
- Use `Task<T>` by default — simpler, safer
- Use `ValueTask<T>` only for high-frequency code paths where profiling shows Task allocation is measurable overhead AND the synchronous path is common (like a cache lookup)
- Never `await` a `ValueTask<T>` more than once — it is consumed on first await
- Never store a `ValueTask<T>` in a field and await it later — use `.AsTask()` if you need that

```csharp
// WRONG: awaiting ValueTask twice is undefined behavior
var vt = GetCachedCountAsync();
var a = await vt; // OK
var b = await vt; // UNDEFINED — may throw, return wrong value, or corrupt state

// RIGHT: convert to Task if you need multiple awaits
Task<int> task = GetCachedCountAsync().AsTask();
var a = await task;
var b = await task; // Safe — Task can be awaited multiple times
```
</Callout>
