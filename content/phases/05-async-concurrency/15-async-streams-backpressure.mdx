## What Is Backpressure?

**Backpressure** is the mechanism by which a slow consumer signals to a fast producer to slow down. Without backpressure, a fast producer fills memory without bound until the process crashes.

```
Without backpressure:
Producer ──[unbounded buffer]──────────────────────→ Consumer
         (fills up, OOM crash)

With backpressure:
Producer ──[bounded buffer]──→ (buffer full?) ──→ Consumer
         ← slow down signal ←

```

In .NET, you have two main tools: `IAsyncEnumerable<T>` (implicit backpressure) and `Channel<T>` (explicit backpressure).

---

## IAsyncEnumerable: Implicit Backpressure

`IAsyncEnumerable<T>` is a **pull-based** model. The consumer controls the pace. The producer only runs when the consumer calls `MoveNextAsync()`:

```csharp
// Producer runs ONE step per consumer MoveNextAsync() call
public async IAsyncEnumerable<int> ProduceAsync(
    [EnumeratorCancellation] CancellationToken ct = default)
{
    for (int i = 0; i < 1_000_000; i++)
    {
        // Each iteration is gated by the consumer calling MoveNextAsync()
        // If consumer is slow, this loop is slow — natural backpressure
        await Task.Delay(1, ct); // Simulate some async production
        yield return i;
    }
}

// Consumer — its pace controls the producer
await foreach (var item in ProduceAsync(ct))
{
    await ProcessSlowlyAsync(item, ct); // If this is slow, producer slows down
}
```

This is clean and elegant, but has a constraint: **producer and consumer are tightly coupled** — they run on the same logical "thread" (the async chain). The producer cannot run ahead to pre-fetch items.

---

## Channel: Explicit Backpressure with Decoupled Lifetimes

`Channel<T>` is a **push-based** model with an explicit buffer. Producer and consumer run independently and concurrently:

```csharp
// Decoupled: producer and consumer run independently in parallel
public async Task RunPipelineAsync(CancellationToken ct)
{
    var channel = Channel.CreateBounded<WorkItem>(new BoundedChannelOptions(50)
    {
        FullMode = BoundedChannelFullMode.Wait // Backpressure: producer awaits when full
    });

    // Producer runs concurrently — can pre-fetch up to 50 items ahead
    var producer = Task.Run(async () =>
    {
        try
        {
            await foreach (var item in _source.StreamAsync(ct))
                await channel.Writer.WriteAsync(item, ct); // Backpressure here
        }
        finally { channel.Writer.Complete(); }
    }, ct);

    // Consumer runs independently — processes at its own pace
    var consumer = Task.Run(async () =>
    {
        await foreach (var item in channel.Reader.ReadAllAsync(ct))
            await ProcessAsync(item, ct);
    }, ct);

    await Task.WhenAll(producer, consumer);
}
```

The channel's bounded buffer allows the producer to be N items ahead, improving throughput when there's latency variance in either stage.

## The Hybrid Pattern: Channel as IAsyncEnumerable Source

You can bridge the two worlds — use a Channel internally but expose `IAsyncEnumerable<T>` to callers:

```csharp
// Channel powers the internal pipeline; IAsyncEnumerable exposed as the API
public async IAsyncEnumerable<ProcessedResult> ProcessStreamAsync(
    IAsyncEnumerable<RawData> input,
    [EnumeratorCancellation] CancellationToken ct = default)
{
    // Internal bounded buffer: decouple I/O reading from processing
    var channel = Channel.CreateBounded<ProcessedResult>(new BoundedChannelOptions(20)
    {
        FullMode = BoundedChannelFullMode.Wait
    });

    // Background producer: processes input and writes to channel
    var producer = Task.Run(async () =>
    {
        try
        {
            await foreach (var raw in input.WithCancellation(ct))
            {
                var result = await TransformAsync(raw, ct); // CPU-heavy transform
                await channel.Writer.WriteAsync(result, ct);
            }
        }
        finally { channel.Writer.TryComplete(); }
    }, ct);

    // Yield results to caller as IAsyncEnumerable
    await foreach (var result in channel.Reader.ReadAllAsync(ct))
        yield return result;

    // Ensure producer completed (and propagate exceptions)
    await producer;
}
```

## Multi-Stage Pipeline with Backpressure

```csharp
// 3-stage pipeline: Fetch → Transform → Persist
// Each stage has its own bounded buffer providing backpressure between stages
public async Task RunMultiStagePipelineAsync(CancellationToken ct)
{
    // Stage 1 → Stage 2 buffer
    var fetchToTransform = Channel.CreateBounded<RawItem>(
        new BoundedChannelOptions(100) { FullMode = BoundedChannelFullMode.Wait });

    // Stage 2 → Stage 3 buffer
    var transformToPersist = Channel.CreateBounded<ProcessedItem>(
        new BoundedChannelOptions(50) { FullMode = BoundedChannelFullMode.Wait });

    // Stage 1: Fetch (1 producer, fast network reader)
    var fetch = Task.Run(async () =>
    {
        try
        {
            await foreach (var item in _api.StreamItemsAsync(ct))
                await fetchToTransform.Writer.WriteAsync(item, ct);
        }
        finally { fetchToTransform.Writer.TryComplete(); }
    }, ct);

    // Stage 2: Transform (3 parallel workers, CPU-heavy)
    var transforms = Enumerable.Range(0, 3).Select(_ => Task.Run(async () =>
    {
        await foreach (var raw in fetchToTransform.Reader.ReadAllAsync(ct))
        {
            var processed = await TransformAsync(raw, ct);
            await transformToPersist.Writer.WriteAsync(processed, ct);
        }
    }, ct)).ToArray();

    // Stage 3: Persist (2 parallel workers, slow DB writes)
    var persists = Enumerable.Range(0, 2).Select(_ => Task.Run(async () =>
    {
        await foreach (var item in transformToPersist.Reader.ReadAllAsync(ct))
            await _db.InsertAsync(item, ct);
    }, ct)).ToArray();

    // Complete stage 2's channel when all transform workers finish
    await Task.WhenAll(transforms);
    transformToPersist.Writer.TryComplete();

    // Wait for all stages
    await Task.WhenAll(fetch);
    await Task.WhenAll(persists);
}
```

<Callout type="senior">
**Senior Insight: Backpressure propagation and flow control**

The multi-stage pipeline above has automatic backpressure propagation:
- If `persists` (stage 3) are slow, `transformToPersist` fills up
- Full `transformToPersist` causes `transforms` (stage 2) workers to await on `WriteAsync`
- Blocked `transforms` workers stop reading from `fetchToTransform`
- Full `fetchToTransform` causes `fetch` (stage 1) to await on `WriteAsync`
- Stage 1 stops fetching from the network

This is the reactive backpressure model in action — slowdown propagates backward through the pipeline without polling, overflow, or manual signaling. The bounded buffers are the key: each one acts as a speed governor between its two adjacent stages.

For production pipelines, tuning buffer sizes is important:
- **Too small**: excessive context switching (producer and consumer wake each other constantly)
- **Too large**: high memory usage; slow consumers fall far behind before backpressure engages
- **Rule of thumb**: buffer size = (producer throughput per second) × (acceptable latency in seconds)
</Callout>
