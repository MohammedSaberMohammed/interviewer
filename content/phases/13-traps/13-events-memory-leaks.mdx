## The Trap

```csharp
// Publisher — long-lived service (singleton)
public class StockPriceService
{
    public event EventHandler<PriceChangedEventArgs>? PriceChanged;

    public void UpdatePrice(string symbol, decimal price)
    {
        PriceChanged?.Invoke(this, new PriceChangedEventArgs(symbol, price));
    }
}

// Subscriber — short-lived view model, created per page navigation
public class StockViewModel
{
    public StockViewModel(StockPriceService service)
    {
        // Subscribe to the singleton's event
        service.PriceChanged += OnPriceChanged; // THE TRAP
    }

    private void OnPriceChanged(object? sender, PriceChangedEventArgs e)
    {
        Console.WriteLine($"{e.Symbol}: {e.Price}");
    }

    // No Dispose, no unsubscribe!
}

// Every time the user navigates to the stock page:
var vm = new StockViewModel(stockService);
// User navigates away — vm goes "out of scope"
// But vm is NOT collected! StockPriceService.PriceChanged holds a delegate
// that references vm via the OnPriceChanged method target.
// The GC sees: StockPriceService → PriceChanged delegate → StockViewModel
// StockViewModel is reachable — it will NEVER be collected!
```

<Callout type="trap">
An event subscription (`publisher.Event += handler`) creates a **strong reference** from the publisher's delegate list to the subscriber object (via the handler's `Target` property). As long as the publisher is alive, all subscribers that have not unsubscribed are kept alive, regardless of whether anything else references them.
</Callout>

## Why It Happens

```csharp
// What event += actually stores:
// event += handler; stores a MulticastDelegate with:
// - Target: the object that contains OnPriceChanged (the StockViewModel instance)
// - Method: the MethodInfo for OnPriceChanged

// GC root chain:
// App (GC root)
// └── StockPriceService (singleton — rooted by DI container)
//     └── PriceChanged (MulticastDelegate)
//         └── [0] Delegate { Target = vm1, Method = OnPriceChanged }  ← KEEPS vm1 ALIVE
//         └── [1] Delegate { Target = vm2, Method = OnPriceChanged }  ← KEEPS vm2 ALIVE
//         └── [2] Delegate { Target = vm3, Method = OnPriceChanged }  ← KEEPS vm3 ALIVE
//         ... one per navigation without unsubscribe
```

Over time: a growing list of zombie view models, each holding their own references to services, DbContexts, timers, etc. Memory grows with each page visit, never released.

## The Fix

**Option 1: Unsubscribe in Dispose (IDisposable pattern)**

```csharp
// CORRECT — always unsubscribe in Dispose
public class StockViewModel : IDisposable
{
    private readonly StockPriceService _service;
    private bool _disposed;

    public StockViewModel(StockPriceService service)
    {
        _service = service;
        _service.PriceChanged += OnPriceChanged; // Subscribe
    }

    private void OnPriceChanged(object? sender, PriceChangedEventArgs e)
    {
        Console.WriteLine($"{e.Symbol}: {e.Price}");
    }

    public void Dispose()
    {
        if (!_disposed)
        {
            _service.PriceChanged -= OnPriceChanged; // Unsubscribe!
            _disposed = true;
        }
    }
}

// Caller must ensure Dispose is called:
using var vm = new StockViewModel(stockService);
// Or DI framework disposes it when the scope ends (if registered as Scoped/Transient)
```

**Option 2: Use CancellationToken instead of events for lifecycle**

```csharp
// Modern pattern: use async streams or channels instead of events
public class StockViewModel
{
    private readonly CancellationTokenSource _cts = new();

    public async Task StartAsync(StockPriceService service)
    {
        await foreach (var price in service.GetPriceStreamAsync(_cts.Token))
        {
            Console.WriteLine($"{price.Symbol}: {price.Price}");
        }
    }

    public void Stop() => _cts.Cancel(); // Clean cancellation, no dangling references
}
```

  truth="The event subscription creates a strong reference FROM the publisher TO the subscriber. The subscriber is kept alive by this reference regardless of other scope. The GC cannot collect anything that is still reachable."
/>

## Static Events — An Even Worse Trap

```csharp
// Static events hold references FOREVER — the publisher never goes away
public static class AppEvents
{
    public static event EventHandler? ThemeChanged;
}

public class ThemeAwareControl : Control
{
    public ThemeAwareControl()
    {
        AppEvents.ThemeChanged += OnThemeChanged; // Static event!
    }

    private void OnThemeChanged(object? s, EventArgs e) { }
    // If not unsubscribed, this control lives FOREVER
    // even after it is removed from the visual tree and seemingly "gone"
}
```

<Callout type="senior">
**The WPF/WinForms memory leak that took 3 weeks to find**

A desktop application's memory usage grew by ~2MB per minute. `dotnet-dump` and memory profiler analysis showed thousands of `MainViewModel` instances alive. Each was subscribed to a singleton `ApplicationStateService.StateChanged` event but never unsubscribed.

The fix: implement `IDisposable` on all ViewModels, have the view's `Unloaded` event call `viewModel.Dispose()`. After this change, memory was stable.

Alternatively, the team moved to `WeakEventManager` (WPF built-in) which uses `WeakReference<T>` under the hood — the subscriber is held weakly, so the GC can collect it even while subscribed. The downside: the handler is silently dropped when the subscriber is collected, which can be unexpected.

**Rule of thumb:** every `+=` on an event should have a corresponding `-=` planned from the moment you write it.
</Callout>
