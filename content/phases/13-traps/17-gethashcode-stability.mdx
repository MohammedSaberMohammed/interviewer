## The Trap

```csharp
// Mutable class used as dictionary key — THE TRAP
public class Tag
{
    public string Name { get; set; } // Mutable!

    public override int GetHashCode() => Name.GetHashCode();
    public override bool Equals(object? obj) => obj is Tag t && t.Name == Name;
}

var dict = new Dictionary<Tag, int>();
var tag = new Tag { Name = "csharp" };

dict[tag] = 42;
Console.WriteLine(dict.ContainsKey(tag)); // true — found in bucket for "csharp".GetHashCode()

// Mutate the key:
tag.Name = "dotnet"; // Hash code CHANGES!

Console.WriteLine(dict.ContainsKey(tag)); // false — bucket for "dotnet".GetHashCode() is empty!
Console.WriteLine(dict.Count);             // 1 — the entry is still there, just lost!

// The value 42 is now ORPHANED — it exists in the dictionary
// but is stored in a bucket that can never be found via tag
```

<Callout type="trap">
A `Dictionary<TKey,TValue>` stores each entry in a bucket determined by `GetHashCode()` at insertion time. If the key's hash code changes after insertion (because the key's state changes), the entry is stranded in the wrong bucket. All subsequent lookups land in the new bucket and find nothing — the value is permanently lost while the `Count` shows it's there.
</Callout>

## Why It Happens

```csharp
// How Dictionary works internally (simplified):
void Add(TKey key, TValue value)
{
    int bucket = key.GetHashCode() % _buckets.Length; // Calculate bucket at ADD time
    _entries[_freeIndex] = new Entry { Key = key, Value = value, Hash = hashCode };
    _buckets[bucket] = _freeIndex; // Store in that specific bucket
}

bool TryGetValue(TKey key, out TValue value)
{
    int bucket = key.GetHashCode() % _buckets.Length; // Calculate bucket at LOOKUP time
    // Look only in this bucket — if hash changed, we're looking in the wrong place!
    var entry = _entries[_buckets[bucket]];
    if (entry.Hash == hashCode && _comparer.Equals(entry.Key, key))
    {
        value = entry.Value;
        return true;
    }
    value = default;
    return false;
}

// If tag.Name changed from "csharp" → "dotnet":
// Add: stored in bucket for hash("csharp") = e.g., bucket 7
// Lookup: looks in bucket for hash("dotnet") = e.g., bucket 3
// Bucket 3 is empty → returns false → orphaned entry!
```

## The Fix

**Option 1: Use immutable keys (recommended)**

```csharp
// Immutable key — GetHashCode cannot change after construction
public sealed class Tag
{
    public string Name { get; } // init-only — immutable after construction

    public Tag(string name) { Name = name; }

    public override int GetHashCode() => Name.GetHashCode(StringComparison.Ordinal);
    public override bool Equals(object? obj) => obj is Tag t && t.Name == Name;
}

// Or use a record (immutable by default):
public record Tag(string Name);
// Record generates GetHashCode and Equals based on properties
```

**Option 2: Use a value-based key that is stable**

```csharp
// If the object must be mutable, use only stable fields for the key identity:
public class Tag
{
    public Guid Id { get; } = Guid.NewGuid(); // Immutable identity
    public string Name { get; set; }           // Mutable, but NOT part of the hash

    public override int GetHashCode() => Id.GetHashCode(); // Based on immutable Id only
    public override bool Equals(object? obj) => obj is Tag t && t.Id == Id;
}

// Name can change freely — GetHashCode is stable because it only uses Id
```

**Option 3: Never mutate keys after insertion**

```csharp
// If you must use a mutable object as a key:
// 1. Do not mutate it while it's in the dictionary
// 2. Remove, mutate, re-insert:
dict.Remove(tag);
tag.Name = "dotnet";
dict[tag] = 42; // Re-insert with new hash
```

## The GetHashCode Contract

```csharp
// The contract (from .NET documentation):
// 1. If a.Equals(b) then a.GetHashCode() == b.GetHashCode()  — REQUIRED
// 2. GetHashCode() must return the same value for the same object's state
//    across multiple calls — REQUIRED for correctness as a dictionary key
// 3. Two objects that are NOT equal MAY have the same hash code (collision) — ALLOWED
// 4. GetHashCode() should be fast — IMPORTANT for performance

// Common violation — inconsistency with Equals:
public class BadKey
{
    public string Name { get; set; }
    // GetHashCode uses Name, Equals uses Name:
    public override int GetHashCode() => Name?.GetHashCode() ?? 0;
    public override bool Equals(object? obj) => obj is BadKey k && k.Name == Name;
    // These are consistent, but mutating Name breaks both after insertion
}

// Best practice: use HashCode.Combine for composite keys
public override int GetHashCode()
    => HashCode.Combine(FirstName, LastName, DateOfBirth);
// ^ stable only if all three fields are immutable
```

<Callout type="senior">
**Strings as dictionary keys — why they are safe**

`string` is the canonical example of a safe dictionary key because it is **immutable**. `"hello".GetHashCode()` always returns the same value for the same content. You can never change the content of a string after creation — you can only create a new string.

This is one reason the .NET guidelines recommend: **make your value objects (especially those used as keys) immutable**. Not just for GetHashCode correctness, but for thread safety, predictability, and preventing this entire class of bugs.

`record` types in C# 9+ generate `GetHashCode` from all init-only properties by default, making them safe key types out of the box. For mutable reference types used as keys, the safest approach is to override `GetHashCode` using only identity fields (like a database ID or a GUID) that never change.
</Callout>
