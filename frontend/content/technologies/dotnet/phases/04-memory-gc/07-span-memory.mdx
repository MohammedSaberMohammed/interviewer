## The Problem Span Solves

Before `Span<T>`, working with a sub-section of an array required allocating a new array and copying:

```csharp
// Old approach: allocate a new array just to look at part of the data
byte[] data = ReceivePacket();
byte[] header = data[0..16];   // copies 16 bytes → new heap allocation
byte[] payload = data[16..];   // copies rest → another heap allocation

ProcessHeader(header);
ProcessPayload(payload);

// With Span<T>: zero allocations, zero copies
byte[] data = ReceivePacket();
ReadOnlySpan<byte> header  = data.AsSpan(0, 16);   // no copy, no allocation
ReadOnlySpan<byte> payload = data.AsSpan(16);       // no copy, no allocation

ProcessHeader(header);   // method takes ReadOnlySpan<byte>
ProcessPayload(payload);
```

---

## What Is Span\<T\>?

`Span<T>` is a **ref struct** — a value type that is constrained to live on the stack. It holds:
1. A reference (pointer) to the start of a contiguous memory block
2. A length

```csharp
// Span<T> can point to:

// 1. Array data (most common)
int[] array = new int[100];
Span<int> span = array.AsSpan();
Span<int> slice = array.AsSpan(10, 20); // elements 10-29

// 2. Stack-allocated memory (stackalloc)
Span<byte> stackBuffer = stackalloc byte[64]; // no heap allocation at all

// 3. Native memory (unsafe code)
unsafe
{
    byte* ptr = stackalloc byte[64];
    Span<byte> nativeSpan = new Span<byte>(ptr, 64);
}

// 4. String characters (ReadOnlySpan<char>)
string s = "Hello, World!";
ReadOnlySpan<char> greeting = s.AsSpan(0, 5); // "Hello" — no string allocation
```

## Span Limitations: The ref struct Rules

Because `Span<T>` is a `ref struct`, it has strict lifetime constraints enforced by the compiler:

```csharp
// CANNOT store Span in a class field:
public class Parser
{
    private Span<byte> _buffer; // COMPILE ERROR: ref struct cannot be a field in a class
}

// CANNOT box a Span:
object boxed = new Span<int>(array); // COMPILE ERROR

// CANNOT use Span across await points in async methods:
public async Task ParseAsync(byte[] data)
{
    Span<byte> span = data.AsSpan(); // OK so far
    await SomeOperationAsync();       // COMPILE ERROR: span cannot survive await
    ProcessSpan(span);                // Span might not be valid after await
}

// CANNOT store in a lambda that outlives the Span:
Span<byte> localSpan = stackalloc byte[64];
Func<byte> first = () => localSpan[0]; // COMPILE ERROR: captures Span in closure
```

These are compile-time errors, not runtime errors — the type system prevents use-after-free bugs.

## Memory\<T\>: The Heap-Safe Alternative

`Memory<T>` solves Span's limitations by being a **regular struct** (not ref struct). It can be stored in fields, used across awaits, and captured in lambdas:

```csharp
// Memory<T> CAN cross async boundaries:
public async Task ParseAsync(byte[] data)
{
    Memory<byte> memory = data.AsMemory(); // OK — regular struct
    await SomeOperationAsync();             // fine — Memory<T> survives await
    ProcessMemory(memory);                  // works
}

// Memory<T> CAN be a field:
public class AsyncParser
{
    private Memory<byte> _buffer; // totally fine

    public AsyncParser(byte[] buffer)
    {
        _buffer = buffer.AsMemory();
    }
}

// Convert Memory<T> to Span<T> when you need to actually read/write:
public void ProcessMemory(Memory<byte> mem)
{
    Span<byte> span = mem.Span; // get a Span for the actual CPU operations
    for (int i = 0; i < span.Length; i++)
        span[i] ^= 0xFF; // XOR in-place, no allocation
}
```

## ReadOnlySpan and ReadOnlyMemory

Both types have read-only variants for passing data without allowing modification:

```csharp
// ReadOnlySpan<char> is particularly powerful for string parsing:
public static bool TryParseVersion(ReadOnlySpan<char> input, out Version version)
{
    // Parse "1.2.3.4" without allocating any strings
    int dot1 = input.IndexOf('.');
    if (dot1 < 0) { version = default; return false; }

    ReadOnlySpan<char> major = input[..dot1];
    ReadOnlySpan<char> rest  = input[(dot1 + 1)..];

    int dot2 = rest.IndexOf('.');
    ReadOnlySpan<char> minor = rest[..dot2];
    rest = rest[(dot2 + 1)..];

    // int.TryParse accepts ReadOnlySpan<char> — no substring allocation!
    if (!int.TryParse(major, out int maj) ||
        !int.TryParse(minor, out int min))
    {
        version = default;
        return false;
    }

    version = new Version(maj, min, 0, 0);
    return true;
    // Zero string allocations for the entire parse!
}
```

## MemoryPool and IMemoryOwner

For scenarios where memory needs to outlive a single method but you want pooled backing storage:

```csharp
using System.Buffers;

// Rent a block from MemoryPool (similar to ArrayPool but for Memory<T>)
IMemoryOwner<byte> owner = MemoryPool<byte>.Shared.Rent(minBufferSize: 4096);
try
{
    Memory<byte> memory = owner.Memory; // use this across async calls
    int bytesRead = await stream.ReadAsync(memory);
    Memory<byte> validData = memory[..bytesRead]; // slice without copying

    await ProcessAsync(validData);
}
finally
{
    owner.Dispose(); // returns buffer to pool
}
```

<Callout type="senior">
**Senior Insight: The Span API Surface in .NET 8**

The .NET 8 BCL has been heavily optimized around `Span<T>` and `ReadOnlySpan<char>`. Many APIs that previously only accepted `string` now have `Span`/`Memory` overloads:

```csharp
// Allocation-free number parsing (.NET 7+)
int.TryParse(span, out int n);
double.TryParse(span, out double d);
Guid.TryParse(span, out Guid g);

// Allocation-free number formatting (.NET 8 — IUtf8SpanFormattable)
Span<byte> utf8Buffer = stackalloc byte[64];
int written;
((IUtf8SpanFormattable)42).TryFormat(utf8Buffer, out written, "", null);

// System.Text.Json with Span (no string intermediate):
JsonDocument.Parse(ReadOnlySpan<byte> utf8Json);

// String.Create: build strings without intermediate allocations
string result = string.Create(length, state, (span, s) =>
{
    // span is a Span<char> over the new string's buffer
    // write directly into the string's memory — zero extra allocation
    s.WriteTo(span);
});
```

The pattern to internalize: **receive as ReadOnlySpan, process as Span, store as Memory**. When you own data temporarily, use Span. When data needs to outlive the current method, use Memory. When callers just need to read, use ReadOnlySpan or ReadOnlyMemory.

In hot paths (JSON parsing, HTTP header processing, binary protocols), switching from string-based APIs to Span-based APIs routinely cuts allocation by 60-90%.
</Callout>
