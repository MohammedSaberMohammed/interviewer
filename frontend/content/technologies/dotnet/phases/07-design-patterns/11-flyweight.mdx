## The Problem: Memory Explosion from Fine-Grained Objects

A text editor renders millions of characters. If each character object stores its font, size, color, style, AND its character value, memory usage explodes. Yet most characters share the same font/size/color — it's only the character value and position that differ per character.

Flyweight splits state into:
- **Intrinsic state** — shared, immutable, stored in the flyweight
- **Extrinsic state** — unique per context, stored by the client or passed in

## Structure

```csharp
// Flyweight — stores only intrinsic (shared) state
public class CharacterFormat // lightweight, shareable
{
    public string FontName { get; }
    public int    FontSize { get; }
    public string Color    { get; }
    public bool   IsBold   { get; }

    public CharacterFormat(string fontName, int fontSize, string color, bool isBold)
    {
        FontName = fontName;
        FontSize = fontSize;
        Color    = color;
        IsBold   = isBold;
    }

    // Intrinsic state — same format object shared by thousands of characters
}

// Flyweight Factory — the key to sharing
public class CharacterFormatFactory
{
    private readonly Dictionary<string, CharacterFormat> _pool = new();

    public CharacterFormat GetFormat(string fontName, int fontSize, string color, bool isBold)
    {
        string key = $"{fontName}|{fontSize}|{color}|{isBold}";
        if (!_pool.TryGetValue(key, out var format))
        {
            format = new CharacterFormat(fontName, fontSize, color, isBold);
            _pool[key] = format;
        }
        return format; // return shared instance
    }

    public int PoolSize => _pool.Count;
}

// Context — stores extrinsic state (char value, position) + reference to flyweight
public class CharacterContext
{
    public char             Character { get; } // extrinsic — unique per position
    public int              X         { get; } // extrinsic
    public int              Y         { get; } // extrinsic
    public CharacterFormat  Format    { get; } // flyweight — SHARED reference

    public CharacterContext(char ch, int x, int y, CharacterFormat format)
    {
        Character = ch; X = x; Y = y; Format = format;
    }

    public void Render() =>
        Console.WriteLine($"'{Character}' at ({X},{Y}) in {Format.FontName}/{Format.FontSize}pt/{Format.Color}");
}

// Usage — 10,000 characters, but only a few Format objects
var factory = new CharacterFormatFactory();
var document = new List<CharacterContext>();

string text = "Hello World! ".PadRight(10_000, 'x');
var bodyFormat    = factory.GetFormat("Arial", 12, "Black", false);
var headingFormat = factory.GetFormat("Arial", 24, "Black", true);

for (int i = 0; i < text.Length; i++)
{
    var format = i < 13 ? headingFormat : bodyFormat; // shared format
    document.Add(new CharacterContext(text[i], i % 80, i / 80, format));
}

Console.WriteLine($"Characters: {document.Count}");
Console.WriteLine($"Format objects: {factory.PoolSize}"); // 2, not 10,000!
```

## Real .NET Example: String Interning

String interning IS the Flyweight pattern for strings. The CLR maintains an intern pool where string literals with the same value share a single reference:

```csharp
// String literals are interned automatically
string a = "hello";
string b = "hello";
Console.WriteLine(object.ReferenceEquals(a, b)); // True — same object!

// Runtime strings are NOT interned by default
string c = new string(new[] { 'h', 'e', 'l', 'l', 'o' });
Console.WriteLine(object.ReferenceEquals(a, c)); // False — different objects

// Explicit interning
string d = string.Intern(c);
Console.WriteLine(object.ReferenceEquals(a, d)); // True — now same object

// Use case: high-cardinality strings that repeat (e.g., country codes, currency codes)
string[] countryCodes = { "US", "UK", "DE", "FR" };
// If reading 1M records, intern to avoid 1M separate string allocations for repeated codes
string interned = string.Intern(rawCode);
```

## Real .NET Example: `ArrayPool<T>`

`ArrayPool<T>` is a Flyweight/Object Pool for arrays — avoiding repeated allocations:

```csharp
using System.Buffers;

// Rent a buffer — get a shared array (may be larger than requested)
byte[] buffer = ArrayPool<byte>.Shared.Rent(4096);

try
{
    // Use the buffer — it's pre-allocated, no GC pressure
    int bytesRead = await stream.ReadAsync(buffer.AsMemory(0, 4096));
    Process(buffer.AsSpan(0, bytesRead));
}
finally
{
    // Return to pool — NOT the GC, the pool reuses it
    ArrayPool<byte>.Shared.Return(buffer, clearArray: false);
}

// In hot paths (serialization, network I/O), this eliminates millions of allocations
```

<Callout type="senior">
**Flyweight in Modern .NET — `frozen collections` and `MemoryMarshal`**

.NET 8 introduced `FrozenDictionary<TKey, TValue>` and `FrozenSet<T>` — optimized read-only collections that share their data structure like flyweights:

```csharp
using System.Collections.Frozen;

// FrozenDictionary — created once, read many times (no locking needed)
// Internally shares optimized hash tables; faster lookup than regular Dictionary
FrozenDictionary<string, int> countryCodes = new Dictionary<string, int>
{
    ["US"] = 1, ["UK"] = 44, ["DE"] = 49, ["FR"] = 33
}.ToFrozenDictionary();

// Safe to share across threads — no lock needed
int dialCode = countryCodes["US"]; // thread-safe, allocation-free

// RecyclableMemoryStream — Flyweight for MemoryStream buffers
// Uses ArrayPool to reuse buffers, avoiding LOH allocations
using var ms = new RecyclableMemoryStream(manager);
await data.SerializeToAsync(ms);
```

**Interview insight**: When asked about Flyweight, go beyond the GoF definition. Connect it to:
1. String interning (`string.Intern`)
2. `ArrayPool<T>` / `MemoryPool<T>`
3. `RecyclableMemoryStream` (Microsoft.IO.RecyclableMemoryStream)
4. Font/glyph caches in UI frameworks
5. EF Core's compiled query cache (queries are "flyweights" shared across executions)
</Callout>

  options={[
