## The Problem: Expensive or Complex Object Creation

Sometimes creating an object from scratch is expensive (loading configuration from disk, querying a database, running complex initialization logic). Other times you need a copy of an existing object with minor tweaks — like a template document with different values.

Prototype solves this by cloning an existing instance as the basis for new objects.

---

## Shallow Copy vs Deep Copy

This distinction is critical for correctness:

```csharp
public class Address
{
    public string Street { get; set; } = "";
    public string City   { get; set; } = "";
}

public class Person
{
    public string  Name    { get; set; } = "";
    public int     Age     { get; set; }
    public Address Address { get; set; } = new();
}

// SHALLOW COPY — MemberwiseClone copies field values
// Reference fields point to the SAME objects
Person original = new Person { Name = "Alice", Age = 30, Address = new Address { Street = "123 Main St", City = "NY" } };
Person shallow  = (Person)original.MemberwiseClone();

shallow.Name           = "Bob";   // OK — Name is a string (immutable in .NET)
shallow.Address.City   = "LA";    // PROBLEM! Modifies original.Address.City too!

Console.WriteLine(original.Address.City); // "LA" — original was mutated!

// DEEP COPY — manually copy all nested objects
public class Person
{
    public string  Name    { get; set; } = "";
    public int     Age     { get; set; }
    public Address Address { get; set; } = new();

    public Person DeepClone() => new Person
    {
        Name    = this.Name,
        Age     = this.Age,
        Address = new Address { Street = this.Address.Street, City = this.Address.City }
    };
}

Person deep = original.DeepClone();
deep.Address.City = "LA"; // safe — original.Address.City unchanged
```

## Implementing the Prototype Pattern

```csharp
// The prototype interface
public interface IPrototype<T>
{
    T Clone();
}

// Concrete prototype — a configurable email template
public class EmailTemplate : IPrototype<EmailTemplate>
{
    public string Subject { get; set; } = "";
    public string Body    { get; set; } = "";
    public List<string> Recipients { get; set; } = new();

    // Deep clone implementation
    public EmailTemplate Clone() => new EmailTemplate
    {
        Subject    = this.Subject,
        Body       = this.Body,
        Recipients = new List<string>(this.Recipients) // copy the list
    };
}

// Prototype registry — stores named prototypes
public class TemplateRegistry
{
    private readonly Dictionary<string, EmailTemplate> _templates = new();

    public void Register(string name, EmailTemplate template)
        => _templates[name] = template;

    public EmailTemplate GetClone(string name)
    {
        if (!_templates.TryGetValue(name, out var template))
            throw new KeyNotFoundException($"Template '{name}' not found.");
        return template.Clone(); // return a clone, not the original
    }
}

// Usage
var registry = new TemplateRegistry();
registry.Register("welcome", new EmailTemplate
{
    Subject    = "Welcome to {AppName}!",
    Body       = "Hello {Name}, welcome aboard.",
    Recipients = new List<string> { "notifications@company.com" }
});

// Get a pre-configured clone and customize it
EmailTemplate welcome = registry.GetClone("welcome");
welcome.Subject = welcome.Subject.Replace("{AppName}", "MyApp");
welcome.Body    = welcome.Body.Replace("{Name}", "Alice");
welcome.Recipients.Add("alice@example.com");
```

## The `ICloneable` Interface — Why Microsoft Recommends Against It

```csharp
// ICloneable is in the BCL but considered problematic
public interface ICloneable
{
    object Clone(); // returns object — must cast, no generic version
}

// The problem: Clone() doesn't specify shallow or deep — ambiguous contract
public class Document : ICloneable
{
    public string Title   { get; set; } = "";
    public List<Page> Pages { get; set; } = new();

    // Is this shallow or deep? The caller can't know without reading the source.
    public object Clone() => this.MemberwiseClone(); // shallow!
}

// Preferred: use your own typed interface
public interface IDeepCloneable<T>
{
    T DeepClone();
}
```

`ICloneable` has no specification on whether `Clone()` is shallow or deep — the .NET team officially recommends not implementing it in new code.

## Modern Alternative: `record` with `with`

C# 9 records have built-in non-destructive mutation via `with` — a form of shallow prototype:

```csharp
public record Address(string Street, string City);

public record Person(string Name, int Age, Address Address);

// 'with' creates a shallow clone with specified properties changed
Person alice = new Person("Alice", 30, new Address("123 Main", "NY"));
Person bob   = alice with { Name = "Bob", Age = 25 }; // clone + override

Console.WriteLine(alice.Name); // "Alice" — original unchanged
Console.WriteLine(bob.Name);   // "Bob"

// Warning: Address is still shared (shallow copy)
Person carol = alice with { Name = "Carol" };
// carol.Address is the same Address reference as alice.Address
// but since Address is a record (immutable), this is safe
```

For immutable record types, shallow `with` is safe because neither the original nor the clone can mutate the shared reference.

<Callout type="senior">
**Serialization-Based Deep Clone**

For complex object graphs, serialization/deserialization gives you an automatic deep clone without writing clone methods for every class:

```csharp
using System.Text.Json;

public static class DeepCloneExtensions
{
    public static T DeepClone<T>(this T obj)
    {
        // Serialize to JSON, then deserialize to a new instance
        string json = JsonSerializer.Serialize(obj);
        return JsonSerializer.Deserialize<T>(json)!;
    }
}

// Usage
var original = new ShoppingCart { Items = { new CartItem { Name = "Book", Price = 29.99m } } };
var copy = original.DeepClone();
copy.Items.Add(new CartItem { Name = "Pen", Price = 1.99m });
// original.Items.Count is still 1

// Tradeoffs of serialization cloning:
// + Simple — no per-class Clone() methods
// + Handles arbitrary depth
// - Slower than manual (allocates strings, uses reflection)
// - Loses non-serializable state (events, delegates, private fields not in JSON)
// - Requires all types to be serializable
```

For hot paths, benchmark before using serialization cloning. For configuration objects cloned at startup, it's usually fine.
</Callout>
