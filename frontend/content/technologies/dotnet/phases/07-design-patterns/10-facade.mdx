## The Problem: Subsystem Complexity

Sending an email involves: finding a mail server, opening an SMTP connection, authenticating, constructing a MIME message, attaching files, encoding content, handling errors, and closing the connection. Exposing all of this to every caller is overwhelming.

A Facade wraps the subsystem and exposes just what most callers need.

---

## Structure

```csharp
// Complex subsystem classes â€” each handles one piece
public class SmtpConnection
{
    public void Connect(string host, int port) { /* ... */ }
    public void Authenticate(string user, string pass) { /* ... */ }
    public void Disconnect() { /* ... */ }
}

public class MimeBuilder
{
    public string BuildMessage(string from, string to, string subject, string body) => "MIME...";
    public string AddAttachment(string mimeMsg, byte[] data, string filename) => "MIME+att...";
}

public class SmtpSender
{
    public void Send(SmtpConnection conn, string mimeMessage) { /* ... */ }
}

public class EmailLogger
{
    public void LogSent(string to, string subject) { /* ... */ }
}

// FACADE â€” simplified interface for the common case
public class EmailFacade
{
    private readonly SmtpConnection _connection;
    private readonly MimeBuilder    _mimeBuilder;
    private readonly SmtpSender     _sender;
    private readonly EmailLogger    _logger;
    private readonly string         _smtpHost;
    private readonly string         _smtpUser;
    private readonly string         _smtpPass;

    public EmailFacade(EmailSettings settings)
    {
        _connection  = new SmtpConnection();
        _mimeBuilder = new MimeBuilder();
        _sender      = new SmtpSender();
        _logger      = new EmailLogger();
        _smtpHost    = settings.Host;
        _smtpUser    = settings.Username;
        _smtpPass    = settings.Password;
    }

    // Simple method for the common case
    public void Send(string to, string subject, string body)
    {
        _connection.Connect(_smtpHost, 587);
        _connection.Authenticate(_smtpUser, _smtpPass);

        string mime = _mimeBuilder.BuildMessage(_smtpUser, to, subject, body);
        _sender.Send(_connection, mime);
        _logger.LogSent(to, subject);

        _connection.Disconnect();
    }

    // Slightly more complex overload for attachments
    public void SendWithAttachment(string to, string subject, string body, byte[] attachment, string filename)
    {
        _connection.Connect(_smtpHost, 587);
        _connection.Authenticate(_smtpUser, _smtpPass);

        string mime = _mimeBuilder.BuildMessage(_smtpUser, to, subject, body);
        mime = _mimeBuilder.AddAttachment(mime, attachment, filename);
        _sender.Send(_connection, mime);
        _logger.LogSent(to, $"{subject} (with attachment)");

        _connection.Disconnect();
    }
}

// Usage â€” simple!
var email = new EmailFacade(settings);
email.Send("alice@example.com", "Welcome!", "Hello Alice, welcome aboard.");
```

The facade doesn't prevent advanced callers from using the subsystem directly â€” it just makes the common case simple.

## Real .NET BCL Example: `System.IO.File`

`System.IO.File` is a facade over `FileStream`, `StreamReader`, `StreamWriter`, and `BinaryReader`:

```csharp
// Without File facade â€” verbose subsystem usage
using FileStream fs = new FileStream("data.txt", FileMode.Open, FileAccess.Read);
using StreamReader reader = new StreamReader(fs, Encoding.UTF8);
string content = reader.ReadToEnd();

// With File facade â€” one line for the common case
string content = File.ReadAllText("data.txt");

// More File facade methods
string[]  lines  = File.ReadAllLines("data.txt");
byte[]    bytes  = File.ReadAllBytes("image.png");
File.WriteAllText("output.txt", "Hello World");
File.AppendAllText("log.txt", $"[{DateTime.Now}] Event occurred\n");

// Facade doesn't restrict â€” you can still use FileStream directly when needed
using var fs = new FileStream("data.bin", FileMode.Open, FileAccess.Read, FileShare.Read,
    bufferSize: 4096, useAsync: true);
```

## Facade in ASP.NET Core

`HttpContext` is a facade over the ASP.NET Core request/response subsystem:

```csharp
// Without facade â€” access raw subsystem types
IHttpContextAccessor accessor = ... ;
HttpContext ctx = accessor.HttpContext!;

// Through HttpContext facade â€” simplified access
string? userId = ctx.User.FindFirstValue(ClaimTypes.NameIdentifier);
string? ipAddress = ctx.Connection.RemoteIpAddress?.ToString();
string host = ctx.Request.Host.Host;
await ctx.Response.WriteAsJsonAsync(new { message = "OK" });

// The facade hides: HttpRequest, HttpResponse, ClaimsPrincipal, ConnectionInfo,
//                   IFeatureCollection, IServiceProvider...
```

<Callout type="senior">
**Facade and the Layer Anti-Pattern**

Facades can become anti-patterns when they morph into "God objects" â€” a single class that knows about everything. Watch for these warning signs:

```csharp
// Facade that's become a God Object â€” avoid this
public class ApplicationFacade
{
    // Too many dependencies â€” this "facade" has become a service locator
    public ApplicationFacade(
        IUserService users, IOrderService orders, IInventoryService inventory,
        IEmailService email, IPaymentService payment, IReportService reports,
        ICacheService cache, ISearchService search /* 10 more... */)
    { }

    // 50+ methods spanning unrelated concerns
    public void PlaceOrder(...) { }
    public void GenerateReport(...) { }
    public void SendNewsletter(...) { }
    // ...
}
```

A Facade should simplify access to **one coherent subsystem**, not wrap your entire application. If the facade has more than 5-7 dependencies or spans unrelated domains, split it into domain-specific facades.

**Facade in API design**: A public REST API is a Facade over your internal services. The endpoint `POST /api/orders` hides the order creation, inventory check, payment processing, and email notification complexity behind a single HTTP call â€” exactly the Facade intent at the service boundary level.
</Callout>
