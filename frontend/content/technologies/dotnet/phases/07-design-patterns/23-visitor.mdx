## The Problem: Adding Operations to a Fixed Hierarchy

You have a stable class hierarchy (geometric shapes, AST nodes, DOM elements). You need to add operations periodically (rendering, serialization, area calculation, SQL translation). Without Visitor, each operation scatters `if (obj is Circle) ... else if (obj is Square) ...` checks everywhere — violating OCP.

Visitor lets you add a new operation by writing one new Visitor class — no changes to the shape hierarchy.

## Classic Visitor (Double Dispatch)

```csharp
// The element hierarchy — rarely changes
public interface IShape
{
    void Accept(IShapeVisitor visitor); // the key: element accepts a visitor
}

public class Circle : IShape
{
    public double Radius { get; }
    public Circle(double radius) => Radius = radius;

    // Double dispatch: calls the visitor's method specific to Circle
    public void Accept(IShapeVisitor visitor) => visitor.VisitCircle(this);
}

public class Rectangle : IShape
{
    public double Width { get; }
    public double Height { get; }
    public Rectangle(double w, double h) { Width = w; Height = h; }

    public void Accept(IShapeVisitor visitor) => visitor.VisitRectangle(this);
}

public class Triangle : IShape
{
    public double Base { get; }
    public double Height { get; }
    public Triangle(double b, double h) { Base = b; Height = h; }

    public void Accept(IShapeVisitor visitor) => visitor.VisitTriangle(this);
}

// Visitor interface — one method per element type
public interface IShapeVisitor
{
    void VisitCircle(Circle circle);
    void VisitRectangle(Rectangle rect);
    void VisitTriangle(Triangle tri);
}

// Concrete visitors — each adds a different operation to ALL shapes
public class AreaCalculator : IShapeVisitor
{
    public double TotalArea { get; private set; }

    public void VisitCircle(Circle c)    => TotalArea += Math.PI * c.Radius * c.Radius;
    public void VisitRectangle(Rectangle r) => TotalArea += r.Width * r.Height;
    public void VisitTriangle(Triangle t)   => TotalArea += 0.5 * t.Base * t.Height;
}

public class SvgRenderer : IShapeVisitor
{
    private readonly StringBuilder _svg = new("<svg>\n");

    public void VisitCircle(Circle c)    => _svg.AppendLine($"  <circle r='{c.Radius}' />");
    public void VisitRectangle(Rectangle r) => _svg.AppendLine($"  <rect width='{r.Width}' height='{r.Height}' />");
    public void VisitTriangle(Triangle t)   => _svg.AppendLine($"  <!-- triangle base={t.Base} -->");

    public string GetSvg() => _svg.Append("</svg>").ToString();
}

// Usage — each shape calls the correct visitor method automatically
var shapes = new IShape[] { new Circle(5), new Rectangle(3, 4), new Triangle(6, 8) };

var areaCalc = new AreaCalculator();
foreach (var shape in shapes)
    shape.Accept(areaCalc);
Console.WriteLine($"Total area: {areaCalc.TotalArea:F2}"); // 78.54 + 12 + 24 = 114.54

var renderer = new SvgRenderer();
foreach (var shape in shapes)
    shape.Accept(renderer);
Console.WriteLine(renderer.GetSvg());
```

## Why "Double Dispatch"?

C# is single-dispatch: method resolution is based on one object's type (the receiver's). Visitor achieves double dispatch via two virtual calls:

```
// First dispatch: shape.Accept(visitor) — resolves based on shape's runtime type
// → Circle.Accept is called

// Inside Circle.Accept:
// Second dispatch: visitor.VisitCircle(this) — resolves based on visitor's runtime type
// → AreaCalculator.VisitCircle is called

// Result: behavior determined by BOTH shape type AND visitor type
```

## Real .NET Example: `ExpressionVisitor`

`System.Linq.Expressions.ExpressionVisitor` is the BCL's Visitor for expression trees:

```csharp
// Built-in Visitor base class — visit every node type
public class ParameterReplacer : ExpressionVisitor
{
    private readonly ParameterExpression _oldParam;
    private readonly ParameterExpression _newParam;

    public ParameterReplacer(ParameterExpression oldParam, ParameterExpression newParam)
    {
        _oldParam = oldParam;
        _newParam = newParam;
    }

    // Override only the node types you care about
    protected override Expression VisitParameter(ParameterExpression node)
        => node == _oldParam ? _newParam : base.VisitParameter(node);
}

// Use case: combine two predicates with different parameter names
Expression<Func<Order, bool>> byStatus = o => o.Status == "Active";
Expression<Func<Order, bool>> byTotal  = p => p.Total > 100;  // different param name!

// Replace 'p' in byTotal with 'o' from byStatus
var replacer = new ParameterReplacer(byTotal.Parameters[0], byStatus.Parameters[0]);
Expression newBody = replacer.Visit(byTotal.Body);

// Combine into one expression: o => o.Status == "Active" && o.Total > 100
var combined = Expression.Lambda<Func<Order, bool>>(
    Expression.AndAlso(byStatus.Body, newBody),
    byStatus.Parameters[0]);

// Now EF Core can translate the combined expression to SQL
var results = dbContext.Orders.Where(combined).ToList();
```

## Modern C# Alternative: Pattern Matching

C# switch expressions with pattern matching can replace Visitor for many cases:

```csharp
// Without Visitor — pattern matching approach
static double CalculateArea(IShape shape) => shape switch
{
    Circle c    => Math.PI * c.Radius * c.Radius,
    Rectangle r => r.Width * r.Height,
    Triangle t  => 0.5 * t.Base * t.Height,
    _           => throw new ArgumentException($"Unknown shape: {shape.GetType().Name}")
};

// Same result — much simpler code
double area = CalculateArea(new Circle(5));

// Adding a new OPERATION is easy — just a new static method with switch
static string Describe(IShape shape) => shape switch
{
    Circle c    => $"Circle with radius {c.Radius}",
    Rectangle r => $"{r.Width}×{r.Height} rectangle",
    Triangle t  => $"Triangle with base {t.Base}",
    _           => "Unknown shape"
};
```

<Callout type="senior">
**Visitor in EF Core Query Translation**

EF Core's query translator IS a Visitor over the LINQ expression tree. It visits each node and translates it to SQL:

```csharp
// EF Core internally uses multiple Visitors to translate LINQ to SQL
// 1. QueryCompilationContext — builds the expression pipeline
// 2. RelationalQueryableMethodTranslatingExpressionVisitor — translates LINQ methods to SQL
// 3. SqlNullabilityProcessor — adds NULL handling
// 4. SearchConditionConvertingExpressionVisitor — converts conditions

// You can hook into this with custom translators:
public class MyCustomDbFunctionTranslator : IMethodCallTranslator
{
    private readonly ISqlExpressionFactory _sqlFactory;

    public MyCustomDbFunctionTranslator(ISqlExpressionFactory sqlFactory)
        => _sqlFactory = sqlFactory;

    public SqlExpression? Translate(
        SqlExpression? instance,
        MethodInfo method,
        IReadOnlyList<SqlExpression> arguments,
        IDiagnosticsLogger<DbLoggerCategory.Query> logger)
    {
        // Translate a C# method call to a SQL function
        if (method.DeclaringType == typeof(StringExtensions) && method.Name == "Soundex")
            return _sqlFactory.Function("SOUNDEX", arguments, typeof(string));

        return null;
    }
}
```

Understanding that EF Core is a Visitor over expression trees explains: why `Expression<Func<T, bool>>` is needed (Visitor can't traverse compiled IL), why some LINQ operations can't be translated (the Visitor has no rule for them), and why custom DB function mappings work (you register a new visit rule).
</Callout>
