## The Problem: Inconsistent Families

Imagine a UI toolkit that supports Windows and macOS themes. You need a `Button`, `Checkbox`, and `TextBox` — each themed consistently. If you create them independently, you risk mixing a Windows button with a macOS checkbox.

Abstract Factory solves this by grouping related object creation behind a single interface — you get a factory that creates a consistent family.

---

## Structure

```csharp
// Abstract products
public interface IButton  { void Render(); }
public interface ICheckbox { void Render(); }

// Concrete products — Windows family
public class WindowsButton  : IButton  { public void Render() => Console.WriteLine("Windows Button"); }
public class WindowsCheckbox : ICheckbox { public void Render() => Console.WriteLine("Windows Checkbox"); }

// Concrete products — macOS family
public class MacOSButton   : IButton  { public void Render() => Console.WriteLine("macOS Button"); }
public class MacOSCheckbox : ICheckbox { public void Render() => Console.WriteLine("macOS Checkbox"); }

// Abstract factory — declares creation methods for each product
public interface IUIFactory
{
    IButton   CreateButton();
    ICheckbox CreateCheckbox();
}

// Concrete factories — one per product family
public class WindowsUIFactory : IUIFactory
{
    public IButton   CreateButton()   => new WindowsButton();
    public ICheckbox CreateCheckbox() => new WindowsCheckbox();
}

public class MacOSUIFactory : IUIFactory
{
    public IButton   CreateButton()   => new MacOSButton();
    public ICheckbox CreateCheckbox() => new MacOSCheckbox();
}

// Client — uses only the abstract factory interface
public class Application
{
    private readonly IButton   _button;
    private readonly ICheckbox _checkbox;

    public Application(IUIFactory factory)
    {
        _button   = factory.CreateButton();
        _checkbox = factory.CreateCheckbox();
    }

    public void RenderUI()
    {
        _button.Render();
        _checkbox.Render();
    }
}

// Usage — swap the entire family by changing the factory
IUIFactory factory = Environment.OSVersion.Platform == PlatformID.Win32NT
    ? new WindowsUIFactory()
    : new MacOSUIFactory();

var app = new Application(factory);
app.RenderUI(); // renders consistent family
```

## Real .NET Example: `ILoggerFactory`

`Microsoft.Extensions.Logging.ILoggerFactory` is an Abstract Factory:

```csharp
// ILoggerFactory is the abstract factory
public interface ILoggerFactory : IDisposable
{
    ILogger CreateLogger(string categoryName); // factory method for loggers
    void AddProvider(ILoggerProvider provider);
}

// Usage — the factory creates consistent loggers
ILoggerFactory loggerFactory = LoggerFactory.Create(builder =>
{
    builder
        .AddConsole()           // ConsoleLoggerProvider family
        .AddDebug()             // DebugLoggerProvider family
        .SetMinimumLevel(LogLevel.Information);
});

// CreateLogger returns a logger wired to all registered providers
ILogger<OrderService> logger = loggerFactory.CreateLogger<OrderService>();
logger.LogInformation("Order {OrderId} created", orderId);

// In DI — factory is registered and injected
builder.Services.AddLogging(b => b.AddConsole());

public class OrderService
{
    private readonly ILogger<OrderService> _logger;

    // ILogger<T> is created by ILoggerFactory behind the scenes
    public OrderService(ILogger<OrderService> logger) => _logger = logger;
}
```

`ILoggerFactory` creates `ILogger` instances configured consistently — you don't mix providers accidentally.

## Database Access Example

```csharp
// Abstract factory for database access
public interface IDbFactory
{
    IDbConnection  CreateConnection(string connectionString);
    IDbTransaction CreateTransaction(IDbConnection connection);
    IDbCommand     CreateCommand(string sql, IDbConnection connection);
}

// SQL Server implementation
public class SqlServerDbFactory : IDbFactory
{
    public IDbConnection  CreateConnection(string cs)   => new SqlConnection(cs);
    public IDbTransaction CreateTransaction(IDbConnection conn)
        => conn.BeginTransaction();
    public IDbCommand     CreateCommand(string sql, IDbConnection conn)
        => new SqlCommand(sql, (SqlConnection)conn);
}

// SQLite implementation (for testing)
public class SqliteDbFactory : IDbFactory
{
    public IDbConnection  CreateConnection(string cs)   => new SqliteConnection(cs);
    public IDbTransaction CreateTransaction(IDbConnection conn)
        => conn.BeginTransaction();
    public IDbCommand     CreateCommand(string sql, IDbConnection conn)
        => new SqliteCommand(sql, (SqliteConnection)conn);
}

// Register via DI
builder.Services.AddSingleton<IDbFactory, SqlServerDbFactory>();
// In tests:
// services.AddSingleton<IDbFactory, SqliteDbFactory>();
```

<Callout type="senior">
**Abstract Factory and the DI Container**

In modern .NET, the DI container often replaces hand-rolled Abstract Factories. Instead of writing a factory interface, you register different concrete families and select them via configuration:

```csharp
// Register the "production" family
if (builder.Environment.IsProduction())
{
    builder.Services.AddSingleton<IDbFactory, SqlServerDbFactory>();
    builder.Services.AddSingleton<ICacheFactory, RedisFactory>();
    builder.Services.AddSingleton<IMessagingFactory, ServiceBusFactory>();
}
else // test/dev family
{
    builder.Services.AddSingleton<IDbFactory, SqliteDbFactory>();
    builder.Services.AddSingleton<ICacheFactory, InMemoryCacheFactory>();
    builder.Services.AddSingleton<IMessagingFactory, InMemoryMessagingFactory>();
}
```

The container acts as the abstract factory — resolving a consistent family of services based on registration.

**Warning**: Avoid making your abstract factory "too abstract". If products are rarely created together or don't need to be compatible, separate Factory Methods are simpler. Abstract Factory shines when you have 3+ products that form a coherent family.
</Callout>
