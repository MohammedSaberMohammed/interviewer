## The Distributed Transaction Problem

In a microservices system, you often need to coordinate changes across multiple services atomically. A classic example: placing an order requires:

1. Deducting payment (Payment Service)
2. Reserving inventory (Inventory Service)
3. Scheduling shipment (Shipping Service)
4. Sending confirmation email (Notification Service)

With a monolith and a single database, this is a database transaction. With four separate services and four separate databases, **2PC (two-phase commit) is off the table** — it requires distributed coordination that creates tight coupling, long-held locks, and availability issues.

The **Saga pattern** is the microservices solution: a sequence of local transactions, each publishing an event or message that triggers the next step, with **compensating transactions** to roll back on failure.

## What Is a Compensating Transaction?

Unlike database rollback (which is automatic and instant), a saga compensating transaction is an explicit business operation that **semantically undoes** a completed step.

```
Step:              Reserve inventory (quantity -= 5)
Compensation:      Release inventory (quantity += 5)

Step:              Charge payment ($100)
Compensation:      Refund payment ($100)

Step:              Send welcome email
Compensation:      ??? — you cannot un-send an email
                   (Send a cancellation email instead — best-effort compensation)
```

Important: compensations must be **idempotent** (safe to run multiple times) and **retried** until they succeed. A failed compensation is worse than a failed saga step.

## Choreography-Based Saga

Each service **listens for events and reacts independently** — there is no central coordinator. Services publish events after completing their local transaction; other services subscribe and act.

```
OrderPlaced ──► InventoryService (reserves stock) ──► StockReserved
                                                           │
                                                           ▼
                                                   PaymentService (charges) ──► PaymentCharged
                                                                                       │
                                                                                       ▼
                                                                               ShippingService (schedules)
```

```csharp
// Choreography with MassTransit

// Step 1: Order Service — starts the saga
public class OrderService
{
    public async Task PlaceOrderAsync(PlaceOrderCommand cmd)
    {
        var order = new Order { Id = Guid.NewGuid(), Status = OrderStatus.Pending };
        await _orderRepo.SaveAsync(order);

        // Publish event — Inventory service will react
        await _bus.Publish(new OrderPlacedEvent(order.Id, cmd.Items, cmd.CustomerId));
    }
}

// Step 2: Inventory Service — reacts to OrderPlaced
public class InventoryConsumer : IConsumer<OrderPlacedEvent>
{
    public async Task Consume(ConsumeContext<OrderPlacedEvent> context)
    {
        var evt = context.Message;
        bool reserved = await _inventoryService.TryReserveAsync(evt.Items);

        if (reserved)
            await context.Publish(new StockReservedEvent(evt.OrderId));
        else
            await context.Publish(new StockReservationFailedEvent(evt.OrderId, "Insufficient stock"));
    }
}

// Step 3: Payment Service — reacts to StockReserved
public class PaymentConsumer : IConsumer<StockReservedEvent>
{
    public async Task Consume(ConsumeContext<StockReservedEvent> context)
    {
        bool charged = await _paymentService.ChargeAsync(context.Message.OrderId);

        if (charged)
            await context.Publish(new PaymentChargedEvent(context.Message.OrderId));
        else
        {
            // Publish failure — Inventory must compensate
            await context.Publish(new PaymentFailedEvent(context.Message.OrderId));
        }
    }
}

// Compensation: Inventory listens for PaymentFailed and releases stock
public class StockReleaseConsumer : IConsumer<PaymentFailedEvent>
{
    public async Task Consume(ConsumeContext<PaymentFailedEvent> context)
    {
        await _inventoryService.ReleaseReservationAsync(context.Message.OrderId);
        await context.Publish(new StockReleasedEvent(context.Message.OrderId));
    }
}
```

**Choreography trade-offs**:
- Decentralized — no single point of failure
- Each service is independent and testable
- Hard to visualize the overall flow — it lives across many services
- Cyclic event chains are easy to create accidentally
- Difficult to debug when something goes wrong mid-saga

## Orchestration-Based Saga

A **central orchestrator** (saga state machine) directs all participants, knows the full state of the saga, and explicitly triggers compensation steps.

```
                    ┌─────────────────────────┐
                    │     OrderSaga           │
                    │  (State Machine)        │
                    │                         │
  PlaceOrder ──────►│ 1. Reserve Inventory   ─┼──► InventoryService
                    │ 2. Charge Payment      ─┼──► PaymentService
                    │ 3. Schedule Shipping   ─┼──► ShippingService
                    │                         │
                    │ On failure: compensate  │
                    └─────────────────────────┘
```

```csharp
// Orchestration with MassTransit Saga State Machine

public class OrderSagaState : SagaStateMachineInstance
{
    public Guid CorrelationId { get; set; }
    public string CurrentState { get; set; } = null!;
    public Guid OrderId { get; set; }
    public bool StockReserved { get; set; }
    public bool PaymentCharged { get; set; }
}

public class OrderSaga : MassTransitStateMachine<OrderSagaState>
{
    public State ReservingStock { get; private set; } = null!;
    public State ChargingPayment { get; private set; } = null!;
    public State Compensating { get; private set; } = null!;
    public State Completed { get; private set; } = null!;
    public State Failed { get; private set; } = null!;

    public Event<OrderPlacedEvent> OrderPlaced { get; private set; } = null!;
    public Event<StockReservedEvent> StockReserved { get; private set; } = null!;
    public Event<StockReservationFailedEvent> StockFailed { get; private set; } = null!;
    public Event<PaymentChargedEvent> PaymentCharged { get; private set; } = null!;
    public Event<PaymentFailedEvent> PaymentFailed { get; private set; } = null!;

    public OrderSaga()
    {
        InstanceState(x => x.CurrentState);
        Event(() => OrderPlaced, x => x.CorrelateById(m => m.Message.OrderId));
        Event(() => StockReserved, x => x.CorrelateById(m => m.Message.OrderId));
        Event(() => StockFailed, x => x.CorrelateById(m => m.Message.OrderId));
        Event(() => PaymentCharged, x => x.CorrelateById(m => m.Message.OrderId));
        Event(() => PaymentFailed, x => x.CorrelateById(m => m.Message.OrderId));

        Initially(
            When(OrderPlaced)
                .Then(ctx => ctx.Saga.OrderId = ctx.Message.OrderId)
                .TransitionTo(ReservingStock)
                .Send(new Uri("queue:inventory"), ctx => new ReserveStockCommand(ctx.Saga.OrderId))
        );

        During(ReservingStock,
            When(StockReserved)
                .Then(ctx => ctx.Saga.StockReserved = true)
                .TransitionTo(ChargingPayment)
                .Send(new Uri("queue:payment"), ctx => new ChargePaymentCommand(ctx.Saga.OrderId)),
            When(StockFailed)
                .TransitionTo(Failed)
                .Finalize()
        );

        During(ChargingPayment,
            When(PaymentCharged)
                .Then(ctx => ctx.Saga.PaymentCharged = true)
                .TransitionTo(Completed)
                .Finalize(),
            When(PaymentFailed)
                .TransitionTo(Compensating)
                .Send(new Uri("queue:inventory"), ctx => new ReleaseStockCommand(ctx.Saga.OrderId))
        );
    }
}
```

<Callout type="senior">
**Senior Insight: Sagas Are Eventually Consistent — Always**

The fundamental guarantee a saga provides is **eventual consistency**, not atomicity. Between step N completing and step N+1 starting, the system is in an intermediate state. Other services and users may observe this partial state.

This means:

1. **Design for it explicitly**: Show "Order Pending" status while the saga runs. Don't show "Order Confirmed" until the saga completes.

2. **Idempotency is non-negotiable**: Every saga step and compensation must be idempotent. The message broker delivers at-least-once; your handlers will run multiple times.

3. **Saga recovery**: Store saga state durably (database, not in memory). If the orchestrator restarts mid-saga, it must be able to resume from the last known state.

4. **Time limits**: Sagas can get stuck waiting for a step that never responds. Always add timeouts with compensation on timeout.

```csharp
// MassTransit saga with timeout
During(ReservingStock,
    When(StockReserved).TransitionTo(ChargingPayment)...,
    When(StockFailed).TransitionTo(Failed)...,
    // If no response in 30 seconds, compensate
    When(ReservingStockTimeout)
        .TransitionTo(Failed)
        .Publish(ctx => new OrderCancelledEvent(ctx.Saga.OrderId, "Timeout"))
);
```
</Callout>
