## Redis in the .NET Ecosystem

Redis (Remote Dictionary Server) is an in-memory data structure store used as a cache, message broker, and database. In .NET, there are two primary ways to access it:

1. **`IDistributedCache`** — the ASP.NET Core abstraction (simple, swappable, limited)
2. **`StackExchange.Redis`** (`IConnectionMultiplexer`) — the full Redis client (powerful, exposes all Redis data types)

Understanding when to use each is a common interview differentiator.

## IDistributedCache: The Abstraction Layer

`IDistributedCache` provides a simple get/set/remove interface, good for basic caching that you might swap out (e.g., switch from Redis to SQL-based cache in tests).

```csharp
// Registration in Program.cs
builder.Services.AddStackExchangeRedisCache(options =>
{
    options.Configuration = builder.Configuration["Redis:ConnectionString"];
    options.InstanceName = "myapp:"; // key prefix for namespacing
});

// Usage — works with any IDistributedCache implementation
public class ProductCache
{
    private readonly IDistributedCache _cache;

    public async Task<Product?> GetAsync(string id)
    {
        var bytes = await _cache.GetAsync(\$"product:{id}");
        return bytes is null ? null : JsonSerializer.Deserialize<Product>(bytes);
    }

    public async Task SetAsync(Product product, TimeSpan ttl)
    {
        var bytes = JsonSerializer.SerializeToUtf8Bytes(product);
        await _cache.SetAsync(\$"product:{product.Id}", bytes, new DistributedCacheEntryOptions
        {
            AbsoluteExpirationRelativeToNow = ttl
        });
    }
}
```

**Limitation**: `IDistributedCache` only exposes byte arrays with TTL. It does not expose Redis sorted sets, lists, streams, pub/sub, or Lua scripting.

## StackExchange.Redis: Direct Access

For anything beyond simple key-value caching, use `IConnectionMultiplexer` directly.

```csharp
// Registration — singleton is correct for StackExchange.Redis
builder.Services.AddSingleton<IConnectionMultiplexer>(sp =>
    ConnectionMultiplexer.Connect(new ConfigurationOptions
    {
        EndPoints = { "redis.myapp.com:6379" },
        Password = "secret",
        Ssl = true,
        AbortOnConnectFail = false, // retry on startup
        ReconnectRetryPolicy = new ExponentialRetry(5000)
    })
);

// Get IDatabase in a service
public class LeaderboardService
{
    private readonly IDatabase _db;

    public LeaderboardService(IConnectionMultiplexer redis)
    {
        _db = redis.GetDatabase();
    }
}
```

<Callout type="trap">
**Trap: Creating a new ConnectionMultiplexer per request**

`ConnectionMultiplexer` is expensive to create and manages a pool of connections internally. Always register it as a **singleton**. Creating one per request will exhaust connections and cause timeouts within minutes under load.
</Callout>

## Pattern 1: Sorted Sets for Leaderboards

Redis sorted sets (`ZADD`, `ZRANGE`, `ZREVRANGE`) are perfect for real-time leaderboards and ranked data.

```csharp
public class LeaderboardService
{
    private readonly IDatabase _db;
    private const string LeaderboardKey = "leaderboard:weekly";

    public async Task RecordScoreAsync(string userId, double score)
    {
        // ZADD leaderboard:weekly score userId
        await _db.SortedSetAddAsync(LeaderboardKey, userId, score);
    }

    public async Task<IEnumerable<LeaderboardEntry>> GetTopNAsync(int n)
    {
        // ZREVRANGE leaderboard:weekly 0 (n-1) WITHSCORES
        var entries = await _db.SortedSetRangeByRankWithScoresAsync(
            LeaderboardKey,
            start: 0,
            stop: n - 1,
            order: Order.Descending
        );

        return entries.Select(e => new LeaderboardEntry(
            UserId: e.Element!,
            Score: e.Score
        ));
    }

    public async Task<long?> GetUserRankAsync(string userId)
    {
        // ZREVRANK — 0-indexed rank from top
        return await _db.SortedSetRankAsync(LeaderboardKey, userId, Order.Descending);
    }
}
```

## Pattern 2: Distributed Lock (Redlock)

A distributed lock ensures only one instance of a process runs at a time across multiple nodes.

```csharp
// Using RedLock.net (wrapper around Redlock algorithm)
// Install: Install-Package RedLock.net

public class JobScheduler
{
    private readonly IDistributedLockFactory _lockFactory;

    public async Task RunExclusiveJobAsync(CancellationToken ct)
    {
        var resource = "locks:nightly-report-job";
        var expiry = TimeSpan.FromMinutes(5);

        await using var redLock = await _lockFactory.CreateLockAsync(
            resource, expiry,
            wait: TimeSpan.FromSeconds(10),   // how long to try acquiring
            retry: TimeSpan.FromMilliseconds(500) // retry interval
        );

        if (!redLock.IsAcquired)
        {
            // Another instance is running — skip
            return;
        }

        await RunReportAsync(ct);
    }
}
```

**Why not a single `SET NX EX`?** The Redlock algorithm uses multiple Redis nodes to prevent false locks when a single Redis node fails. For Azure Cache for Redis in production, use the Premium tier with clustering.

## Pattern 3: Pub/Sub for Event Fanout

Redis pub/sub enables lightweight event broadcasting. It is **fire-and-forget** — messages are not persisted and late subscribers miss events.

```csharp
// Publisher
public class EventPublisher
{
    private readonly IConnectionMultiplexer _redis;

    public async Task PublishAsync<T>(string channel, T payload)
    {
        var subscriber = _redis.GetSubscriber();
        var message = JsonSerializer.Serialize(payload);
        await subscriber.PublishAsync(RedisChannel.Literal(channel), message);
    }
}

// Subscriber (e.g., in a BackgroundService)
public class PriceUpdateListener : BackgroundService
{
    private readonly IConnectionMultiplexer _redis;

    protected override async Task ExecuteAsync(CancellationToken ct)
    {
        var subscriber = _redis.GetSubscriber();

        await subscriber.SubscribeAsync(
            RedisChannel.Pattern("price-updates:*"),
            (channel, message) =>
            {
                var update = JsonSerializer.Deserialize<PriceUpdate>(message!);
                // Handle update — invalidate cache, notify clients, etc.
                Console.WriteLine(\$"Price update on {channel}: {update?.NewPrice}");
            }
        );

        await Task.Delay(Timeout.Infinite, ct);
    }
}
```

**Pub/Sub vs Streams**: Redis Streams (`XADD`, `XREAD`, `XREADGROUP`) are the durable alternative to pub/sub — messages are persisted and consumer groups allow at-least-once delivery. Use Streams when you cannot afford to miss messages.

## Pattern 4: Atomic Rate Limiting with Lua

Atomicity is critical for rate limiting — check-then-increment must be a single operation to be race-condition-free.

```csharp
public class RedisRateLimiter
{
    private readonly IDatabase _db;

    // Lua script executes atomically on the Redis server — no race conditions
    private static readonly string RateLimitScript = @"
        local key = KEYS[1]
        local limit = tonumber(ARGV[1])
        local window = tonumber(ARGV[2])
        local current = redis.call('INCR', key)
        if current == 1 then
            redis.call('EXPIRE', key, window)
        end
        return current
    ";

    public async Task<bool> IsAllowedAsync(string clientId, int limit, int windowSeconds)
    {
        var key = \$"rate:{clientId}:{DateTimeOffset.UtcNow.ToUnixTimeSeconds() / windowSeconds}";

        var result = await _db.ScriptEvaluateAsync(
            RateLimitScript,
            keys: new RedisKey[] { key },
            values: new RedisValue[] { limit, windowSeconds }
        );

        var count = (long)result;
        return count <= limit;
    }
}
```

<Callout type="senior">
**Senior Insight: Connection Multiplexer Health and Timeouts**

`StackExchange.Redis` is battle-tested but has gotchas that trip up teams in production:

1. **Sync over async deadlocks**: Never call `.Result` or `.Wait()` on Redis operations in ASP.NET Core. Always `await`.

2. **`AbortOnConnectFail = false`**: Set this to prevent your app from crashing on startup if Redis is temporarily unavailable.

3. **Command timeouts**: The default `syncTimeout` is 5 seconds. Under load, commands queue up. Use `connectTimeout` and `syncTimeout` conservatively and add a circuit breaker (Polly) around Redis calls.

4. **Key naming conventions**: Always use a colon-separated hierarchy: `{service}:{entity}:{id}`. This enables Redis's logical namespace grouping and makes `SCAN` patterns tractable.

5. **Avoid `KEYS *` in production**: `KEYS` is O(N) and blocks the server. Use `SCAN` with a cursor for key enumeration.

```csharp
// Safe key scanning with SCAN cursor
var cursor = 0L;
do
{
    var result = await _db.ExecuteAsync("SCAN", cursor, "MATCH", "product:*", "COUNT", 100);
    var arr = (RedisResult[])result!;
    cursor = (long)arr[0];
    var keys = (RedisKey[])arr[1]!;
    // process keys...
} while (cursor != 0);
```
</Callout>
