## The Trap

You're logging events and comparing timestamps. Looks fine locally:

```csharp
// THE BUG — using DateTime.Now for persistence
public class OrderService
{
    public Order CreateOrder(Cart cart)
    {
        return new Order
        {
            Id = Guid.NewGuid(),
            CreatedAt = DateTime.Now, // Local server time — THE TRAP
            Items = cart.Items
        };
    }
}

// Later, sorting orders chronologically:
var sorted = orders.OrderBy(o => o.CreatedAt).ToList();
// Works on one server. Breaks when:
// - Server moves to a different timezone
// - Daylight saving time changes
// - Two servers in different regions both write to the same DB
// - You migrate to a cloud provider in a different region
```

<Callout type="trap">
`DateTime.Now` returns local server time with `DateTimeKind.Local`. When this value is stored in a database or serialized to JSON, the timezone offset is usually lost. Comparing or sorting `DateTime.Now` values from different servers or across DST boundaries produces incorrect ordering.
</Callout>

## Why It Happens: DateTimeKind.Unspecified

The three states of `DateTime.Kind` are a constant source of bugs:

```csharp
DateTime now   = DateTime.Now;   // Kind = Local   (server's timezone)
DateTime utc   = DateTime.UtcNow; // Kind = Utc     (always UTC, no ambiguity)
DateTime plain = new DateTime(2024, 1, 15, 9, 0, 0); // Kind = Unspecified (DANGER)

Console.WriteLine(now.Kind);   // Local
Console.WriteLine(utc.Kind);   // Utc
Console.WriteLine(plain.Kind); // Unspecified — what timezone? nobody knows

// The comparison trap:
var storedTime = new DateTime(2024, 6, 15, 14, 0, 0); // Unspecified — was it UTC or local?
var currentUtc = DateTime.UtcNow;

// This comparison is meaningless if storedTime was stored as local time on a US-East server
// and you are now running on a EU-West server:
bool isExpired = storedTime < currentUtc; // WRONG — comparing apples to oranges
```

**DST double-whammy:**

```csharp
// In Europe/London, clocks go back in October:
// At 2:00 AM, time jumps back to 1:00 AM
// There are now TWO distinct instants that are both "1:30 AM local time"
// DateTime.Now cannot distinguish them!

// With DateTimeOffset:
DateTimeOffset a = new DateTimeOffset(2024, 10, 27, 1, 30, 0, TimeSpan.FromHours(1));  // BST
DateTimeOffset b = new DateTimeOffset(2024, 10, 27, 1, 30, 0, TimeSpan.FromHours(0));  // GMT
// a != b — unambiguous!
```

## The Fix

```csharp
// RULE 1: Always store UTC in databases and APIs
public class Order
{
    // Always UTC — DateTimeKind.Utc, no timezone ambiguity
    public DateTime CreatedAt { get; set; } = DateTime.UtcNow;
}

// RULE 2: Use DateTimeOffset when you need both UTC and offset
public class AuditLog
{
    // Stores the exact instant AND the original offset — best of both worlds
    public DateTimeOffset Timestamp { get; set; } = DateTimeOffset.UtcNow;
}

// RULE 3: Convert to local time only for display, never for storage
public string FormatForUser(DateTime utcTime, string userTimeZoneId)
{
    var tz = TimeZoneInfo.FindSystemTimeZoneById(userTimeZoneId);
    var localTime = TimeZoneInfo.ConvertTimeFromUtc(utcTime, tz);
    return localTime.ToString("g"); // Display only — never store this back
}

// RULE 4: Assert Kind when you receive a DateTime
void SaveOrder(DateTime createdAt)
{
    if (createdAt.Kind != DateTimeKind.Utc)
        throw new ArgumentException("Timestamps must be UTC", nameof(createdAt));
}
```

## DateTimeOffset: The Better Default

```csharp
// DateTimeOffset stores both the instant AND the offset
// It is unambiguous and compares correctly across timezones

DateTimeOffset utcNow    = DateTimeOffset.UtcNow;   // Offset = +00:00
DateTimeOffset localNow  = DateTimeOffset.Now;       // Offset = server's local offset
DateTimeOffset specific  = new DateTimeOffset(2024, 6, 15, 9, 0, 0, TimeSpan.FromHours(5));

// Comparison is always correct — converted to UTC internally
Console.WriteLine(utcNow < localNow); // Correct — compares absolute instants

// SQL Server: use datetimeoffset column type
// EF Core: DateTimeOffset maps to datetimeoffset automatically
// PostgreSQL: use timestamptz (timestamp with time zone)
```

<Callout type="senior">
**Production story: the 1-hour payment discrepancy**

A payment processing system used `DateTime.Now` to record transaction timestamps. Every year in October (UK DST rollback), the finance team reported that some transactions appeared to be processed "before they were initiated" — a 1-hour discrepancy in audit logs.

The root cause: the web server (UK, BST/GMT) recorded `DateTime.Now`, and the reporting database server (UTC-configured) compared them to `DateTime.UtcNow`. During the DST transition, one hour of transactions had locally-ambiguous timestamps.

The fix required a migration to add a `datetimeoffset` column, a backfill script to convert existing data, and a codebase-wide search-and-replace of `DateTime.Now` with `DateTimeOffset.UtcNow`. The migration took three sprints to complete safely.

**Default rule for all new code: `DateTimeOffset.UtcNow` for everything that goes into a database or API response. `DateTime.Now` only for transient UI display.**
</Callout>
