## The Trap

```csharp
// Looks like a simple equality check — THE TRAP
double price = 0.1 + 0.2;
Console.WriteLine(price == 0.3);        // False!
Console.WriteLine(price);               // 0.30000000000000004

// Financial calculation gone wrong:
double total = 0.0;
for (int i = 0; i < 10; i++)
{
    total += 0.1; // Adding ten tenths
}
Console.WriteLine(total == 1.0);        // False!
Console.WriteLine(total);               // 0.9999999999999999

// The tax calculation trap:
double subtotal = 19.99;
double taxRate  = 0.08;
double tax      = subtotal * taxRate;   // 1.5992000000000002 — not 1.5992
double invoice  = subtotal + tax;       // 21.589200000000002
```

<Callout type="trap">
`double` is an IEEE 754 binary floating-point type. It **cannot represent most decimal fractions exactly** — 0.1 in binary is an infinitely repeating fraction, like 1/3 in decimal. Every arithmetic operation accumulates this rounding error. In financial code, this means cents appear or disappear.
</Callout>

## Why It Happens: Binary vs Decimal Representation

```csharp
// 0.1 in binary is approximately:
// 0.0001100110011001100110011001100110011001100110011... (repeating)
// Stored as the nearest representable double:
// 0.1000000000000000055511151231257827021181583404541015625

// When you write 0.1 + 0.2:
// 0.1 → stored as 0.1000000000000000055511151231257827...
// 0.2 → stored as 0.200000000000000011102230246251565404...
// Sum → 0.3000000000000000444089209850062616169452667...
// Nearest representable double: 0.30000000000000004

// decimal uses base-10 representation internally:
decimal d = 0.1m; // Stored EXACTLY as 1 × 10^-1
decimal sum = 0.1m + 0.2m;
Console.WriteLine(sum == 0.3m); // True! Exact.
```

## The Fix: Use decimal for Money

```csharp
// CORRECT — use decimal for all monetary values
public class Invoice
{
    public decimal Subtotal { get; set; }
    public decimal TaxRate  { get; set; }

    public decimal Tax   => Math.Round(Subtotal * TaxRate, 2, MidpointRounding.AwayFromZero);
    public decimal Total => Subtotal + Tax;
}

var invoice = new Invoice
{
    Subtotal = 19.99m,  // Note the 'm' suffix — decimal literal
    TaxRate  = 0.08m
};

Console.WriteLine(invoice.Tax);   // 1.60 — correct (19.99 * 0.08 = 1.5992, rounds to 1.60)
Console.WriteLine(invoice.Total); // 21.59 — correct

// Always use decimal literals (m suffix):
decimal price = 9.99m;   // Correct
decimal price2 = (decimal)9.99; // WRONG — 9.99 was already imprecise as a double before cast!
```

## The HasFlag / Enum Double Trap (bonus)

```csharp
// Another double-equality trap: never use == for doubles
double a = 1.0 / 3.0;
double b = 0.3333333333333333;

Console.WriteLine(a == b);              // Might be true or false — unreliable

// Correct: use a tolerance (epsilon comparison)
const double Epsilon = 1e-10;
Console.WriteLine(Math.Abs(a - b) < Epsilon); // Reliable

// For financial totals: ALWAYS use decimal, no epsilon games needed
```

<Callout type="senior">
**Production story: the $0.01 discrepancy that triggered a regulatory audit**

A fintech startup processed millions of micro-transactions using `double` for currency. Each transaction was individually correct when rounded to 2 decimal places for display. But the database stored the raw `double` values, and the end-of-day reconciliation summed them in double arithmetic.

Over a full trading day, the accumulated rounding errors across 2 million transactions produced a $47.23 discrepancy between the sum of individual transactions and the aggregate. This was flagged by the clearing house as a reconciliation failure. The regulatory body opened an inquiry.

The fix: migrate all monetary columns from `FLOAT` to `DECIMAL(19,4)` in SQL Server, and replace all `double` with `decimal` in the C# codebase. The migration took a week and required careful backfill arithmetic to avoid converting already-imprecise doubles to decimals incorrectly.

**The rule is absolute: `decimal` for money, always, everywhere, no exceptions.**
</Callout>
