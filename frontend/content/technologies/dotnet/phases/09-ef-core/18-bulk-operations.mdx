## The Problem with Row-by-Row EF Operations

Traditional EF updates require loading entities into memory, modifying them, and saving:

```csharp
// SLOW: loads 10,000 entities, tracks them all, sends 10,000 UPDATE statements
var products = await db.Products
    .Where(p => p.CategoryId == 5)
    .ToListAsync(); // 10,000 rows → 10,000 entity objects in memory

foreach (var product in products)
{
    product.IsActive = false;
}

await db.SaveChangesAsync(); // 10,000 individual UPDATE statements (or batched but still slow)
```

For set-based operations, this approach is:
- Memory-intensive (thousands of tracked entities)
- Slow (many database round-trips even with batching)
- Unnecessary (the database can do this in one statement)

## ExecuteUpdateAsync (EF Core 7+)

`ExecuteUpdateAsync` generates a single `UPDATE ... SET ... WHERE ...` statement:

```csharp
// Update all products in a category — one SQL statement, no entity loading
int rowsUpdated = await db.Products
    .Where(p => p.CategoryId == 5)
    .ExecuteUpdateAsync(setters => setters
        .SetProperty(p => p.IsActive, false)
        .SetProperty(p => p.UpdatedAt, DateTime.UtcNow));
// SQL: UPDATE Products SET IsActive = 0, UpdatedAt = @p0 WHERE CategoryId = 5

// Using computed values (reference other columns)
await db.Products
    .Where(p => p.Stock < 10)
    .ExecuteUpdateAsync(setters => setters
        .SetProperty(p => p.Price, p => p.Price * 1.1m)  // 10% price increase
        .SetProperty(p => p.UpdatedAt, DateTime.UtcNow));
// SQL: UPDATE Products SET Price = Price * 1.1, UpdatedAt = @p0 WHERE Stock < 10

// Combine with any IQueryable composition
await db.Orders
    .Where(o => o.Status == OrderStatus.Pending)
    .Where(o => o.CreatedAt < DateTime.UtcNow.AddDays(-30))
    .ExecuteUpdateAsync(setters => setters
        .SetProperty(o => o.Status, OrderStatus.Cancelled)
        .SetProperty(o => o.CancelledAt, DateTime.UtcNow)
        .SetProperty(o => o.CancelledReason, "Auto-cancelled: no payment"));
```

## ExecuteDeleteAsync (EF Core 7+)

`ExecuteDeleteAsync` generates a single `DELETE ... WHERE ...`:

```csharp
// Delete all expired sessions — one SQL DELETE
int deleted = await db.Sessions
    .Where(s => s.ExpiresAt < DateTime.UtcNow)
    .ExecuteDeleteAsync();
// SQL: DELETE FROM Sessions WHERE ExpiresAt < @p0

// Soft-delete alternative: use ExecuteUpdate instead
await db.Sessions
    .Where(s => s.ExpiresAt < DateTime.UtcNow)
    .ExecuteUpdateAsync(s => s
        .SetProperty(x => x.IsDeleted, true)
        .SetProperty(x => x.DeletedAt, DateTime.UtcNow));
```

<Callout type="trap">
**ExecuteDelete and Global Query Filters**

`ExecuteDeleteAsync` and `ExecuteUpdateAsync` respect global query filters by default. If you have a soft-delete filter (`!IsDeleted`), the generated DELETE will include `AND IsDeleted = 0` in the WHERE clause. To bypass, call `IgnoreQueryFilters()` before `ExecuteDeleteAsync()`:

```csharp
// Permanently delete already-soft-deleted records (admin operation)
await db.Products
    .IgnoreQueryFilters()
    .Where(p => p.IsDeleted && p.DeletedAt < DateTime.UtcNow.AddDays(-90))
    .ExecuteDeleteAsync();
```

Also: `ExecuteUpdateAsync` and `ExecuteDeleteAsync` **bypass the change tracker entirely**. Entities already in memory are not updated — you must reload them or evict them from the tracker.
</Callout>

## Bulk Inserts

EF Core does not have a built-in bulk insert (single-statement `INSERT ... SELECT` or table-valued parameter). For high-volume inserts, use one of these strategies:

**Strategy 1: AddRange + batched SaveChanges**

```csharp
// EF Core batches INSERT statements automatically (default batch size: 42 for SQL Server)
// Good for up to ~5,000 rows

db.ChangeTracker.AutoDetectChangesEnabled = false;
const int batchSize = 500;

for (int i = 0; i < records.Count; i += batchSize)
{
    db.Products.AddRange(records.Skip(i).Take(batchSize));
    await db.SaveChangesAsync();
    db.ChangeTracker.Clear(); // free memory after each batch
}

db.ChangeTracker.AutoDetectChangesEnabled = true;
```

**Strategy 2: EFCore.BulkExtensions (third-party, most popular)**

```csharp
// NuGet: EFCore.BulkExtensions
await db.BulkInsertAsync(products); // single SQL bulk insert
await db.BulkUpdateAsync(products); // SQL Server MERGE or UPDATE ... FROM
await db.BulkDeleteAsync(products); // bulk DELETE

// Configuration options
await db.BulkInsertAsync(products, options =>
{
    options.BatchSize = 1000;
    options.SetOutputIdentity = true; // populate auto-generated IDs
    options.BulkCopyTimeout = 30;
});
```

**Strategy 3: SqlBulkCopy for maximum throughput**

```csharp
// Raw ADO.NET bulk copy — fastest possible, no EF overhead
using var connection = new SqlConnection(connectionString);
await connection.OpenAsync();

using var bulkCopy = new SqlBulkCopy(connection)
{
    DestinationTableName = "Products",
    BatchSize = 5000,
    BulkCopyTimeout = 60
};

// Map columns
bulkCopy.ColumnMappings.Add("Name", "Name");
bulkCopy.ColumnMappings.Add("Price", "Price");
bulkCopy.ColumnMappings.Add("CategoryId", "CategoryId");

var table = ToDataTable(products); // convert to DataTable
await bulkCopy.WriteToServerAsync(table);
```

<Callout type="senior">
**Senior Insight: ExecuteUpdate Does Not Fire Interceptors**

`ExecuteUpdateAsync` and `ExecuteDeleteAsync` are set-based SQL operations that bypass the EF change tracker entirely — which also means they bypass `SaveChangesInterceptor`. Your audit interceptor setting `UpdatedAt` will NOT fire for `ExecuteUpdateAsync` calls.

Solutions:
1. Manually set audit fields in the `ExecuteUpdateAsync` call itself (`.SetProperty(e => e.UpdatedAt, DateTime.UtcNow)`)
2. Use database triggers for audit fields (the one case where triggers are justified)
3. Accept the trade-off: bulk operations sacrifice per-entity tracking for throughput

This is a deliberate design choice — if you needed per-entity lifecycle hooks, you would not use set-based operations.
</Callout>
