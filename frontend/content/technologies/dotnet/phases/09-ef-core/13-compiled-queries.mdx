## How EF Core Caches Queries

Every LINQ query goes through an expression tree compilation pipeline:

1. **Parse** the LINQ expression tree
2. **Translate** to SQL (using the provider)
3. **Cache** the compiled query plan (keyed by expression shape)
4. **Execute** with the current parameter values

Steps 1–3 happen only once per unique query shape — subsequent calls hit the cache. The cache uses a hash of the expression tree structure as the key.

**The cache lookup itself is not free.** For a query called 100,000 times per second, even a ~5µs cache lookup adds 500ms of overhead per second. Compiled queries eliminate this.

## EF.CompileQuery

`EF.CompileQuery` creates a delegate that bypasses the expression parsing and cache lookup entirely:

```csharp
// Define at the class level — not inside a method (compile once, reuse many times)
public static class CompiledQueries
{
    // Compile a query that takes DbContext + one parameter
    public static readonly Func<AppDbContext, int, Task<Product?>> GetProductById =
        EF.CompileAsyncQuery((AppDbContext db, int id) =>
            db.Products.FirstOrDefault(p => p.Id == id));

    // Multiple parameters
    public static readonly Func<AppDbContext, int, bool, IAsyncEnumerable<Product>> GetProductsByCategory =
        EF.CompileAsyncQuery((AppDbContext db, int categoryId, bool activeOnly) =>
            db.Products
                .Where(p => p.CategoryId == categoryId)
                .Where(p => !activeOnly || p.IsActive)
                .OrderBy(p => p.Name));

    // Query returning a collection (use IAsyncEnumerable for streaming)
    public static readonly Func<AppDbContext, decimal, IAsyncEnumerable<Product>> GetExpensiveProducts =
        EF.CompileAsyncQuery((AppDbContext db, decimal minPrice) =>
            db.Products
                .AsNoTracking()
                .Where(p => p.Price >= minPrice)
                .OrderByDescending(p => p.Price)
                .Take(100));
}
```

**Usage:**

```csharp
public class ProductService
{
    private readonly AppDbContext _db;

    public ProductService(AppDbContext db) => _db = db;

    public async Task<Product?> GetByIdAsync(int id)
    {
        // Direct delegate call — no expression parsing, no cache lookup
        return await CompiledQueries.GetProductById(_db, id);
    }

    public async Task<List<Product>> GetByCategoryAsync(int categoryId, bool activeOnly)
    {
        var results = new List<Product>();
        await foreach (var product in CompiledQueries.GetProductsByCategory(_db, categoryId, activeOnly))
        {
            results.Add(product);
        }
        return results;
    }
}
```

## Sync vs. Async Compiled Queries

```csharp
// Synchronous compiled query
public static readonly Func<AppDbContext, int, Product?> GetProductByIdSync =
    EF.CompileQuery((AppDbContext db, int id) =>
        db.Products.FirstOrDefault(p => p.Id == id));

// Async compiled query returning single entity
public static readonly Func<AppDbContext, int, Task<Product?>> GetProductByIdAsync =
    EF.CompileAsyncQuery((AppDbContext db, int id) =>
        db.Products.FirstOrDefault(p => p.Id == id));

// Async compiled query returning collection — returns IAsyncEnumerable
public static readonly Func<AppDbContext, IAsyncEnumerable<Product>> GetAllActiveProducts =
    EF.CompileAsyncQuery((AppDbContext db) =>
        db.Products.Where(p => p.IsActive));
```

## Performance Impact

```csharp
// Benchmark comparison (BenchmarkDotNet)
[MemoryDiagnoser]
public class QueryBenchmarks
{
    private AppDbContext _db = null!;

    [Benchmark(Baseline = true)]
    public async Task<Product?> StandardLinq()
    {
        return await _db.Products.FirstOrDefaultAsync(p => p.Id == 42);
    }

    [Benchmark]
    public async Task<Product?> CompiledQuery()
    {
        return await CompiledQueries.GetProductById(_db, 42);
    }
}
// Typical results on a local SQL Server:
// StandardLinq  : 45µs,  0 B allocated  (query plan cached)
// CompiledQuery : 38µs,  0 B allocated
// Difference per call: ~7µs — meaningful at 100K calls/sec, negligible at 100 calls/sec
```

## Limitations of Compiled Queries

```csharp
// 1. No dynamic LINQ — the query shape must be fixed at compile time
// This does NOT work:
var sortField = "Price"; // dynamic sort
var compiled = EF.CompileQuery((AppDbContext db) =>
    db.Products.OrderBy(p => EF.Property<object>(p, sortField))); // Error

// 2. No Include() in many configurations (works in some EF versions)
// Test your specific query to confirm

// 3. Parameters are limited to primitive types + DbContext
// No IEnumerable parameters — so no .Contains(list) in compiled queries
// Workaround: use raw SQL for IN queries, or accept this limitation

// 4. Cannot capture closure variables — must be explicit parameters
int categoryId = 5;
// WRONG — captures closure:
var wrong = EF.CompileQuery((AppDbContext db) =>
    db.Products.Where(p => p.CategoryId == categoryId));
// RIGHT — explicit parameter:
var correct = EF.CompileQuery((AppDbContext db, int catId) =>
    db.Products.Where(p => p.CategoryId == catId));
```

<Callout type="senior">
**Senior Insight: When Compiled Queries Are Worth It**

Compiled queries are a micro-optimization. Apply the rule: **measure first, optimize second**.

Compiled queries give the most value when:
1. The same query runs thousands of times per second on a hot path (e.g., product catalog lookup, session validation)
2. The query is simple (single entity by ID, short filtered list)
3. You have already eliminated N+1, added correct indexes, and used AsNoTracking — and still need more throughput

They are NOT worth it for:
- Complex queries with many joins (translation time is already spent, tiny relative savings)
- Queries called rarely (the compilation cost is paid once per process lifetime anyway via cache)
- Queries with dynamic shape requirements (use raw SQL instead)

In practice, `AsNoTracking` + `Select()` projection often gives more improvement than compiled queries for the same query.
</Callout>

<MythBuster
  myth="EF Core recompiles LINQ queries from scratch every time they are executed"
  truth="EF Core has a query plan cache (enabled by default) that stores the compiled query plan keyed by expression tree structure. After the first execution of a unique query shape, subsequent calls hit the cache and only incur a cache lookup (not full recompilation). Compiled queries via EF.CompileQuery eliminate even the cache lookup by storing the compiled delegate directly, providing a smaller but measurable improvement on very hot paths."
/>
