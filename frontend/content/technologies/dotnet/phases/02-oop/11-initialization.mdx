## Three Ways to Initialize Objects

C# gives you three primary patterns for creating initialized objects, each with different trade-offs:

1. **Constructor parameters** — enforced at compile time, can validate, can be `required`
2. **Object initializers** — flexible, readable, but bypasses constructor validation
3. **Factory methods** — full control, can be async, can return different subtypes

---

## Constructor Parameters: Enforced Initialization

Constructors are the most powerful form of initialization because they can **guarantee** an object is valid the moment it's constructed:

```csharp
public class EmailMessage
{
    public string To      { get; }
    public string Subject { get; }
    public string Body    { get; }
    public DateTime SentAt { get; }

    public EmailMessage(string to, string subject, string body)
    {
        To      = to      ?? throw new ArgumentNullException(nameof(to));
        Subject = subject ?? throw new ArgumentNullException(nameof(subject));
        Body    = body    ?? throw new ArgumentNullException(nameof(body));

        if (!to.Contains('@')) throw new ArgumentException("Invalid email address", nameof(to));
        if (subject.Length > 200) throw new ArgumentException("Subject too long", nameof(subject));

        SentAt = DateTime.UtcNow;
    }
}

// Caller MUST provide all values — compiler enforces it
var msg = new EmailMessage("alice@example.com", "Hello", "Hi there!"); // OK
// var bad = new EmailMessage("notanemail", "Hi", ""); // Throws ArgumentException
// var missing = new EmailMessage("alice@example.com"); // COMPILE ERROR — wrong arg count
```

**Use constructors when:**
- The object has required properties without which it's invalid
- You need to validate input before storing it
- You want to guarantee immutability (readonly properties only settable in constructor)
- The type has complex initialization logic

## Object Initializers: Flexibility and Readability

Object initializers set properties after construction. They require no changes to the class — any settable property can be set this way:

```csharp
public class HttpRequestOptions
{
    public string Method    { get; set; } = "GET";
    public TimeSpan Timeout { get; set; } = TimeSpan.FromSeconds(30);
    public bool FollowRedirects { get; set; } = true;
    public int MaxRetries       { get; set; } = 3;
    public Dictionary<string, string> Headers { get; set; } = new();
}

// Object initializer — set only what you need, rest use defaults
var options = new HttpRequestOptions
{
    Method  = "POST",
    Timeout = TimeSpan.FromSeconds(60),
    Headers = { ["Authorization"] = "Bearer token123" } // collection initializer inside!
};

// Equivalent to:
var options2 = new HttpRequestOptions();
options2.Method  = "POST";
options2.Timeout = TimeSpan.FromSeconds(60);
options2.Headers["Authorization"] = "Bearer token123";
```

<Callout type="trap">
**Object initializers bypass constructor logic.** If you have validation in a setter and an object initializer sets the property, the setter validation runs. But if validation lives only in the constructor, object initializers can produce invalid objects:

```csharp
public class Range
{
    public int Min { get; set; }
    public int Max { get; set; }

    public Range(int min, int max)
    {
        if (min >= max) throw new ArgumentException("Min must be less than Max");
        Min = min;
        Max = max;
    }
}

// This bypasses the cross-property validation in the constructor!
// Using object initializer skips the constructor logic:
// WAIT — actually this DOES use the constructor (parameterless) — but Range has no parameterless ctor
// So this would compile only if a parameterless constructor exists.
// The trap: if you ADD a parameterless ctor later, object initializers can create invalid states.
```

With `init`-only properties and `required` members (C# 9/11), you get a middle ground: object initializer flexibility with some enforcement.
</Callout>

## Combining `required` + `init` (The Modern Pattern)

C# 9's `init` and C# 11's `required` together give you object-initializer convenience with compile-time enforcement:

```csharp
public class ApiClientOptions
{
    // required + init: must provide in initializer, cannot change after
    public required string BaseUrl { get; init; }
    public required string ApiKey  { get; init; }

    // Optional with defaults
    public TimeSpan Timeout     { get; init; } = TimeSpan.FromSeconds(30);
    public int      MaxRetries  { get; init; } = 3;
    public bool     UseHttps    { get; init; } = true;
}

// COMPILE ERROR if required members missing:
// var opts = new ApiClientOptions { BaseUrl = "https://api.example.com" }; // Missing ApiKey!

// Correct:
var opts = new ApiClientOptions
{
    BaseUrl = "https://api.example.com",
    ApiKey  = "secret-key",
    Timeout = TimeSpan.FromSeconds(60) // optional override
};
```

## Factory Methods: Maximum Control

Static factory methods replace constructors when you need:
- **Async initialization** (constructors can't be async)
- **Different subtypes** depending on input
- **Named constructors** with clearer intent than overloaded parameter lists
- **Caching or pooling** of instances

```csharp
public class DatabaseConnection
{
    private readonly string _connectionString;
    private bool _isOpen;

    // Private constructor — forces use of factory methods
    private DatabaseConnection(string connectionString)
        => _connectionString = connectionString;

    // Named factory methods — intent is clearer than constructor overloads
    public static DatabaseConnection ForSqlServer(string server, string database)
        => new($"Server={server};Database={database};Trusted_Connection=true");

    public static DatabaseConnection ForPostgres(string host, string database, string password)
        => new($"Host={host};Database={database};Password={password}");

    // Async factory — constructors can't be async, factories can
    public static async Task<DatabaseConnection> OpenAsync(string connectionString)
    {
        var conn = new DatabaseConnection(connectionString);
        await conn.OpenInternalAsync(); // async initialization
        return conn;
    }

    // Can return null or throw based on validation
    public static DatabaseConnection? TryCreate(string connectionString)
    {
        if (string.IsNullOrWhiteSpace(connectionString)) return null;
        return new DatabaseConnection(connectionString);
    }

    private async Task OpenInternalAsync()
    {
        // simulate async open
        await Task.Delay(10);
        _isOpen = true;
    }
}

// Usage:
var sqlConn = DatabaseConnection.ForSqlServer("localhost", "MyDb");
var pgConn  = DatabaseConnection.ForPostgres("localhost", "MyDb", "secret");
var asyncConn = await DatabaseConnection.OpenAsync("Server=localhost;Database=MyDb");
```

<Callout type="senior">
**Senior Insight: The Builder Pattern with Validation Accumulation**

Production builders accumulate errors rather than throwing on the first one, providing better UX for configuration errors:

```csharp
public class ReportBuilder
{
    private string? _title;
    private List<string> _sections = new();
    private string _pageSize = "A4";
    private readonly List<string> _errors = new();

    public ReportBuilder WithTitle(string title)
    {
        if (string.IsNullOrWhiteSpace(title))
            _errors.Add("Title is required and cannot be empty");
        _title = title;
        return this;
    }

    public ReportBuilder WithSection(string section)
    {
        if (string.IsNullOrWhiteSpace(section))
            _errors.Add($"Section {_sections.Count + 1} cannot be empty");
        else
            _sections.Add(section);
        return this;
    }

    public ReportBuilder WithPageSize(string size)
    {
        if (size is not ("A4" or "Letter" or "Legal"))
            _errors.Add($"Unsupported page size: {size}. Use A4, Letter, or Legal.");
        _pageSize = size;
        return this;
    }

    public async Task<Report> BuildAsync(CancellationToken ct = default)
    {
        if (!_sections.Any())
            _errors.Add("At least one section is required");

        if (_errors.Any())
            throw new ValidationException($"Report configuration invalid:\n- {string.Join("\n- ", _errors)}");

        // Async I/O: load template
        var template = await LoadTemplateAsync(_pageSize, ct);

        return new Report(_title!, _sections.AsReadOnly(), template, _pageSize);
    }

    private static async Task<byte[]> LoadTemplateAsync(string pageSize, CancellationToken ct)
    {
        // Load template files from disk/blob storage
        await Task.Delay(1, ct); // simulated
        return Array.Empty<byte>();
    }
}
```

This pattern — accumulate errors, validate in `Build()` — is used extensively in configuration validation frameworks like FluentValidation and ASP.NET Core's `IOptions` validation.
</Callout>
