## How the ThreadPool Works

The .NET ThreadPool manages a pool of worker threads that run async continuations, `Task.Run` work, and other scheduled callbacks. Key properties:

- **Minimum threads**: initially set to `Environment.ProcessorCount` (e.g., 8 threads on an 8-core CPU)
- **Maximum threads**: typically 32,767 workers (configurable)
- **Thread injection rate**: when all threads are busy, the pool adds ~1 new thread per second (the "hill-climbing algorithm")

```csharp
// Query current ThreadPool state
ThreadPool.GetMinThreads(out int minWorkers, out int minIo);
ThreadPool.GetMaxThreads(out int maxWorkers, out int maxIo);
ThreadPool.GetAvailableThreads(out int freeWorkers, out int freeIo);

Console.WriteLine($"Min workers: {minWorkers}"); // ~processor count
Console.WriteLine($"Max workers: {maxWorkers}"); // 32767+
Console.WriteLine($"Free workers: {freeWorkers}"); // Should be high; low = trouble
```

## What Causes Starvation

Starvation occurs when all ThreadPool threads are **blocked** waiting for something (I/O, locks, `.Result` calls). New work queues up but no threads are free to run it.

```csharp
// Classic starvation pattern: blocking sync calls inside async context
[HttpGet("reports/{count}")]
public async Task<IActionResult> GenerateReports(int count)
{
    var tasks = Enumerable.Range(0, count).Select(i =>
        Task.Run(() =>
        {
            // Each Task.Run claims a ThreadPool thread, then BLOCKS it:
            var data = _legacyService.GetData(i); // Synchronous, blocks 2 seconds
            return Process(data);
        })
    );

    var results = await Task.WhenAll(tasks);
    return Ok(results);
}

// With count=100 and only 8+ ThreadPool threads:
// - 8 threads start, each blocks for 2 seconds
// - 92 tasks queue up, waiting for threads
// - ThreadPool injects ~1 thread/second — takes 92+ seconds to recover
// - All 100 requests timeout before completing
// - Symptom: requests queue up, response times spike, CPU is at 0%
```

## Symptoms of ThreadPool Starvation

1. **Requests/tasks queuing with 0% CPU** — the box is idle but unresponsive
2. **Steadily increasing response times** — as more threads block, more requests queue
3. **Thread count climbing slowly** — ThreadPool hill-climbing adds threads slowly
4. **Event log or metrics show high thread count with low throughput**
5. **Timeouts on operations that aren't slow themselves** — queuing delay is the culprit

```csharp
// Detect starvation programmatically:
public class ThreadPoolMonitor : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            ThreadPool.GetMaxThreads(out int maxWorkers, out _);
            ThreadPool.GetAvailableThreads(out int freeWorkers, out _);
            int queueDepth = maxWorkers - freeWorkers;

            if (queueDepth > 100) // Threshold for alerting
                _metrics.RecordStarvationAlert(queueDepth);

            await Task.Delay(1000, stoppingToken);
        }
    }
}
```

## The Root Causes and Fixes

**Root cause 1: Blocking I/O (`.Result`, `.Wait()`)**

```csharp
// BAD: blocks a ThreadPool thread for the duration of the HTTP call
public Task<string> GetDataBad(string url)
    => Task.Run(() => new HttpClient().GetStringAsync(url).Result); // Double wrong

// GOOD: use async all the way
public async Task<string> GetDataGood(string url)
    => await new HttpClient().GetStringAsync(url);
```

**Root cause 2: Thread.Sleep in async code**

```csharp
// BAD: blocks a ThreadPool thread for 5 seconds
await Task.Run(() => Thread.Sleep(5000)); // Holds a thread while sleeping

// GOOD: non-blocking delay
await Task.Delay(5000); // No thread consumed during delay
```

**Root cause 3: Synchronous database calls in web handlers**

```csharp
// BAD: synchronous EF Core SaveChanges blocks a thread
public async Task<IActionResult> UpdateBad(Model m)
{
    _db.Models.Update(m);
    _db.SaveChanges();  // Blocks! Use SaveChangesAsync
    return Ok();
}

// GOOD:
public async Task<IActionResult> UpdateGood(Model m)
{
    _db.Models.Update(m);
    await _db.SaveChangesAsync();
    return Ok();
}
```

<Callout type="senior">
**Senior Insight: Minimum thread count tuning for legacy sync code**

When you have unavoidable blocking calls (legacy SDK, third-party library with no async API), you can tune the minimum thread count to reduce the starvation window:

```csharp
// In Program.cs or startup — increase minimum worker threads
int processorCount = Environment.ProcessorCount;
ThreadPool.SetMinThreads(
    workerThreads: processorCount * 4, // Bump up minimum (default is processorCount)
    completionPortThreads: processorCount * 4);

// This pre-allocates more threads so they're immediately available
// and reduces the hill-climbing delay from ~1 thread/second

// Alternatively, in Docker/Kubernetes:
// Set DOTNET_THREAD_POOL_MIN_THREADS environment variable
// DOTNET_THREAD_POOL_MIN_THREADS=50
```

This is a band-aid, not a fix. The real fix is making everything async. But during migrations or for third-party dependencies, bumping minimum threads buys time and reduces latency spikes.

Another approach: use a dedicated `ThreadPool` via `TaskCreationOptions.LongRunning` for genuinely long-blocking operations:

```csharp
// TaskCreationOptions.LongRunning creates a DEDICATED thread (not from the pool)
// Use this ONLY for truly long-running synchronous work (minutes, not seconds)
var dedicatedTask = Task.Factory.StartNew(
    () => LegacyLongBlockingOperation(),
    CancellationToken.None,
    TaskCreationOptions.LongRunning, // Creates a new OS thread, not from pool
    TaskScheduler.Default);

await dedicatedTask;
```

This keeps the blocking work off the ThreadPool entirely.
</Callout>

<MythBuster
  myth="async/await uses more threads and is slower than synchronous code"
  reality="async/await actually uses FEWER threads than synchronous blocking code. A synchronous web handler holds a ThreadPool thread for the entire duration of I/O. An async handler releases the thread during I/O and reclaims one only to process the response. Under load, async allows hundreds of concurrent requests on dozens of threads. Sync handlers need hundreds of threads — quickly exhausting the ThreadPool. async is slower per single request (state machine overhead), but 10x-100x better for throughput under load."
/>
