## The Primitive Landscape

.NET provides several synchronization primitives. Choosing the wrong one has real consequences for throughput, correctness, and debuggability:

## lock and Monitor

`lock(obj) { ... }` is syntactic sugar for `Monitor.Enter` / `Monitor.Exit`:

```csharp
private readonly object _syncRoot = new object();
private int _count;

// lock keyword — always prefer this over raw Monitor for simple cases
public void Increment()
{
    lock (_syncRoot)
    {
        _count++;
    }
}

// Equivalent Monitor code (what lock compiles to):
public void IncrementViaMonitor()
{
    bool lockTaken = false;
    try
    {
        Monitor.Enter(_syncRoot, ref lockTaken);
        _count++;
    }
    finally
    {
        if (lockTaken) Monitor.Exit(_syncRoot);
    }
}

// Monitor.TryEnter — non-blocking attempt with timeout
public bool TryIncrement(TimeSpan timeout)
{
    if (Monitor.TryEnter(_syncRoot, timeout))
    {
        try
        {
            _count++;
            return true;
        }
        finally
        {
            Monitor.Exit(_syncRoot);
        }
    }
    return false; // Could not acquire — contended
}
```

<Callout type="trap">
**Never lock on `this`, `typeof(T)`, or a string literal.** These are public objects — external code can lock on the same object, creating deadlock opportunities you cannot control. Always lock on a **private `readonly object`** field created specifically for synchronization.
</Callout>

## lock Cannot Be Used with await

This is a critical limitation — the compiler prevents it:

```csharp
// This does NOT compile:
public async Task UpdateAsync()
{
    lock (_syncRoot)
    {
        await DoWorkAsync(); // CS1996: Cannot await in the body of a lock statement
    }
}

// Why? The lock is held by the current thread.
// After await, execution resumes on a potentially DIFFERENT thread.
// Monitor.Exit on a different thread throws SynchronizationLockException.
```

The solution is `SemaphoreSlim`:

```csharp
private readonly SemaphoreSlim _semaphore = new SemaphoreSlim(1, 1); // mutex (count 1)

public async Task UpdateAsync()
{
    await _semaphore.WaitAsync(); // Async acquire
    try
    {
        await DoWorkAsync(); // Safe to await while semaphore is held
    }
    finally
    {
        _semaphore.Release(); // Always release — threads change between await points
    }
}
```

## SemaphoreSlim: More Than Just an Async Lock

`SemaphoreSlim(initialCount, maxCount)` is a counting semaphore — up to `maxCount` concurrent acquirers:

```csharp
// As an async mutex (count=1): mutual exclusion for async code
private readonly SemaphoreSlim _mutex = new SemaphoreSlim(1, 1);

// As a concurrency limiter (count=N): throttle parallel operations
private readonly SemaphoreSlim _throttle = new SemaphoreSlim(5, 5); // Max 5 concurrent

// Throttled parallel HTTP calls
public async Task<string[]> FetchManyAsync(string[] urls, CancellationToken ct)
{
    var tasks = urls.Select(async url =>
    {
        await _throttle.WaitAsync(ct);
        try
        {
            return await _httpClient.GetStringAsync(url, ct);
        }
        finally
        {
            _throttle.Release();
        }
    });

    return await Task.WhenAll(tasks);
}

// WaitAsync with timeout
bool acquired = await _semaphore.WaitAsync(TimeSpan.FromSeconds(5), ct);
if (!acquired)
    throw new TimeoutException("Could not acquire semaphore");
```

## ReaderWriterLockSlim: Read-Heavy Workloads

When reads vastly outnumber writes, `ReaderWriterLockSlim` allows multiple concurrent readers but exclusive writers:

```csharp
private readonly ReaderWriterLockSlim _rwLock = new ReaderWriterLockSlim();
private Dictionary<string, User> _cache = new();

// Multiple threads can read simultaneously
public User? GetUser(string id)
{
    _rwLock.EnterReadLock();
    try
    {
        return _cache.TryGetValue(id, out var user) ? user : null;
    }
    finally
    {
        _rwLock.ExitReadLock();
    }
}

// Only one thread can write at a time — all readers blocked during write
public void AddUser(User user)
{
    _rwLock.EnterWriteLock();
    try
    {
        _cache[user.Id] = user;
    }
    finally
    {
        _rwLock.ExitWriteLock();
    }
}

// Upgradeable read: "maybe write" — check then conditionally upgrade
public void EnsureUser(User user)
{
    _rwLock.EnterUpgradeableReadLock();
    try
    {
        if (!_cache.ContainsKey(user.Id))
        {
            _rwLock.EnterWriteLock();
            try
            {
                _cache[user.Id] = user; // Double-checked inside write lock
            }
            finally
            {
                _rwLock.ExitWriteLock();
            }
        }
    }
    finally
    {
        _rwLock.ExitUpgradeableReadLock();
    }
}
```

<Callout type="warning">
**ReaderWriterLockSlim is not async-compatible** — the same thread must acquire and release the lock. Don't use it with async methods. For async read-heavy scenarios, consider `ConcurrentDictionary` (which uses fine-grained internal locks) or an immutable replacement pattern.
</Callout>

<Callout type="senior">
**Senior Insight: Prefer lock-free data structures and immutability**

The best lock is often no lock at all:

```csharp
// Option 1: ConcurrentDictionary — lock-free reads, fine-grained write locks
private readonly ConcurrentDictionary<string, User> _cache = new();

public User GetOrAdd(string id, Func<string, User> factory)
    => _cache.GetOrAdd(id, factory); // Thread-safe, often lock-free for reads

// Option 2: ImmutableDictionary with Interlocked.CompareExchange (snapshot replacement)
private volatile ImmutableDictionary<string, User> _cache = ImmutableDictionary<string, User>.Empty;

public void AddUser(User user)
{
    ImmutableDictionary<string, User> current, updated;
    do
    {
        current = _cache;
        updated = current.Add(user.Id, user);
    } while (Interlocked.CompareExchange(ref _cache, updated, current) != current);
    // No lock — wait-free snapshot replacement
}

// Option 3: Channel<T> — eliminate shared state by message passing
// Producers write commands to a channel; a single consumer owns the state
// No locking needed because only one consumer touches the state
```

The immutable snapshot pattern works well for read-heavy data that changes infrequently (config, feature flags). Reads are completely lock-free — they just read the current `_cache` reference.
</Callout>

<Quiz
  id="05-async-concurrency/lock-types/quiz-1"
  question="Why can't you use 'lock' with 'await' inside the locked region?"
  options={[
    "Performance reasons — the compiler disallows it to prevent slowdowns",
    "After an await, execution may resume on a different thread, but Monitor.Exit must be called on the same thread that called Monitor.Enter",
    "lock uses a kernel object that doesn't support async operations",
    "await requires a SynchronizationContext, which is incompatible with lock"
  ]}
  correctAnswer={1}
  explanation="Monitor.Enter and Monitor.Exit are thread-affine: Exit must be called from the same thread that called Enter. After an await point, the code can resume on a different ThreadPool thread. If you could await inside a lock, the Monitor.Exit in the finally block would run on the wrong thread and throw SynchronizationLockException. This is why the C# compiler makes it a compile error. SemaphoreSlim.WaitAsync() / Release() have no thread affinity — they work correctly across thread switches."
/>
