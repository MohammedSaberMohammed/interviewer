## The Problem: All-or-Nothing Async

Before `IAsyncEnumerable<T>` (C# 8 / .NET Core 3.0), async methods returned all results at once:

```csharp
// OLD PATTERN: loads ALL rows into memory before returning
public async Task<List<Order>> GetAllOrdersAsync(CancellationToken ct)
{
    var orders = new List<Order>();
    await using var reader = await _db.ExecuteReaderAsync("SELECT * FROM Orders", ct);
    while (await reader.ReadAsync(ct))
        orders.Add(MapOrder(reader));
    return orders; // Returns 1 million orders at once — OOM risk
}

// Caller has to wait for ALL data before processing any of it
var orders = await GetAllOrdersAsync(ct);
foreach (var order in orders) // Only starts here, after full load
    Process(order);
```

This has two problems: memory pressure (everything buffered) and latency (processing can't start until everything is loaded).

## IAsyncEnumerable\<T\> — Streaming Results

With async streams, results flow one at a time:

```csharp
// NEW PATTERN: yields rows as they arrive from the database
public async IAsyncEnumerable<Order> StreamOrdersAsync(
    [EnumeratorCancellation] CancellationToken ct = default)
{
    await using var reader = await _db.ExecuteReaderAsync("SELECT * FROM Orders", ct);
    while (await reader.ReadAsync(ct))
        yield return MapOrder(reader); // Each row streamed as it's read
}

// Caller processes each item as it arrives — no buffering
await foreach (var order in StreamOrdersAsync(cancellationToken))
{
    await ProcessOrderAsync(order, cancellationToken);
}
```

The database read and the processing are interleaved. Caller processes row 1 while the DB is reading row 2.

## The [EnumeratorCancellation] Attribute

When using `await foreach` with `WithCancellation()`, pass the token correctly:

```csharp
// Method declares token with [EnumeratorCancellation]
public async IAsyncEnumerable<Item> GetItemsAsync(
    [EnumeratorCancellation] CancellationToken ct = default)
{
    for (int i = 0; i < 1000; i++)
    {
        ct.ThrowIfCancellationRequested(); // Manual check for CPU-bound parts
        yield return await FetchItemAsync(i, ct);
    }
}

// Caller can cancel via WithCancellation()
using var cts = new CancellationTokenSource(TimeSpan.FromSeconds(10));

await foreach (var item in GetItemsAsync()
    .WithCancellation(cts.Token)
    .ConfigureAwait(false))
{
    Process(item);
}
```

`[EnumeratorCancellation]` makes the compiler merge the token passed via `WithCancellation()` with the one in the method signature. Without it, you'd have to pass the token directly to the method call.

## ConfigureAwait on IAsyncEnumerable

```csharp
// Library code: suppress context capture on the whole enumeration
await foreach (var item in source.ConfigureAwait(false))
{
    // Each MoveNextAsync() resumes without capturing context
    Process(item);
}

// Combine ConfigureAwait and WithCancellation:
await foreach (var item in source
    .WithCancellation(ct)
    .ConfigureAwait(false))
{
    Process(item);
}
```

## Writing a Custom IAsyncEnumerable

```csharp
// Custom producer: reads messages from a queue asynchronously
public async IAsyncEnumerable<Message> ReadMessagesAsync(
    string queueName,
    [EnumeratorCancellation] CancellationToken ct = default)
{
    while (!ct.IsCancellationRequested)
    {
        Message? message = await _queue.TryDequeueAsync(
            timeout: TimeSpan.FromSeconds(1),
            cancellationToken: ct);

        if (message is null)
        {
            await Task.Delay(100, ct); // Back-off when queue is empty
            continue;
        }

        yield return message;
    }
}

// Usage in a background service:
public class MessageProcessor : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        await foreach (var msg in ReadMessagesAsync("orders", stoppingToken))
        {
            await HandleMessageAsync(msg, stoppingToken);
        }
    }
}
```

## EF Core Integration

EF Core supports `IAsyncEnumerable<T>` natively via `AsAsyncEnumerable()`:

```csharp
// Stream all large records without loading into memory
public async IAsyncEnumerable<Report> ExportReportsAsync(
    DateTime since,
    [EnumeratorCancellation] CancellationToken ct = default)
{
    // AsAsyncEnumerable() streams rows from the database
    await foreach (var row in _dbContext.Reports
        .Where(r => r.CreatedAt >= since)
        .AsAsyncEnumerable()
        .WithCancellation(ct))
    {
        yield return _mapper.Map<Report>(row);
    }
}
```

<Callout type="warning">
**DbContext lifetime with async streams**: when streaming from EF Core, the `DbContext` must stay alive for the entire iteration. Don't use `using var context = new MyDbContext()` inside the async iterator — it will be disposed before the caller finishes iterating. Use dependency-injected scoped context and control the scope at the caller level.
</Callout>

---

<Callout type="senior">
**Senior Insight: IAsyncEnumerable and backpressure**

`IAsyncEnumerable<T>` provides natural backpressure: the producer's `yield return` doesn't execute until the consumer calls `MoveNextAsync()`. If the consumer is slow (e.g., writing to a slow disk), the producer slows down proportionally — no unbounded buffering.

This is unlike `Channel<T>` or `Subject<T>` where the producer can get ahead of the consumer and fill unbounded memory. For true flow-controlled pipelines with decoupled producer/consumer, combine `Channel<T>` with `IAsyncEnumerable<T>` as covered in lessons 09 and 15.

```csharp
// IAsyncEnumerable provides implicit backpressure:
// producer only runs when consumer calls MoveNextAsync()
public async IAsyncEnumerable<ProcessedItem> ProcessPipelineAsync(
    IAsyncEnumerable<RawItem> source,
    [EnumeratorCancellation] CancellationToken ct = default)
{
    await foreach (var raw in source.WithCancellation(ct))
    {
        // Expensive processing — consumer waits for this
        var processed = await ExpensiveTransformAsync(raw, ct);
        yield return processed; // Consumer's MoveNextAsync() unblocks here
    }
}
```
</Callout>
