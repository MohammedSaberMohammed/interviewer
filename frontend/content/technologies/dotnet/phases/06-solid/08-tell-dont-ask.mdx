## What Is Tell-Don't-Ask?

Tell-Don't-Ask (TDA) is an object-oriented principle that says:

> **Tell** an object what to do. Don't **ask** it for data, make a decision, and then tell it the result.

The "ask" pattern pulls data out of an object, makes a decision in external code, and acts on the object from outside. The "tell" pattern pushes the decision-making responsibility into the object itself, where it belongs.

TDA directly supports encapsulation: if you ask for data to make a decision, the class's internal data is effectively public knowledge — you have broken encapsulation even if the property has a `private set`.

## The Ask Pattern: Anemic Domain Model

```csharp
// BAD: Anemic domain model — Order is just a data bag
// All behavior lives in the service, not in the object that owns the data

public class Order
{
    public int Id { get; set; }
    public OrderStatus Status { get; set; }
    public List<LineItem> LineItems { get; set; } = new();
    public decimal TotalAmount { get; set; }
    public DateTime? ShippedAt { get; set; }
    public string TrackingNumber { get; set; } = "";
}

// The service asks Order for its data, makes decisions, then modifies Order
public class OrderService
{
    public void ShipOrder(Order order, string trackingNumber)
    {
        // Asking: is the order confirmed?
        if (order.Status != OrderStatus.Confirmed)
            throw new InvalidOperationException("Only confirmed orders can be shipped");

        // Asking: does it have items?
        if (!order.LineItems.Any())
            throw new InvalidOperationException("Cannot ship empty order");

        // Making decisions and setting data from outside
        order.Status = OrderStatus.Shipped;
        order.ShippedAt = DateTime.UtcNow;
        order.TrackingNumber = trackingNumber;
    }

    public void CancelOrder(Order order, string reason)
    {
        // Asking again
        if (order.Status == OrderStatus.Shipped)
            throw new InvalidOperationException("Cannot cancel a shipped order");

        if (order.Status == OrderStatus.Cancelled)
            throw new InvalidOperationException("Order is already cancelled");

        // Setting from outside
        order.Status = OrderStatus.Cancelled;
    }
}
```

The `Order` class is a **data bag** — it holds data but no behavior. The service duplicates business rules across methods and must know Order's internal state to make every decision. This scatters business logic across the codebase.

## The Tell Pattern: Rich Domain Model

```csharp
// GOOD: Rich domain model — Order knows its own business rules

public class Order
{
    public int Id { get; private set; }
    public OrderStatus Status { get; private set; }
    public DateTime? ShippedAt { get; private set; }
    public string TrackingNumber { get; private set; } = "";

    private readonly List<LineItem> _lineItems = new();
    public IReadOnlyList<LineItem> LineItems => _lineItems;

    // Tell the order to add an item — it validates itself
    public void AddLineItem(LineItem item)
    {
        if (item is null) throw new ArgumentNullException(nameof(item));
        if (Status != OrderStatus.Draft)
            throw new InvalidOperationException("Can only add items to a draft order");
        _lineItems.Add(item);
    }

    // Tell the order to confirm itself — it knows the rules
    public void Confirm()
    {
        if (!_lineItems.Any())
            throw new InvalidOperationException("Cannot confirm an empty order");
        if (Status != OrderStatus.Draft)
            throw new InvalidOperationException("Only draft orders can be confirmed");

        Status = OrderStatus.Confirmed;
    }

    // Tell the order to ship — it validates its own state
    public void Ship(string trackingNumber)
    {
        if (string.IsNullOrWhiteSpace(trackingNumber))
            throw new ArgumentException("Tracking number is required", nameof(trackingNumber));
        if (Status != OrderStatus.Confirmed)
            throw new InvalidOperationException("Only confirmed orders can be shipped");

        Status = OrderStatus.Shipped;
        ShippedAt = DateTime.UtcNow;
        TrackingNumber = trackingNumber;
    }

    // Tell the order to cancel — it knows when it is too late
    public void Cancel()
    {
        if (Status == OrderStatus.Shipped)
            throw new InvalidOperationException("Cannot cancel a shipped order");
        if (Status == OrderStatus.Cancelled)
            throw new InvalidOperationException("Order is already cancelled");

        Status = OrderStatus.Cancelled;
    }
}

// The service becomes trivial — it just coordinates, not decides
public class OrderService
{
    private readonly IOrderRepository _repository;

    public OrderService(IOrderRepository repository) => _repository = repository;

    public void ShipOrder(int orderId, string trackingNumber)
    {
        var order = _repository.GetById(orderId)
            ?? throw new KeyNotFoundException($"Order {orderId} not found");

        order.Ship(trackingNumber); // TELL — Order handles the validation
        _repository.Save(order);
    }

    public void CancelOrder(int orderId)
    {
        var order = _repository.GetById(orderId)
            ?? throw new KeyNotFoundException($"Order {orderId} not found");

        order.Cancel(); // TELL — Order handles the validation
        _repository.Save(order);
    }
}
```

The service no longer needs to know **anything** about the internal rules of Order. Business rules are in one place: inside `Order`.

## Spotting the Ask Pattern in Code

```csharp
// These patterns signal "asking" instead of "telling":

// 1. Reading state to make a decision about the object
if (account.Balance >= amount)
    account.Balance -= amount;       // ASK then act
// vs.
account.Withdraw(amount);            // TELL

// 2. Reading multiple properties to compute something the class should compute
decimal discount = product.IsOnSale
    ? product.OriginalPrice * product.DiscountPercent / 100m
    : 0m;                            // ASK (3 properties) to compute
// vs.
decimal discount = product.CurrentDiscount(); // TELL

// 3. Type-checking to decide behavior (also an LSP smell)
if (shape is Circle c)
    area = Math.PI * c.Radius * c.Radius;
else if (shape is Rectangle r)
    area = r.Width * r.Height;      // ASK what type, then compute
// vs.
area = shape.Area();                 // TELL
```

<Callout type="senior">
**Senior Insight: TDA and Domain-Driven Design**

Tell-Don't-Ask is the practical OOP expression of the DDD principle of **rich domain models**. Eric Evans' seminal Domain-Driven Design book argues that the domain model should contain both data *and* behavior — the rules that govern that data.

The anemic domain model — data bags + service classes that do all the work — is one of the most common patterns in .NET codebases and one of the most-criticized by experienced engineers. Martin Fowler calls it an anti-pattern because it distributes business logic across services rather than encapsulating it in the domain objects that own the data.

In practice, the trade-off is real: anemic models are easy to serialize, easy to map with ORMs like Entity Framework, and easy to reason about in isolation. This is why they proliferate in CRUD-heavy apps.

The right balance: apply TDA and rich models in the **core domain** — the parts of the system where business rules are complex, frequently evolving, and central to the application's purpose. For simple CRUD operations on reference data, an anemic model with a thin service may be pragmatic.

Entity Framework Core works well with rich domain models when you use **private setters** (EF can set them via reflection) and configure navigation properties in `OnModelCreating` rather than exposing public collections directly.
</Callout>

<MythBuster
  myth="TDA means services should never read data from domain objects"
  reality="TDA does not prohibit reading data from objects. It targets the pattern of reading data specifically to make a decision that should be made inside the object. Reading `order.TrackingNumber` to display it in a view is perfectly fine — you are consuming the result of an object's state, not interrogating it to make a mutation decision on its behalf. The smell is: reading state → making a decision → calling a setter or mutating the object."
/>
