## What Is the Interface Segregation Principle?

Robert C. Martin's definition: **clients should not be forced to depend on interfaces they do not use.**

A "fat" or "polluted" interface forces implementing classes to provide methods they don't need, typically resulting in empty implementations or `throw new NotImplementedException()`. ISP says: prefer many small, focused interfaces over one large, general-purpose interface.

---

## The Fat Interface Problem

```csharp
// BAD: One interface that tries to cover everything
public interface IWorker
{
    void Work();
    void TakeBreak();
    void AttendMeeting();
    void SubmitTimesheet();
    void ManageTeam();      // Not all workers manage teams!
    void ApproveExpenses(); // Not all workers approve expenses!
}

// RobotWorker must implement methods that make no sense for it
public class RobotWorker : IWorker
{
    public void Work() { /* does actual work */ }
    public void TakeBreak() { /* robots don't take breaks */ }
    public void AttendMeeting() { /* robots don't attend meetings */ }
    public void SubmitTimesheet() { /* robots don't have timesheets */ }

    // These are forced stubs — the interface demands them
    public void ManageTeam()      => throw new NotImplementedException();
    public void ApproveExpenses() => throw new NotImplementedException();
}

// ContractorWorker also suffers
public class ContractorWorker : IWorker
{
    public void Work() { /* works */ }
    public void TakeBreak() { /* takes breaks */ }
    public void AttendMeeting() { /* attends meetings */ }
    public void SubmitTimesheet() { /* submits timesheet */ }

    // Contractors don't manage teams or approve expenses
    public void ManageTeam()      => throw new NotImplementedException();
    public void ApproveExpenses() => throw new NotImplementedException();
}
```

The forcing problem: any code that depends on `IWorker` is coupled to all six methods, even if it only ever calls `Work()`. When `ManageTeam()` changes its signature, every `IWorker` implementor must be recompiled — including `RobotWorker`, which never wanted `ManageTeam()` in the first place.

## The Refactored Version: Role Interfaces

```csharp
// GOOD: Each interface represents a single capability (role)

public interface IWorker
{
    void Work();
}

public interface IBreakTaker
{
    void TakeBreak();
}

public interface IMeetingAttendee
{
    void AttendMeeting();
}

public interface ITimesheetSubmitter
{
    void SubmitTimesheet();
}

public interface ITeamManager
{
    void ManageTeam();
}

public interface IExpenseApprover
{
    void ApproveExpenses();
}

// Each class implements only what it actually supports
public class RobotWorker : IWorker
{
    public void Work() { /* actual work */ }
    // No forced stubs — robots just don't implement the other interfaces
}

public class ContractorWorker : IWorker, IBreakTaker, IMeetingAttendee, ITimesheetSubmitter
{
    public void Work()            { /* works */ }
    public void TakeBreak()       { /* breaks */ }
    public void AttendMeeting()   { /* meetings */ }
    public void SubmitTimesheet() { /* timesheet */ }
}

public class ManagerWorker : IWorker, IBreakTaker, IMeetingAttendee,
                              ITimesheetSubmitter, ITeamManager, IExpenseApprover
{
    public void Work()            { /* works */ }
    public void TakeBreak()       { /* breaks */ }
    public void AttendMeeting()   { /* meetings */ }
    public void SubmitTimesheet() { /* timesheet */ }
    public void ManageTeam()      { /* manages team */ }
    public void ApproveExpenses() { /* approves expenses */ }
}

// Callers depend only on what they need
public class WorkScheduler
{
    // Only needs IWorker — doesn't care about meetings or expenses
    public void ScheduleWork(IWorker worker) => worker.Work();
}

public class MeetingOrganizer
{
    // Only needs IMeetingAttendee
    public void Schedule(IEnumerable<IMeetingAttendee> attendees)
    {
        foreach (var a in attendees) a.AttendMeeting();
    }
}
```

## ISP in the .NET BCL

The .NET BCL is a masterclass in ISP. Instead of one giant `ICollection` that does everything, there is a carefully layered hierarchy:

```csharp
// IEnumerable<T>     — only: iterate
// IReadOnlyCollection<T> — iterate + Count
// IReadOnlyList<T>   — iterate + Count + index access
// ICollection<T>     — iterate + Count + Add/Remove
// IList<T>           — all of the above + index access + Insert/RemoveAt

// Each client depends only on what it needs:

// A method that only reads items — use the narrowest interface
void PrintAll<T>(IEnumerable<T> items)
{
    foreach (var item in items) Console.WriteLine(item);
}

// A method that needs the count too
void ProcessBatch<T>(IReadOnlyCollection<T> batch)
{
    Console.WriteLine($"Processing {batch.Count} items");
    foreach (var item in batch) Process(item);
}

// A method that needs to add items
void Populate<T>(ICollection<T> target, IEnumerable<T> source)
{
    foreach (var item in source) target.Add(item);
}

static void Process<T>(T item) { }
```

By depending on `IEnumerable<T>` instead of `List<T>`, `PrintAll` works with arrays, lists, queues, lazy sequences, and anything else that implements `IEnumerable<T>` — without being coupled to mutation methods it doesn't use.

<Callout type="senior">
**Senior Insight: ISP and Testability**

ISP has a direct impact on testability. Fat interfaces are painful to mock:

```csharp
// BAD: Mocking IWorker forces you to set up 6 methods
// even when your test only exercises Work()
var mockWorker = new Mock<IWorker>();
mockWorker.Setup(w => w.Work()); // the only thing you care about
mockWorker.Setup(w => w.TakeBreak()); // forced by the interface
mockWorker.Setup(w => w.AttendMeeting()); // forced
// ... and so on

// GOOD: Mocking IWorker (the segregated version) is trivial
var mockWorker = new Mock<IWorker>();
mockWorker.Setup(w => w.Work());
// Done — only one method to care about
```

In practice, when you find yourself stubbing out five methods in a mock setup to test one behavior, the interface is probably violating ISP.

C# 8 default interface methods provide an alternative for backward compatibility: you can add a new method to an existing interface with a default implementation, so existing implementors aren't broken. This is useful for evolving interfaces in published libraries, but it is not a substitute for ISP — it is a migration tool. The interface is still fat from the client's perspective.
</Callout>

<MythBuster
  myth="ISP means every interface should have exactly one method"
  reality="ISP means interfaces should be cohesive around a single client's needs — not that they must be minimal. IDisposable has one method and that is correct. IComparer<T> has one method and that is correct. But ICollection<T> has many methods and is also fine because all of those methods represent a single coherent capability: mutable collection management. The smell ISP targets is when an interface forces implementors to provide methods that are irrelevant to their core purpose."
