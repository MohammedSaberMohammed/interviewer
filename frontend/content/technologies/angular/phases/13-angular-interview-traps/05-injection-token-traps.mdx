## Trap 1: NullInjectorError — Missing Provider

The most common DI error:

```
NullInjectorError: No provider for UserService!
```

```typescript
// ❌ Cause 1: Service not provided anywhere
@Injectable() // No providedIn!
class UserService { }

// Fix: Add providedIn
@Injectable({ providedIn: 'root' })
class UserService { }
```

```typescript
// ❌ Cause 2: Service provided in lazy module, injected in root-level component
// LazyModule provides UserService
// AppComponent (in root) injects UserService
// → AppComponent's injector (root) doesn't have UserService

// Fix: Provide in root OR only inject inside the lazy module
```

```typescript
// ❌ Cause 3: Testing without providing mock
// In test: providers: [ComponentUnderTest]
// ComponentUnderTest injects AuthService
// But test doesn't provide AuthService → Error!

// Fix:
TestBed.configureTestingModule({
  providers: [
    ComponentUnderTest,
    { provide: AuthService, useValue: mockAuthService } // ← Required!
  ]
});
```

## Trap 2: Token Identity — Same Token, Different Instances

`InjectionToken` identity is based on **object reference**, not the token name string:

```typescript
// tokens.ts
export const API_URL = new InjectionToken<string>('api-url');
```

```typescript
// ❌ WRONG: Re-creating the token in a different file
// config.ts
const API_URL = new InjectionToken<string>('api-url'); // NEW instance!

// This token is NOT the same as the one in tokens.ts
// Even though the description string is identical
```

```typescript
// ✅ CORRECT: Always import the same token instance
import { API_URL } from './tokens';

// Provide it:
providers: [{ provide: API_URL, useValue: 'https://api.example.com' }]

// Inject it:
const apiUrl = inject(API_URL); // Uses the same API_URL reference
```

<Callout type="warning">
InjectionToken identity is reference-based (like object identity). Two `new InjectionToken('same-name')` calls create two different, incompatible tokens even with the same description string. The description is only for debugging — it appears in error messages and DevTools.
</Callout>

## Trap 3: Circular Dependency

```typescript
// ❌ ServiceA injects ServiceB, ServiceB injects ServiceA → circular!

@Injectable({ providedIn: 'root' })
class ServiceA {
  private serviceB = inject(ServiceB); // ServiceB depends on ServiceA!
}

@Injectable({ providedIn: 'root' })
class ServiceB {
  private serviceA = inject(ServiceA); // Circular!
}
// Error: Circular dependency in DI detected for ServiceA
```

Solutions:

```typescript
// Fix 1: Restructure — extract shared logic to a third service
@Injectable({ providedIn: 'root' })
class SharedLogicService { /* Common functionality */ }

@Injectable({ providedIn: 'root' })
class ServiceA {
  private shared = inject(SharedLogicService); // No circular dep
}

@Injectable({ providedIn: 'root' })
class ServiceB {
  private shared = inject(SharedLogicService); // No circular dep
}
```

```typescript
// Fix 2: Use forwardRef (delays resolution)
@Injectable({ providedIn: 'root' })
class ServiceA {
  // forwardRef: resolve ServiceB lazily to break the circular reference
  private serviceB = inject(forwardRef(() => ServiceB));
}
```

## Trap 4: Using inject() Outside Injection Context

```typescript
// ❌ inject() only works in injection contexts

class MyClass {
  // ❌ Not in a component/service/directive — inject() fails
  private service = inject(SomeService);
}

// ❌ Outside of class field initializer or constructor
function helper() {
  const service = inject(SomeService); // Error: not in injection context
}
```

```typescript
// ✅ Valid injection contexts:
@Injectable()
class MyService {
  // Class field initializer — ✅
  private dep = inject(OtherService);

  constructor() {
    // Constructor body — ✅
    const dep2 = inject(ThirdService);
  }
}

// Function called within injection context (e.g., factory function):
const factory = () => {
  const service = inject(SomeService); // ✅ If called as InjectionToken factory
  return new SomeThing(service);
};
```

## Trap 5: multi: true Providers Accumulate

```typescript
// ❌ Multiple modules providing the same multi-token accumulate ALL values

// Modules A and B both provide:
providers: [{ provide: HTTP_INTERCEPTORS, useClass: MyInterceptor, multi: true }]

// If both modules are imported, the interceptor runs TWICE
// This is a common source of duplicate HTTP headers, duplicate logging, etc.
```

```typescript
// ✅ Provide interceptors only once, in the root config
// app.config.ts:
provideHttpClient(withInterceptors([myInterceptor])) // Only here

// Never provide the same interceptor in feature modules
```

## Trap 6: Provider Scope Mismatch (Captive Dependency)

```typescript
// ❌ Singleton service injecting a request-scoped service
@Injectable({ providedIn: 'root' }) // Singleton
class AppService {
  // RequestScopedService should be new per request,
  // but AppService is created once and holds one instance forever
  private requestService = inject(RequestScopedService);
}

@Injectable({ providedIn: 'root' }) // Also singleton!
class RequestScopedService {
  // Intended to be per-request but providedIn: 'root' makes it a singleton
}
```

The captive dependency problem: a short-lived service injected into a longer-lived service is "captured" and becomes as long-lived as its consumer.

## Trap 7: useExisting vs useClass Confusion

```typescript
// ❌ useExisting vs useClass confusion

// useClass: Creates a NEW instance of the class
providers: [
  { provide: AbstractLogger, useClass: ConcreteLogger }
  // Two separate instances: one for AbstractLogger, one for ConcreteLogger if injected
]

// useExisting: Reuses the EXISTING instance from another token
providers: [
  ConcreteLogger, // First, provide the concrete
  { provide: AbstractLogger, useExisting: ConcreteLogger } // Alias — same instance!
]
```

```typescript
// Common mistake: using useClass when you want the same instance
// If ConcreteLogger has state, useClass creates separate instances
// while useExisting ensures all consumers share one instance
```

## Senior Insight: DI Design Principles

The best defense against DI traps:
1. **Single responsibility in services** — avoids circular dependencies
2. **Explicit provider configuration** — `providedIn: 'root'` for singletons, component providers for local state
3. **Shared token files** — all `InjectionToken` definitions in one file, imported everywhere
4. **One interceptor registration point** — always in `app.config.ts`, never in feature modules
5. **Test-first mindset** — if a service is hard to provide in tests, its DI configuration is probably wrong

The hardest DI bugs are provider scope mismatches (captive dependency) and token identity issues — both require understanding DI as a hierarchical system, not just a "magic service locator".
