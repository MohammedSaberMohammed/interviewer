## Why Lazy Loading Matters

Without lazy loading, Angular bundles ALL your application code into one large JavaScript file. The user must download and parse the entire app before seeing anything — even pages they'll never visit.

Lazy loading splits your app into route-level chunks. The user downloads only what they need, when they need it:

```
Without lazy loading:
Initial bundle: 2.5MB (entire app)
Time to interactive: 8 seconds on 3G

With lazy loading:
Initial bundle: 400KB (core + eagerly loaded routes)
On-demand bundles: 50-200KB each
Time to interactive: 1.2 seconds on 3G
```

## `loadComponent` — Lazy Load a Single Standalone Component

The modern approach for standalone components:

```typescript
// app.routes.ts
const routes: Routes = [
  // Eagerly loaded — included in initial bundle
  { path: 'home', component: HomeComponent },

  // Lazily loaded — separate bundle downloaded when user navigates to /products
  {
    path: 'products',
    loadComponent: () => import('./products/products.component')
      .then(m => m.ProductsComponent)
  },

  // With guards and resolvers on lazy routes
  {
    path: 'admin',
    loadComponent: () => import('./admin/admin.component')
      .then(m => m.AdminComponent),
    canActivate: [authGuard, adminGuard],
    resolve: { dashboardData: adminDataResolver }
  }
];
```

## `loadChildren` — Lazy Load a Route Module

For feature areas with multiple child routes:

```typescript
// app.routes.ts
const routes: Routes = [
  {
    path: 'orders',
    loadChildren: () => import('./orders/orders.routes')
      .then(m => m.ORDER_ROUTES) // Named export of a Routes array
  }
];

// orders/orders.routes.ts — a standalone routes file (no NgModule needed)
export const ORDER_ROUTES: Routes = [
  { path: '', component: OrderListComponent },
  { path: ':id', component: OrderDetailComponent },
  {
    path: ':id/edit',
    component: OrderEditComponent,
    canDeactivate: [unsavedChangesGuard]
  }
];
```

<Callout type="tip">
With standalone components, `loadChildren` routes to a `Routes` array, not a `NgModule`. This is simpler and more tree-shakeable. The array pattern (`ORDER_ROUTES`) lets you co-locate route definitions with the feature without NgModule boilerplate.
</Callout>

## Legacy NgModule Lazy Loading

You'll encounter this pattern in existing codebases:

```typescript
// app.routes.ts — old NgModule approach
{
  path: 'orders',
  loadChildren: () => import('./orders/orders.module').then(m => m.OrdersModule)
}

// orders/orders.module.ts
@NgModule({
  imports: [
    RouterModule.forChild([  // forChild — NOT forRoot!
      { path: '', component: OrderListComponent },
      { path: ':id', component: OrderDetailComponent },
    ])
  ],
  declarations: [OrderListComponent, OrderDetailComponent]
})
export class OrdersModule {}
```

The critical rule: use `RouterModule.forChild()` in lazy-loaded modules, never `forRoot()`. Using `forRoot()` creates a second Router instance and breaks routing.

## What Gets Bundle-Split

Everything imported by the lazy route module/component:

```typescript
// This component and ALL its imports go into a separate bundle:
@Component({
  standalone: true,
  imports: [
    NgxChartsModule,   // ← This charting library goes in the lazy bundle
    MatTableModule,    // ← Only downloaded when user visits this route
    DataGridComponent  // ← Child components too
  ]
})
class AdminDashboardComponent {}
```

If `NgxChartsModule` is 500KB, it only downloads when the user actually visits the admin route — not on initial page load.

## Preloading Lazy Routes

Lazy loading downloads on demand, but this means a slight delay on first navigation. Preloading downloads lazy bundles in the background after the initial page loads:

```typescript
import { PreloadAllModules, withPreloading } from '@angular/router';

provideRouter(routes, withPreloading(PreloadAllModules))
// Loads ALL lazy routes in the background after initial paint
// Good for small apps; bad for very large ones
```

## Senior Insight: What NOT to Lazy Load

Don't lazy load:
- Routes users visit immediately on landing (home, login)
- Small components (overhead of separate HTTP request isn't worth it)
- Core services and utilities — these should be in root providers

Do lazy load:
- Feature areas behind auth (admin, dashboard)
- Heavy third-party dependencies (charting, PDF, video)
- Rarely visited pages (settings, help, terms)
- Routes that aren't part of the critical user path

A good target: keep the initial bundle under 300KB (gzipped). If you're over that, lazy load more aggressively.
